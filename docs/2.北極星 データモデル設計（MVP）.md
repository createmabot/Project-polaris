# 北極星 データモデル設計（MVP）

更新日: 2026-03-08

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVP開発に向けたデータモデル設計を整理するものである。

前提として、北極星は TradingView を **表示・監視・一次検証** に活用し、北極星側では **要約・比較・仮説・履歴管理** を担う。そのため、データモデル設計においても、TradingView の表示内容そのものを完全複製するのではなく、北極星独自の資産となるデータを中心に設計する。

本ドキュメントでは以下を定義する。

- MVPで保持すべきデータの範囲
    
- エンティティ一覧
    
- 主なリレーション
    
- テーブル設計案
    
- 画面との対応
    
- 将来拡張を見据えた設計方針
    

---

## 2. データモデル設計の基本方針

### 2.1 北極星側で正本として保持するもの

北極星が正本として保持すべきデータは以下とする。

- 監視対象銘柄
    
- 保有銘柄
    
- TradingViewアラート受信履歴
    
- AI要約結果
    
- 銘柄研究ノート
    
- 銘柄比較履歴
    
- 売買ルール定義
    
- 自然言語→Pine変換結果
    
- バックテスト取込結果
    
- 検証レポート
    

### 2.2 北極星側で完全保持しなくてよいもの

以下は TradingView または外部データソースを参照前提とし、MVPでは完全な永続保持を必須としない。

- 高機能チャート描画データ
    
- 全期間のOHLCV生データ
    
- テクニカル指標の全計算結果
    
- 会社概要の詳細マスタ
    
- ニュース本文全文
    

ただし、AI要約や比較の根拠として使った情報の最低限のメタ情報は北極星側に保持する。

### 2.3 設計原則

- 画面中心ではなく、**利用フロー中心** に設計する
    
- 外部ソース由来データと、北極星生成データを分離する
    
- AI生成物には必ず入力元・生成時刻・バージョンを紐づける
    
- 検証結果は再利用性を重視し、実行条件を明示的に保存する
    
- MVPでは正規化しすぎず、将来の検索性・履歴性を優先する

## 5. リレーション概要

主要な関係は以下の通り。

- users は複数の watchlists を持つ
    
- watchlists は複数の watchlist_items を持つ
    
- watchlist_items は symbols を参照する
    
- users は複数の positions を持つ
    
- positions は symbols を参照する
    
- symbols は複数の alert_events を持つ
    
- alert_events は複数の external_references を持ちうる
    
- ai_summaries は symbols または alert_events または comparison_sessions または backtest_runs に紐づく
    
- research_notes は users と symbols に紐づく
    
- research_notes は複数の note_revisions を持つ
    
- comparison_sessions は users に紐づく
    
- comparison_sessions は複数の comparison_symbols を持つ
    
- comparison_sessions は1件以上の comparison_results を持つ
    
- strategy_rules は users に紐づく
    
- strategy_rules は複数の strategy_rule_versions を持つ
    
- strategy_rule_versions は pine_scripts を持つ
    
- strategy_rule_versions は複数の backtest_runs を持つ
    
- backtest_runs は backtest_metrics と backtest_trade_summaries を持つ
    
- backtest_runs は backtest_import_files を持つ

## 10. 設計上の重要判断

### 10.1 AI生成物は共通テーブルで持つ

用途別にテーブルを分けすぎると検索性が悪くなるため、AI要約は共通化する。

### 10.2 ルール本体とバージョンを分ける

自然言語ルールは試行錯誤が前提のため、親子構造を持たせる。

### 10.3 比較結果は一旦JSON保持でよい

MVPでは比較ロジックの変化が大きいため、初期は柔軟性優先とする。

### 10.4 ノート履歴は最初から持つ

投資仮説の変化履歴は後から重要になるため、MVPでも削らない。

### 10.5 外部参照は本文ではなく根拠メタ情報重視

ニュースや開示の全文保存より、根拠リンクと要約入力の再現性を優先する。

### 11.1 内製バックテストエンジン着手前の最小保存単位（未実装固定）

本節は、北極星内バックテストエンジンの**着手前固定**であり、本資料更新時点では未実装である。

- TradingView 側の一次検証フロー（表示 / 監視 / 一次検証）は維持する
- 北極星側は、内製実行結果の保存・再現・レポート責務を最小単位で固定する

最小保存単位は以下の4つとする。

1. execution job
   - 実行要求と状態遷移の正本
   - 想定ステータス: `queued` / `running` / `succeeded` / `failed` / `canceled`
2. execution input snapshot
   - 実行時に固定した入力条件（strategy version, rule/pine snapshot, market/timeframe, data source 条件, engine config）
3. execution result summary
   - UI/レポートで一次表示する最小結果（主要指標・実行時刻・結果種別）
4. execution artifact pointer
   - 詳細結果の参照先（将来の詳細レポート/明細を指すポインタ）

既存モデルとの責務分離は以下とする。

- `backtest_runs` / `backtest_import_files`: TradingView 由来の取込履歴管理
- 内製 execution job: 北極星内実行の状態管理
- 両者は strategy/version と実行時 snapshot を介して比較・再利用可能に接続する

再現性責務として、少なくとも以下を input snapshot 側で固定保存する。

- strategy_rule_version_id
- 実行時 natural language rule / generated pine
- market / timeframe
- データソース条件（期間・粒度・取得元）
- engine version / config

### 11.2 engine_estimated の最小実データ接続方針（次段固定・未実装）

`summary_kind = engine_estimated` を一段現実寄りにするため、次段で扱う最小範囲を以下で固定する。

- 対象データ種別（最小）: 日足 OHLCV
- 対象市場 / timeframe（最初の入口）: `JP_STOCK` / `D`
- 取得責務: route/worker ではなく data source adapter に集約する
- 取得失敗時: execution を `failed` にし、`errorCode = DATA_SOURCE_UNAVAILABLE` を保存する

再現性のため、execution input snapshot（または同等責務）に以下の `data_source_snapshot` を保持する。

- `source_kind`（例: `daily_ohlcv`）
- `market`
- `timeframe`
- `from`
- `to`
- `fetched_at`
- `data_revision`
- `bar_count`

段階差（固定）:

- `scaffold_deterministic`: 入力条件のみを使う決定論的な足場値
- `engine_estimated`: 実データ取得結果を参照する推定段階（ただし本格シミュレーションではない）
- `engine_actual`: 売買シミュレーション、コスト反映、再現可能レポートまで含む本段階（未実装）

## 13. 結論

北極星MVPのデータモデルは、TradingViewの表示や監視能力を前提としつつ、北極星独自の価値である **要約・比較・仮説・検証履歴** を中心に構築する。

そのために、MVPでは以下を中核資産として保持する。

- 銘柄
    
- アラート履歴
    
- AI要約
    
- 研究ノート
    
- ルール定義とバージョン
    
- Pine生成結果
    
- バックテスト実行結果
    

この構成により、短期的にはMVP実装を進めやすくし、中長期的には分析資産が北極星側に蓄積される設計とする。
