# 北極星 画面別の入力 / 出力サンプル（MVP）

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVPにおいて、各画面が必要とする **入力 / 出力の具体例** を整理するものである。

これまでに、以下の資料を作成してきた。

- MVP画面一覧・画面遷移・主要コンポーネント
    
- データモデル設計
    
- API / ユースケース単位の入出力設計
    
- APIごとの JSON サンプル集
    
- MVPタスク分解
    

本資料は、それらを **画面実装の視点** で再整理し、  
「どの画面が、どのタイミングで、どの入力を受け、何を表示するか」  
をフロントエンド実装可能な粒度に落とすことを目的とする。

---

## 2. この資料の見方

各画面について、以下の項目で整理する。

- 画面名
    
- 画面の目的
    
- 画面入力（URLパラメータ・フォーム入力・内部状態）
    
- 取得API
    
- 主な表示ブロック
    
- 出力サンプル（画面用 view model）
    
- ユーザー操作
    
- 遷移先
    

---

## 3. 対象画面一覧

MVPで扱う画面は以下の6つ。

1. ホーム
    
2. 銘柄詳細

## 8.8 内製バックテストエンジン（最小実装 + 段階固定）

本資料更新時点では、内製バックテストエンジンは最小契約（execution/status/result、estimated/scaffold 段階）まで実装済み。  
`engine_actual` は初回最小スライスを実装済みとし、画面責務は最小単位に限定する。

- 実行開始: Strategy Version を起点に execution job を作成する
- `execution_target.symbol` canonical（`market = JP_STOCK`）は `7203` を正とし、`TYO:7203` は受理後に `7203` へ正規化する
- `execution_target.symbol` canonical（`market = JP_STOCK`）は前後空白を trim し、4桁数字以外は `INVALID_EXECUTION_TARGET`
- 実行状態: `queued / running / succeeded / failed / canceled` を表示する
- 結果表示: summary（主要指標）と artifact 参照を分離する
- 再現性表示: 実行時 rule/pine/market/timeframe/data_range/engine_version を表示する

責務分担は維持する。

- TradingView: 表示 / 監視 / 一次検証
- 北極星: 自然言語変換 / 履歴保存 / レポート / 内製実行結果管理

### 8.8.1 設計固定（実装済み最小）: engine_estimated の実データ接続表示

`engine_estimated` は `daily_ohlcv / JP_STOCK / D` の最小 provider 経路で実データ接続し、画面表示上も再現性条件を確認できる前提を固定する。

- data source kind（例: `daily_ohlcv`）
- market / timeframe
- from / to
- fetched_at
- data_revision
- bar_count

`engine_estimated` の `result_summary.metrics` は normalized bars 由来の価格要約として扱う。
- `bar_count`
- `first_close` / `last_close`
- `price_change` / `price_change_percent`
- `period_high` / `period_low` / `range_percent`
- これらは売買シミュレーションの最終成績ではなく、`engine_actual` 導入前の推定段階メトリクス
- empty bars（`bar_count = 0`）でも `engine_estimated` は失敗にせず、0件要約を返す

### 8.8.2 設計固定（最小接続済み）: result consumer 表示判定

internal-backtests の consumer/UI は以下で判定する。

- 成否の正本: `GET /api/internal-backtests/executions/:executionId` の `status`
- 失敗種別: `error_code`
- 0件成功判定: `status = succeeded` かつ `summary_kind = engine_estimated` かつ `metrics.bar_count = 0`
- 再現性確認: `input_snapshot.data_source_snapshot.bar_count`

最小表示パターン:

- 通常成功: metrics 通常表示
- データ0件成功: 「対象期間のデータがありません」系表示（失敗色にしない）
- 取得失敗（`DATA_SOURCE_UNAVAILABLE`）: 「データ取得に失敗しました」系表示
- その他失敗: 「内部エラーが発生しました」系表示

判定カテゴリ（共有 helper）:

- `success_with_data`
- `success_no_data`
- `data_source_unavailable`
- `internal_failure`
- `not_ready`

`summary_kind = scaffold_deterministic` は現時点では `success_with_data` と同カテゴリ扱い。

上記カテゴリは view-model helper で以下に変換して表示層へ渡す。

- `state_label`
- `is_error`
- `is_empty`
- `can_show_metrics`
- `recommended_message_key`

最小対応:

- `success_with_data` -> `state_label=success`
- `success_no_data` -> `state_label=success_no_data`, `recommended_message_key=internal_backtest.result.success_no_data`
- `data_source_unavailable` -> `state_label=error_data_source_unavailable`, `recommended_message_key=internal_backtest.result.data_source_unavailable`
- `internal_failure` -> `state_label=error_internal`, `recommended_message_key=internal_backtest.result.internal_failure`
- `not_ready` -> `state_label=not_ready`, `recommended_message_key=internal_backtest.result.not_ready`

`GET /api/internal-backtests/executions/:executionId/result` は `succeeded` 以外で `RESULT_NOT_READY` のため、
失敗系は status API 側で表示判定する。

最小接続として `Strategy Version Detail` に internal-backtests 表示領域を追加済み。
- status/result の最小取得
- view-model helper 経由の表示分岐
- `success_with_data` / `success_no_data` / `data_source_unavailable` / `not_ready` の表示差
- `engine_actual` の場合は artifact read API（`/api/internal-backtests/executions/:executionId/artifacts/engine_actual/trades-and-equity`）を追加で呼び、`trades / equity_curve` を read-only 表示
- artifact 表示状態は `success(with trades)` / `success(no-trade)` / `RESULT_NOT_READY` / `NOT_FOUND` / `fetch error` を最小分岐で扱う

### 8.8.3 実装反映（最小）: engine_actual 初回スライスの表示前提

`engine_actual` 初回スライスを実装反映し、UI は最小表示前提を維持する。

- 対象: `JP_STOCK` / `D` / 単一銘柄
- 売買モデル: long-only / 1ポジション
- `engine_config.actual_rules` は最小3系統を受理:
    - `close_above_previous_close` / `close_below_previous_close`（既定）
    - `price_above_sma` / `price_below_sma`（`period`: 2..200）
    - `price_above_threshold` / `price_below_threshold`（`threshold`: 正数）
- rule 設定不正は `POST /api/internal-backtests/executions` で `VALIDATION_ERROR`
- summary_kind: `engine_actual`
- 表示優先:
    1. 実行状態（status）
    2. 実績系 summary（trade_count, win_rate, total_return_percent, max_drawdown_percent）
    3. artifact 参照（`/internal-backtests/executions/:executionId/artifacts/engine_actual/trades-and-equity`）

初回スライスでの非対象表示:

- short / 複数ポジション / intraday
- 複数銘柄同時実行
- portfolio 最適化
- corporate action 完全対応 UI

### 8.8.4 実装反映（最小追加）: engine_actual execution サマリー比較表示

`StrategyVersionDetail` の内製バックテストセクション内に、`engine_actual` execution ごとの結果サマリーを最小で読める UI を追加した（read-only のみ）。

**表示対象:** `summary_kind = engine_actual` かつ artifact 取得成功時のみ表示。

**summary card（`data-testid="engine-actual-summary-card"`）:**

| 表示項目 | フィールド | 備考 |
|---|---|---|
| ルールパターン | `input_snapshot.actual_rules` からラベル整形 | 生 JSON ではなく readable 形式 |
| 取引数 | `result_summary.metrics.trade_count` | null 時は `-` |
| 勝率 | `result_summary.metrics.win_rate` | `"62.5%"` 形式 |
| 総リターン | `result_summary.metrics.total_return_percent` | `"+8.30%"` / `"-3.10%"` 形式（符号付き） |
| 最大DD | `result_summary.metrics.max_drawdown_percent` | `"-5.20%"` 形式（符号付き） |
| 平均保有期間 | `result_summary.metrics.holding_period_avg_bars` | 値ある場合のみ表示 |
| 初回取引日 | `result_summary.metrics.first_trade_at` | 値ある場合のみ表示 |
| 最終取引日 | `result_summary.metrics.last_trade_at` | 値ある場合のみ表示 |

**rule パターンラベル整形規則（最小）:**

- `actual_rules` が null / 空: `"default"`
- `close_above_previous_close` / `close_below_previous_close`: `"default (close_above_previous_close)"` 等
- `price_above_sma`（`period=25`）: `"price_above_sma (period=25)"`
- `price_above_threshold`（`threshold=500`）: `"price_above_threshold (threshold=500)"`
- 複数 rule: `/` 区切りで結合

**artifact 状態分岐（既存維持）:**

summary card は `internalEngineActualArtifactData` 取得成功時のみ表示。以下の既存分岐は変更なし。

- `success(with trades)` / `success(no-trade)` / `RESULT_NOT_READY` / `NOT_FOUND` / `fetch error`
- no-trade（`trades.length === 0`）は error 扱いしない
- `engine_estimated` の表示はそのまま維持

**整形ヘルパー:**

`buildEngineActualSummaryDisplay(metrics, actualRules)` を `internalBacktestResultViewModel.ts` に追加。
`EngineActualSummaryDisplay` 型（`tradeCount / winRatePct / totalReturnPct / maxDrawdownPct / holdingAvgBars / firstTradeAt / lastTradeAt / rulePatternLabel`）を返す。

---

### 8.8.5 実装反映（最小追加）: engine_actual rule preset 選択フォーム

`StrategyVersionDetail` の内製バックテスト実行フォームに、`engine_actual` モード選択時に
rule preset を選んで実行できる最小 UI を追加した（read-only 比較 UI の実行導線）。

**追加した入力フォーム:**

| 入力要素 | data-testid | 説明 |
|---|---|---|
| モード切替 | `summary-mode-select` | `engine_estimated` / `engine_actual` 選択 |
| preset 選択 | `engine-actual-preset-select` | 3択 preset を選ぶ（engine_actual 時のみ表示） |
| period 入力 | `engine-actual-period-input` | sma_cross 選択時のみ表示（整数 2〜200） |
| threshold 入力 | `engine-actual-threshold-input` | threshold_cross 選択時のみ表示（正の数） |
| validation エラー | `engine-actual-form-error` | frontend validation 失敗時に表示 |

**preset 一覧:**

| preset | 追加入力 | entry_rule | exit_rule |
|---|---|---|---|
| `default_previous_close` | なし | `close_above_previous_close` | `close_below_previous_close` |
| `sma_cross` | `period` (2〜200) | `price_above_sma` | `price_below_sma` |
| `threshold_cross` | `threshold` (正の数) | `price_above_threshold` | `price_below_threshold` |

**frontend validation:**

- `sma_cross`: period 未入力 or 範囲外 → エラー表示・実行ブロック
- `threshold_cross`: threshold 未入力 or 0以下 → エラー表示・実行ブロック
- backend `400 VALIDATION_ERROR` は既存 API error ハンドリングで表示（変更なし）

**実行ボタン表示:** `内製バックテストを開始 (engine_actual)` のように summaryMode を明示。

---

### 8.8.6 実装反映（最小追加）: 保存済み execution 条件の preset 復元

`StrategyVersionDetail` で `engine_actual` 実行結果（`status = succeeded` かつ `summary_kind = engine_actual`）を表示中に、
同画面内の実行フォームへ条件を復元して再実行できる最小導線を追加した。

**追加 action:**

- `この条件で再実行`（同画面内、実行フォームへ復元）

**復元対象（最小）:**

- `summary_mode = engine_actual`
- preset 種別（`default_previous_close` / `sma_cross` / `threshold_cross`）
- `period` または `threshold`
- 可能な場合は `execution_target.symbol`
- 可能な場合は `data_range.from/to`

**復元元:**

- `input_snapshot.engine_config.actual_rules`
- `input_snapshot.execution_target`
- `input_snapshot.data_range`

**復元不能時の扱い（安全側）:**

- `actual_rules` が既存 preset へ対応づけできない場合は復元ボタンを出さず、
  `この execution のルール条件は preset 復元できません。` を表示
- 推測変換は行わない

### 8.8.8 実装反映（最小追加）: 元 execution vs 再実行 execution 比較UI

`StrategyVersionDetail` で `engine_actual` の「この条件で再実行」を使った直後に、元 execution と再実行 execution を同一セクション内で最小比較できる read-only UI を追加した。

- 表示条件:
  - 元 execution / 再実行 execution の両方が `status = succeeded`
  - 両方の `result_summary.summary_kind = engine_actual`
- 比較項目（最小）:
  - `rule`
  - `trade_count`
  - `win_rate`
  - `total_return_percent`
  - `max_drawdown_percent`
- 既存状態分岐維持:
  - `success(with trades)` / `success(no-trade)` / `RESULT_NOT_READY` / `NOT_FOUND` / `fetch error` は従来どおり
  - `engine_estimated` 側には比較UIを表示しない
