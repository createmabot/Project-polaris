# 北極星 開発着手用 README / セットアップ手順書（MVP）

更新日: 2026-03-18

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVP開発に着手するメンバー向けに、**ローカル開発環境を立ち上げ、最低限の動作確認まで進めるための手順** をまとめたものである。

北極星は、以下の要素をまたぐ構成になっている。

- フロントエンド
    
- バックエンド API
    
- DB
    
- 非同期ジョブ
    
- AI 要約 / 生成
    
- TradingView webhook 連携
    
- CSV 取込
    

そのため、単に `npm install` や `npm run dev` だけでは開発を開始しにくい。  
本資料では、開発開始時に必要な準備、ローカルセットアップ、seed 投入、疎通確認、代表的なトラブル対処までを一貫して整理する。

## 3. 北極星MVPの前提

北極星MVPは、以下の3コンセプトを実現することを目標にしている。

- 毎日5分で状況把握
    
- 銘柄研究の高速化
    
- ルール運用の検証
    

役割分担は以下。

- TradingView: 表示、監視、一次検証
    
- 北極星: 要約、比較、仮説、履歴管理
    

開発時には、次の3本の縦導線が重要である。

1. TradingView alert → 北極星受信 → AI要約 → ホーム
    
2. 銘柄詳細 → references → AI論点 → ノート
    
3. 自然言語 → Pine → CSV取込 → backtest 総評

## 7. 環境変数

## 7. 環境変数

北極星はプロジェクトルートの `.env` を唯一の正本として扱います。

```env
APP_ENV=development
PORT=3000

DATABASE_URL="postgresql://polaris:password@localhost:5432/polaris"
REDIS_URL="redis://localhost:6379"

LOCAL_LLM_ENDPOINT=http://localhost:11434  
PRIMARY_LOCAL_MODEL=qwen3-30b-a3b-2507  
FALLBACK_API_KEY=  
MAX_LOCAL_RETRY_COUNT=2

WEBHOOK_TOKEN=test-webhook-token-123
```

### 補足

- `DB_URL` を使う場合は個別 DB 項目とどちらを正とするか決める
    
- `WEBHOOK_SHARED_SECRET` は使う場合のみ設定する
    
- AI model 名はコード上のデフォルトがあるなら省略可能

## 13. webhook ローカル確認

## 13.1 まず fixture で試す

最初から TradingView 実機確認に行かず、fixture で疎通する。

- 受信時に owner_user_id が解決される
    
- `received_at` と `triggered_at` が別に保存される
    
- UTC 保存 / JST 表示の変換が崩れない
    
- `webhook_receipts` と `alert_events` の両方に痕跡が残る
### 代表 fixture

- 正常 JSON payload
    
- token 不正 payload
    
- duplicate payload
    
- symbol 未解決 payload
    
- plain text payload
    

## 13.2 curl で送る例

```bash
curl -X POST "http://localhost:8000/api/integrations/tradingview/webhook?token=tv_local_dev_token" \
  -H "Content-Type: application/json" \
  -d '{
  "provider": "tradingview",
  "alert_type": "technical",
  "alert_name": "MA25 breakout",
  "tradingview_symbol": "TSE:7203",
  "symbol": "7203",
  "market_code": "TSE",
  "timeframe": "D",
  "triggered_at": "2026-03-11T00:15:00Z",
  "trigger_price": 3021.5,
  "currency": "JPY",
  "message": "7203 triggered technical alert",
  "condition_summary": "close crossed above MA25",
  "metadata": {
    "volume": 18450000
  }
  }'
```

## 13.3 期待結果

-  200 が返る
    
-  alert_event が作成される
    
-  symbol_id が解決される
    
-  raw payload が保存される
    
-  ai_job enqueue まで進む（実装済みなら）
    

## 13.4 token 不正確認

-  無効 token で 401 になる
    

## 13.5 重複確認

## 15. AI疎通確認

## 15.1 ローカルLLM最小確認

通常処理の正本であるローカルQwen3系モデルとの疎通確認を先に行う。

以下を確認する。

- ローカルLLM endpoint に接続できる
- `PRIMARY_LOCAL_MODEL` で代表ジョブが1件成功する
- ai_jobs に status が残る
- ai_summaries または対応する生成結果が保存できる
- schema バリデーションが通る

## 15.2 fallback API 最小確認

fallback API は常用確認ではなく、例外時にのみ使う前提で確認する。

以下を確認する。

- `FALLBACK_API_KEY` が有効
- fallback 用モデルで代表ジョブが1件成功する
- fallback 実行時に使用モデルがログへ残る
- escalation_reason がログへ残る

## 15.3 モデル選択ロジック確認

以下を確認する。

- 通常時はローカルモデルが使われる
- Pine コンパイルエラー時に fallback 判定へ進める
- 2回以上の修正ループ失敗時に fallback 判定へ進める
- 制約が多い入力で escalation_conditions が評価される
- 最終提出用高精度版の明示指定で fallback を使える

## 15.4 注意

MVPでは AI の文面品質よりも、まず以下を優先する。

- JSON schema に合うか
- structured_json を保存できるか
- insufficient_context を表現できるか
- 初回実行モデル / 最終採用モデル / retry_count / escalation_reason が追えるか

## Prisma drift troubleshooting（追加）

ローカルセットアップ時に `prisma migrate dev` で drift が出た場合は以下を順に実施する。

1. `cd backend && npx prisma validate`
2. `cd backend && npx prisma migrate status`
3. `cd backend && npx prisma migrate dev`
4. `cd backend && npx prisma migrate deploy`

`The migration "..." was modified after it was applied` が出る場合:

- 原則: repo の `schema.prisma` + `prisma/migrations/*` を正本として扱う
- ローカル開発DBは再作成して整合を回復する

```bash
cd backend
npx prisma migrate reset --force --skip-seed
npx prisma migrate status
npx prisma migrate dev --name verify_noop
npx prisma migrate deploy
```

注意:
- `migrate resolve` は最終手段にする（通常は reset + 再適用を優先）
- reset 実行前に、消してよいローカルデータであることを確認する
Prisma 7 設定運用メモ（追加）:
- Prisma の設定正本は backend/prisma.config.ts を使用する。
- package.json#prisma は Prisma 7 で廃止予定のため利用しない。
