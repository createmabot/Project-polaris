# 北極星 バックテスト結果比較ロジック設計（MVP）

更新日: 2026-03-09

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」における **バックテスト結果比較ロジック** を定義するものである。

北極星では、TradingView を一次検証基盤として利用し、バックテスト結果を CSV で取り込んだうえで、北極星側に保存・比較・要約する方針を採用している。

単に 1 回のバックテスト結果を見るだけではなく、以下のような比較ができることが重要である。

- 同一戦略のバージョン比較
    
- 同一戦略の銘柄別比較
    
- 同一戦略の時間足別比較
    
- 同一戦略の期間別比較
    
- 類似戦略同士の比較
    

そのため本仕様では、MVP段階で実装すべき比較単位、比較指標、比較ルール、表示ロジック、AI総評との接続を整理する。

---

## 2. この機能の役割

バックテスト結果比較機能の役割は以下。

1. 複数の backtest_run を並べて比較する
    
2. どの戦略・条件が優れているかを判断しやすくする
    
3. 改善前後の差分を可視化する
    
4. AI総評に使える比較用入力を生成する
    
5. ユーザーが「次に何を改善すべきか」を考えやすくする
    

---

## 3. MVPで扱う比較の範囲

MVPでは、比較対象を **backtest_run 単位** とする。

### 3.1 比較可能なパターン

- 同一 strategy_rule の複数 version
    
- 同一 version の複数 symbol
    
- 同一 version の複数 timeframe
    
- 同一 version の複数期間
    
- 任意の 2〜5 件の backtest_run
    

### 3.2 MVPで比較対象外にするもの

- 数百件単位の大規模ランキング
    
- 自動最適化結果の網羅比較
    
- ポートフォリオ単位の組み合わせ比較
    
- Monte Carlo やウォークフォワード分析の比較
    

---

## 4. 比較の基本方針

### 4.1 比較の単位は backtest_run を基本にする

比較対象は最終的に `backtest_run` とする。

理由:

- strategy version
    
- symbol
    
- timeframe
    
- 期間
    
- assumptions  
    がすべてその run に紐づくため。
    

### 4.2 「良い / 悪い」を単一指標で決めない

MVPでも、単純に net profit の大小だけで順位付けしない。

最低限、以下を分けて見る。

- 収益性
    
- 安定性
    
- 取引頻度
    
- 実運用しやすさ
    

### 4.3 比較は「数値差分」と「解釈」の2層に分ける

- 数値差分: 機械的に計算
    
- 解釈: AI総評やユーザー判断で使う
    

### 4.4 まずは pairwise 比較を中心にする

MVPでは 2 件比較を最重要とし、複数比較は表形式で拡張可能にする。

## 9. 指標の方向性定義

比較ロジックでは、各指標について「大きい方が良い」か「小さい方が良い」かを明示する必要がある。

## 9.1 高い方が良い

- net_profit
    
- gross_profit
    
- profit_factor
    
- win_rate
    
- avg_trade_profit
    
- best_trade
    

## 9.2 低い方が良い

- gross_loss（絶対値比較時は注意）
    
- max_drawdown（絶対値の小ささが望ましい）
    
- avg_bars_in_trade（用途次第だが、MVPでは参考値扱い）
    

## 9.3 単純な優劣をつけにくい

- total_trades
    
- avg_bars_in_trade
    
- long_trade_count / short_trade_count
    

これらは戦略の性質を示す指標として扱い、優劣の決定打にはしすぎない。

## 10. 差分計算ルール

## 10.1 基本差分

2 件比較では、各指標について以下を計算する。

- absolute_diff = A - B
    
- relative_diff_percent = ((A - B) / abs(B)) * 100
    

ただし B=0 の場合は relative_diff_percent は null とする。

## 10.2 drawdown の扱い

`max_drawdown` は通常マイナス値または絶対値で扱われうる。  
MVPでは比較前に正規化する。

### 方針

- DB保存時点で「負の値」でもよい
    
- 比較ロジックでは絶対値を用いた `drawdown_size` を内部計算して比較する
    

例:

- A = -8%
    
- B = -12%
    

→ A の方が良い（浅い）

## 10.3 win_rate の扱い

MVPでは 0〜100 の percent 値で保持する。

---

## 11. pairwise 比較ロジック

2 件比較では、各指標について以下を返す。

```json
{
  "metric": "profit_factor",
  "a_value": 1.42,
  "b_value": 1.18,
  "absolute_diff": 0.24,
  "relative_diff_percent": 20.34,
  "winner": "a",
  "interpretation": "Aの方が優位"
}
```

## 11.1 winner 判定

- 高い方が良い指標: 値が高い方
    
- 低い方が良い指標: 値が低い方
    
- 差が十分小さい場合: `tie`
    

## 11.2 tie のしきい値

MVPでは簡易しきい値を持つ。

例:

- 差分率 1% 未満なら tie 候補
    
- ただし total_trades のような整数値は別扱いも可
    

---

## 12. 複数 run 比較ロジック

3 件以上の比較では、pairwise の全組み合わせではなく、  
表形式と簡易ランキングを返す。

## 12.1 出力方針

- 各 run を行に並べる
    
- 各指標を列に並べる
    
- 指標ごとに best / worst を付与する
    

## 12.2 例

```json
{
  "metric": "profit_factor",
  "best_run_id": "run_2",
  "worst_run_id": "run_5",
  "ranked_run_ids": ["run_2", "run_1", "run_3"]
}
```

## 12.3 MVPでのランキング方針

単一総合スコアによる完全順位付けは行わず、指標ごとの順位を出す。

理由:

- 戦略比較では指標間トレードオフが大きい
    
- 総合スコアは誤解を生みやすい
    

---

## 13. 比較スナップショット構造

MVPでは比較結果を保存可能にするため、比較用 JSON を構築する。

## 13.1 pairwise 比較結果の例

```json
{
  "comparison_type": "pairwise",
  "run_ids": ["run_a", "run_b"],
  "metrics": [
    {
      "metric": "net_profit",
      "a_value": 120000,
      "b_value": 95000,
      "absolute_diff": 25000,
      "relative_diff_percent": 26.31,
      "winner": "a"
    },
    {
      "metric": "max_drawdown",
      "a_value": -8.2,
      "b_value": -12.5,
      "absolute_diff": 4.3,
      "relative_diff_percent": 34.4,
      "winner": "a"
    }
  ],
  "summary_flags": {
    "a_has_better_return": true,
    "a_has_better_drawdown": true,
    "tradeoff_detected": false
  }
}
```

## 13.2 multi-run 比較結果の例

```json
{
  "comparison_type": "table",
  "run_ids": ["run_1", "run_2", "run_3"],
  "metric_rows": [
    {
      "metric": "profit_factor",
      "values": {
        "run_1": 1.12,
        "run_2": 1.42,
        "run_3": 1.31
      },
      "best_run_id": "run_2",
      "worst_run_id": "run_1"
    }
  ]
}
```

## 14. トレードオフ検出ロジック

北極星では単純な勝敗だけでなく、トレードオフを明示する。

## 14.1 代表的なトレードオフ

- net_profit は高いが drawdown も大きい
    
- profit_factor は高いが trade 数が少ない
    
- win_rate は高いが avg_trade_profit が低い
    

## 14.2 MVPでの検出ルール

以下のようなルールベースでよい。

### パターンA

- A の net_profit が B より高い
    
- かつ A の max_drawdown が B より悪い
    

→ `tradeoff_detected = true`

### パターンB

- A の profit_factor が高い
    
- かつ total_trades が極端に少ない
    

→ `sample_size_warning = true`

### パターンC

- win_rate は高い
    
- だが worst_trade が極端に悪い
    

→ `tail_risk_warning = true`

## 15.1 AIに渡す主な材料

- run 文脈情報
    
- 主要指標の差分
    
- best / worst
    
- tradeoff flags
    

## 15.2 生成したい総評例

- どちらが収益面で有利か
    
- どちらが安定性で有利か
    
- どの点が改善余地か
    
- 実運用向きなのはどちらか
    

## 15.3 schema 接続

比較総評は将来的に AI schema を追加して扱う。  
MVPでは optional にしてもよい。

## 19. バリデーション方針

## 19.1 比較対象数

- 最低2件必要
    
- 最大5件程度を推奨
    

## 19.2 指標不足

- backtest_metrics が無い run は比較不可
    
- trade_summaries が無い場合は trade 系指標を null にして比較継続可
    

## 19.3 同一 run 重複

- 同じ run_id の重複選択は不可
    

## 19.4 異種比較

- symbol や timeframe が異なっても比較は可能
    
- ただし UI / AI総評で「前提差あり」を明示する
    

---

## 20. 典型的な比較ケース

## 20.1 ルール改善前後比較

- version 1 vs version 2
    
- 同一 symbol / timeframe / period
    

最も重要な MVP ケース。

## 20.2 同一戦略の銘柄横比較

- 同一 version
    
- symbol だけ変更
    

ロジックの汎用性を見るケース。

## 20.3 同一戦略の時間足比較

- 同一 version
    
- D vs 60 など
    

MVPでも扱えるが、前提差の明示が重要。

## 20.4 同一戦略の期間比較

- bull 相場期 vs range 相場期
    

MVPでは period 違いとして扱う。

---

## 21. 例外ケースの扱い

## 21.1 total_trades が極端に少ない

例:

- 1〜2回しかトレードがない
    

扱い:

- sample_size_warning を出す
    
- AI総評でも信頼性注意を述べる
    

## 21.2 profit_factor が異常値

例:

- gross_loss = 0 で極端に高い
    

扱い:

- raw 値は保持
    
- UI では注意表示
    
- winner 判定に使う際は警告フラグを立てる
    

## 21.3 drawdown 欠損

扱い:

- drawdown 比較はスキップ
    
- comparison_result に missing_metrics を残す

## 22. 設計上の重要判断

### 22.1 単一総合スコアで順位付けしない

MVPでは誤解を避けるため、指標ごとの差分比較を重視する。

### 22.2 pairwise を最優先にする

ユーザーが実際にやりたいのは、改善前後比較であることが多いため。

### 22.3 tradeoff を明示する

収益と安定性の両立が重要であり、単純な winner 表示だけでは不十分。

### 22.4 trade 系指標は補助扱いにする

MVPでは明細より summary ベースなので、あくまで補助に留める。

### 22.5 異種前提の比較を禁止しない

時間足や期間が違っても、比較ニーズ自体はあるため。ただし注記を必須にする。

## 25. 結論

北極星MVPのバックテスト結果比較ロジックでは、比較対象を **backtest_run** とし、  
**収益性・安定性・取引特性・文脈情報** を分けて比較する方針を採用する。

そのうえで、MVPでは以下を重視する。

- pairwise 比較を最優先にする
    
- net profit, profit factor, max drawdown, win rate, total trades を主要指標にする
    
- 単一総合スコアではなく指標ごとの差分を示す
    
- tradeoff を明示する
    
- AI総評は比較スナップショットを入力にして生成する
    

この設計により、北極星の「自然言語→バックテスト→比較→改善」の流れを支える比較基盤を構築する。
