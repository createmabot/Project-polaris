# 北極星 normalized_rule_json schema 詳細設計（MVP）

更新日: 2026-03-09

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」における **normalized_rule_json** の詳細 schema を定義するものである。

北極星では、自然言語で入力された売買ルールをそのまま Pine Script に変換せず、いったん **中間表現** に正規化したうえで Pine 生成へ進める方針を採用している。

この `normalized_rule_json` は、以下の役割を担う。

- 自然言語ルールの正本
    
- Pine 生成の入力
    
- 再現性の担保
    
- warnings / assumptions の発生箇所の特定
    
- 将来の別検証エンジン対応の土台
    

---

## 2. 設計方針

### 2.1 自然言語の曖昧さを吸収しすぎない

`normalized_rule_json` は、自然言語の意味を勝手に拡張する場ではない。

- 解釈できたことを明示的に構造化する
    
- 解釈に default を使った場合は別途記録する
    
- 未対応表現は捨てずに metadata 側へ残す
    

### 2.2 Pine に近すぎず、自然言語にも寄りすぎない

中間表現は Pine の AST そのものではない。  
一方で、人間の文章に近すぎても再利用しにくい。  
そのため、

- 条件カテゴリ
    
- 比較対象
    
- 演算子
    
- 閾値
    
- 論理結合  
    を明示する構造を採る。
    

### 2.3 MVPでは対応範囲を絞る

MVPでは、

- ロング主体
    
- 単一銘柄
    
- 単一時間足
    
- シンプルな entry / exit / TP / SL  
    に寄せる。
    

### 2.4 schema は「読みやすさ」と「生成しやすさ」を両立する

- 人間が読んで確認しやすい
    
- AI / コード生成器が Pine 化しやすい
    

### 2.5 schema_version を持つ

将来拡張に備え、schema version を定義する。

---

## 3. schema の全体構造

MVPにおける `normalized_rule_json` のトップレベルは以下とする。

```json
{
  "schema_name": "normalized_rule",
  "schema_version": "1.0",
  "strategy_type": "long_only",
  "market": "JP_STOCK",
  "timeframe": "D",
  "entry": {},
  "exit": {},
  "execution": {},
  "metadata": {}
}
```

---

## 4. トップレベルキー定義

## 4.1 必須キー

- `schema_name`
    
- `schema_version`
    
- `strategy_type`
    
- `market`
    
- `timeframe`
    
- `entry`
    
- `exit`
    
- `execution`
    
- `metadata`
    

---

## 4.2 各キーの意味

### schema_name

固定値: `normalized_rule`

### schema_version

MVPでは `1.0`

### strategy_type

戦略の売買方向・構造を表す。

候補:

- `long_only`
    
- `short_only`
    
- `long_short`
    

MVP標準は `long_only`

### market

対象市場。

候補例:

- `JP_STOCK`
    
- `US_STOCK`
    
- `FX`
    
- `CRYPTO`
    

### timeframe

対象時間足。

例:

- `1`
    
- `5`
    
- `15`
    
- `60`
    
- `D`
    
- `W`
    

### entry

エントリー条件定義

### exit

イグジット条件定義

### execution

バックテスト実行前提の補助情報

### metadata

変換時の assumptions / warnings / omitted などを保持

---

## 5. entry schema

## 5.1 役割

エントリー条件を表す。  
MVPでは、entry は **論理結合 + 条件配列** を基本とする。

## 5.2 schema

```json
{
  "enabled": true,
  "logic": "and",
  "conditions": []
}
```

## 5.3 必須キー

- `enabled`
    
- `logic`
    
- `conditions`
    

## 5.4 logic の候補

- `and`
    
- `or`
    

MVPでは入れ子の論理木までは扱わず、トップレベル 1 段に限定する。

## 5.5 conditions

条件オブジェクト配列。  
1件以上を推奨する。

---

## 6. exit schema

## 6.1 役割

ポジション解消条件を表す。

MVPでは以下の 3 種類を分ける。

- close_conditions: 条件成立でクローズ
    
- take_profit_percent: 利確率
    
- stop_loss_percent: 損切率
    

## 6.2 schema

```json
{
  "close_logic": "or",
  "close_conditions": [],
  "take_profit_percent": null,
  "stop_loss_percent": null,
  "trailing_stop_percent": null
}
```

## 6.3 必須キー

- `close_logic`
    
- `close_conditions`
    
- `take_profit_percent`
    
- `stop_loss_percent`
    
- `trailing_stop_percent`
    

### 補足

MVPでは `trailing_stop_percent` は原則 `null` を想定し、未対応でもキーは残す。

---

## 7. execution schema

## 7.1 役割

バックテスト実行や生成時に必要な補助設定を保持する。

## 7.2 schema

```json
{
  "commission_percent": 0.1,
  "slippage": 0.05,
  "period_from": "2024-01-01",
  "period_to": "2025-12-31",
  "position_sizing": {
    "mode": "fixed",
    "value": 1
  }
}
```

## 7.3 必須キー

- `commission_percent`
    
- `slippage`
    
- `period_from`
    
- `period_to`
    
- `position_sizing`
    

## 7.4 position_sizing

MVPでは厳密な資金管理は扱わないため、簡略化する。

候補:

- `mode`: `fixed`
    
- `value`: 1
    

### 補足

MVPでは「1回あたり1ユニット」のような抽象化で十分。  
Pine 側で実運用向け sizing まで行わない。

---

## 8. metadata schema

## 8.1 役割

自然言語→正規化の過程で生じた補足情報を保持する。

## 8.2 schema

```json
{
  "source_text": "25日移動平均線の上で、RSIが50以上なら買い。",
  "assumptions": [],
  "warnings": [],
  "omitted_expressions": [],
  "unsupported_expressions": [],
  "parser_version": "1.0"
}
```

## 8.3 必須キー

- `source_text`
    
- `assumptions`
    
- `warnings`
    
- `omitted_expressions`
    
- `unsupported_expressions`
    
- `parser_version`

## 9. condition object 共通仕様

entry.conditions / exit.close_conditions に入る条件オブジェクトは、共通の基本形を持つ。

## 9.1 共通形

```json
{
  "id": "cond_1",
  "type": "indicator_compare",
  "enabled": true,
  "subject": {},
  "operator": "above",
  "object": {},
  "note": null
}
```

## 9.2 必須キー

- `id`
    
- `type`
    
- `enabled`
    
- `subject`
    
- `operator`
    
- `object`
    
- `note`
    

## 9.3 id

同一ルール内で一意。  
生成時や UI 表示で使う。

## 9.4 enabled

将来の条件 ON/OFF を見据えたフラグ。  
MVPでは基本 true。

## 9.5 note

人間向け補足。MVPでは null 可。

## 10. condition type 一覧（MVP）

MVPでは以下の type を正式対応とする。

1. `indicator_compare`
    
2. `indicator_threshold`
    
3. `price_breakout`
    
4. `volume_multiple`
    
5. `cross_event`
    
6. `percent_take_profit`
    
7. `percent_stop_loss`
    

※ 6,7 は মূলतः exit 専用だが、表現上の統一のため type 名として定義してよい。

---

## 11. type: indicator_compare

## 11.1 用途

価格と移動平均線の比較など。

例:

- 終値が25日線より上
    
- 終値がEMA20より下
    

## 11.2 schema

```json
{
  "id": "cond_1",
  "type": "indicator_compare",
  "enabled": true,
  "subject": {
    "kind": "series",
    "source": "close"
  },
  "operator": "above",
  "object": {
    "kind": "indicator",
    "name": "sma",
    "params": {
      "source": "close",
      "length": 25
    }
  },
  "note": null
}
```

## 11.3 operator 候補

- `above`
    
- `below`
    
- `>=`
    
- `<=`
    

MVPでは `above` / `below` を優先採用する。

## 21. logical 制約

## 21.1 entry.conditions

- 0件は原則不可
    
- 1件以上必要
    

## 21.2 exit.close_conditions

- 0件でも可
    
- ただしその場合、`take_profit_percent` または `stop_loss_percent` の少なくとも一方を持つことを推奨
    

## 21.3 strategy_type

- `long_only` の場合、short 用条件は含めない
    
- MVPでは long / short 条件の別枝は持たない
    

## 21.4 timeframe

- 単一値のみ
    
- 複数時間足は MVP 非対応

## 23. バリデーション方針

## 23.1 トップレベル

- `schema_name == normalized_rule`
    
- `schema_version` 必須
    
- `strategy_type` は列挙値内
    
- `market` 必須
    
- `timeframe` 必須
    

## 23.2 entry

- `enabled` は boolean
    
- `logic` は and/or
    
- `conditions` は配列
    
- 少なくとも 1 条件推奨
    

## 23.3 exit

- `close_logic` は and/or
    
- `close_conditions` は配列
    
- TP/SL は number or null
    

## 23.4 condition object

- `id` 必須
    
- `type` は正式対応 type に含まれる
    
- `subject.kind` / `object.kind` が type に対して妥当
    
- `operator` が type に対して妥当
    

## 23.5 metadata

- arrays は必ず配列
    
- `source_text` は string
    
- `parser_version` 必須

## 24. 変換不能・部分変換の表現

## 24.1 部分変換可能な場合

例:

- 「強い銘柄だけ買うが、25日線の上でRSI50以上」
    

扱い:

- 25日線と RSI 条件は schema 化
    
- 「強い銘柄だけ」は `unsupported_expressions` に残す
    

## 24.2 変換不能な場合

例:

- 裁量語のみで構成され、条件が具体化できない
    

扱い:

- `normalized_rule_json` 自体を作らない、または failed とする
    
- どの表現が原因かを metadata 相当のエラー情報で返す

## 27. 設計上の重要判断

### 27.1 entry と exit を明示的に分ける

自然言語の読みやすさより、戦略構造の明瞭さを優先する。

### 27.2 条件をオブジェクトで持つ

文字列化しすぎると Pine 生成ロジックが不安定になるため。

### 27.3 metadata を schema に含める

変換時の assumptions / warnings が再現性に重要だから。

### 27.4 TP / SL は close_conditions と分ける

Pine 生成時に `strategy.exit()` と対応づけやすい。

### 27.5 MVPでは論理木のネストを持たない

複雑さを抑えるため、トップレベル 1 段の and/or に限定する。

## 30. 結論

北極星MVPの `normalized_rule_json` schema は、自然言語ルールを **再現可能・検証可能・確認可能** な形に落とすための中間表現である。

そのため、MVPでは以下を採用する。

- トップレベルに strategy_type / market / timeframe / entry / exit / execution / metadata を持つ
    
- entry / exit は条件オブジェクト配列で表現する
    
- TP / SL は明示フィールドで分離する
    
- assumptions / warnings / unsupported を metadata に残す
    
- MVPではロング主体・単一時間足・単純論理に限定する
    

この設計により、北極星の「自然言語→Pine→バックテスト→レポート化」の中核となるルール正規化層を安定して運用できる状態を目指す。
