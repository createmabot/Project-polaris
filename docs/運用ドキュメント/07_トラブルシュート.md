# 北極星 トラブルシュート

更新日: 2026-05-25
分類: 運用ドキュメント

## 1. 目的

本資料は、ローカル開発・確認時によくある失敗の切り分け入口である。機能別の詳細 runbook がある場合は、該当 docs を優先する。

## 2. 起動できない

確認順:

1. PostgreSQL / Redis が起動しているか確認する。
2. `.env` が `.env.example` から作られているか確認する。
3. `pnpm install` が完了しているか確認する。
4. backend / frontend の dev process が既に起動して port を占有していないか確認する。
5. DB migration と Prisma generate が実行済みか確認する。

必要なら次をやり直す。

```bash
pnpm run down
pnpm run up
cd backend
pnpm exec prisma validate
pnpm exec prisma migrate deploy
pnpm exec prisma generate
pnpm exec prisma db seed
```

## 3. Prisma / DB で詰まる

- drift が出た場合は、repo の Prisma schema と migrations を正本として扱う。
- Windows で `query_engine-windows.dll.node` の file lock が出る場合は、backend / frontend dev process を止めてから再実行する。
- seed が失敗する場合は、migration 適用状況と DB 接続先を確認する。
- 本番相当データを local troubleshooting に使わない。

## 4. 日本語 JSON が文字化けする

PowerShell から日本語 JSON を送る場合は、文字列 body ではなく UTF-8 byte array に変換して送る。詳細は `docs/walkthrough.md` の PowerShell 手順を参照する。

確認観点:

- `Content-Type` に `charset=utf-8` を付ける。
- `-Body` には UTF-8 byte array を渡す。
- UI 経由の日本語入力と shell 経由の encoding 問題を分けて扱う。

## 5. TradingView webhook が届かない / 落ちる

確認順:

1. webhook URL が外部から到達可能か確認する。
2. token の渡し方を確認する。
3. `trigger_price` が number になっているか確認する。
4. `webhook_receipts` の auth / parse / symbol resolution / dedupe を確認する。
5. `alert_events` と `ai_jobs` を確認する。

詳細は `docs/運用ドキュメント/03_TradingView_webhook運用.md` と `docs/32.北極星 TradingView実送信 webhook運用手順（MVP）.md` を参照する。

## 6. CSV import が failed になる

確認順:

1. 対応形式の CSV か確認する。
2. header と data row があるか確認する。
3. required columns が揃っているか確認する。
4. `parse_error` と補助説明文を読む。
5. 最新 import が failed でも、過去 parsed import が残っていないか確認する。

詳細は `docs/運用ドキュメント/04_CSV取込運用.md` を参照する。

## 7. AI summary が unavailable / failed になる

確認順:

1. provider 設定を確認する。
2. local LLM を使う場合は provider process が起動しているか確認する。
3. `AI_ENABLE_STUB_FALLBACK` の設定を確認する。
4. `ai_jobs` の status と sanitized error を確認する。
5. 同一 input の failed job がある場合、自動 retry ではなく手動生成または別タスクで判断する。

詳細は `docs/運用ドキュメント/05_AI_provider運用.md` を参照する。

## 7-1. Pine 生成結果の警告 / 前提が英語で見える

Pine generation provider には warnings / assumptions をユーザー向け日本語文で返すよう prompt で要求する。deterministic generator と post-processing が追加する warning / assumption も日本語文にする。provider 自由文は後段固定 mapping や翻訳 provider で変換しないため、英語文が残る場合は表示変換の不足ではなく provider prompt compliance / provider quality の確認対象として扱う。

確認観点:

1. StrategyLab と StrategyVersionDetail の表示で、`警告` / `前提` が日本語文言になっているか確認する。
2. `Leading explanatory text was removed from generated script.` などに相当する post-processing warning は、backend が追加する時点で日本語文になっているか確認する。
3. provider 由来の自由文が英語で残る場合は、provider prompt / schema instruction が warnings / assumptions の日本語出力を十分に要求しているか確認する。外部翻訳 API、LLM 翻訳、固定 mapping 追加では対応しない。
4. generated Pine script 本文は localization で変更しない。
5. raw prompt、raw provider response、endpoint、model 実値、secret、local path、stack trace は確認・保存しない。

詳細は `docs/仕様書/11_LLM_Strategy_Proposal仕様.md` の Pine generation market / timeframe 境界を参照する。

## 7-2. Pine generation LLM-first が failed になる

Pine generation は `PINE_GENERATION_PROVIDER=local_llm|deterministic|openai_api` で切り替える。既定は `local_llm` の LLM-first path であり、generator -> reviewer -> repair pipeline で output を既存 normalization / 最小 validation / bounded repair に通してから保存する。deterministic generator は baseline / emergency fallback / test fixture 用であり、LLM 生成品質の主経路としては扱わない。`openai_api` は明示 opt-in / cost guard 設計後の後続候補として扱う。

job-based progress を使う場合は、start endpoint が返す `job_id` を status polling endpoint に渡し、queued / running / succeeded / failed と backend stage を確認する。polling は status 確認だけであり、SSE / WebSocket / streaming ではない。既存 sync endpoint も互換維持のため残る。

確認順:

1. `PINE_GENERATION_PROVIDER` と `AI_ENABLE_STUB_FALLBACK` の設定を確認する。
2. local LLM を使う場合は provider process が起動しているか確認する。ただし endpoint / model 実値は docs、PR、screenshot、log に残さない。
3. API response / UI の `failure_reason`、`repair_attempts`、`invalid_reason_codes`、warnings を確認する。
4. job-based progress では `job_id`、`status`、現在 stage、完了時の sanitized Pine response を確認する。failed job では sanitized `invalid_reason_codes` と `pine_reviewer_issues` が返る場合があるため、reviewer repair exhausted の切り分けに使う。raw prompt / raw provider response / raw reviewer response / endpoint / model 実値は確認・保存しない。
5. `missing_version_declaration` など retryable な invalid output や reviewer issue が bounded repair に回ったか確認する。repair は最大 2 回で、無限 retry は行わない。
6. `empty_output`、`missing_strategy_or_indicator_declaration`、provider timeout / unavailable は failed として扱い、必要ならユーザー操作で再生成する。
7. TradingView compile error は北極星が自動取得しない。TradingView で確認した error / note を `compile_error_text`、`validation_note`、`revision_request` として Pine 修正再生成に手動入力する。
8. Pine generation 成功 / 失敗から Strategy 保存、backtest、AI summary が自動起動していないことを確認する。

safe strategy coding の確認:

- `strategy.entry` / `strategy.exit` が position guard 付きになっているか確認する。
- long-only / no-pyramiding 相当の `strategy.entry` が `strategy.position_size == 0` などの flat guard 配下にあるか確認する。guard がない場合は `entry_guard_risk` の reviewer issue として扱う。
- ATR stop / take profit など entry 時点で固定すべき値が、position が open になった直後の `strategy.position_size > 0 and strategy.position_size[1] == 0` pattern で `entryAtr` などの `var` 変数に保持されているか確認する。
- entry-time state の reset が単純な `strategy.position_size == 0` ではなく、open から flat への遷移を示す `strategy.position_size == 0 and strategy.position_size[1] > 0` pattern になっているか確認する。
- stop price / limit price が position open 中、かつ entry-time state が利用可能な場合だけ計算されているか確認する。
- stop loss が signal bar の `close` ではなく、actual entry price として position open 後の `strategy.position_avg_price` を基準にしているか確認する。ユーザーが明示していない `entry_price := close` は避ける。
- percentage stop の場合も、position open 中に `strategy.position_avg_price` を基準に計算されているか確認する。`entryPrice := close` や entry block 内の `entryPrice := strategy.position_avg_price` が出ている場合は prompt compliance / provider quality の確認対象にする。
- ATR を要求していない percentage stop strategy に `entryAtr` や `ta.atr` が混入していないか確認する。
- RSI / oscillator threshold の方向が維持されているか確認する。例: 「RSI が 60 を上回る」は `rsi > 60` または wording に応じた crossover であり、明示がない限り crossunder ではない。
- setup -> trigger 型の条件が、同一 bar では同時成立しない条件の単純 `and` になっていないか確認する。必要な setup は state variable で保持されていることを見る。
- `setupActive` を使う entry block では、`strategy.entry` 後に `setupActive := false` があり、同じ setup が意図せず次 bar へ残らないことを確認する。
- 「below / 下回った場合」が状態条件として扱われているか確認する。crossunder は「下抜け」「cross below」など cross wording が明示された場合だけ使う。
- `overlay=true` の価格 chart strategy で、ユーザーが明示していない oscillator plot が追加されていないか確認する。oscillator plot は明示要求または separate pane 意図がある場合だけ扱う。
- ATR stop が `strategy.exit(..., stop=...)` で表現されているか確認する。`low <= stopLossPrice` と `strategy.close()` の manual bar-based stop はユーザー明示時以外は避ける。
- Pine compile typo / unsupported plot style がないか確認する。`color.color.*` や `plot.style_dashed` が出ている場合は provider quality の確認対象にする。
- unsupported function alias がないか確認する。`ta.crossabove` / `ta.crossbelow` は `ta.crossover` / `ta.crossunder` へ補正または reviewer issue 化する。
- stop loss plot がある場合は、position open 中だけ値を返し、それ以外は `na` にするなど position / `na` guard があるか確認する。
- `strategy.exit(..., stop=stopLossPrice)` は `not na(stopLossPrice)` guard 配下にあり、plot で参照する `stopLossPrice` は outer scope で typed declaration されているか確認する。
- `strategy.position_avg_price` ベースの `stopLossPrice` は top-level や flat 中ではなく、`strategy.position_size > 0` guard 配下で計算されているか確認する。
- unused variable / unused state が増えていないか確認する。ATR を使わない戦略に `entryAtr` / `ta.atr` が残っている場合は修正対象にする。
- `strategy.close()` は stop loss order の代替ではなく、rule-based exit に使われているか確認する。
- entry block 内で `strategy.position_avg_price` から stop / limit を計算していないか確認する。
- ユーザーが明示していない volume plot が追加されていないか確認する。volume condition 自体は signal 条件として許容する。
- generated_script の comment は短い section comment だけに留まり、実装理由や provider limitation の長文説明が warnings / assumptions に出ているか確認する。
- 表現できない risk control、pyramiding、trailing、time stop、volume plot などがある場合、warnings / assumptions に日本語で限界が出ているか確認する。

reviewer pipeline の確認:

- deterministic reviewer は明らかな syntax / style / safety issue を structured issue として扱う。
- AI reviewer provider boundary を使う場合も raw reviewer response は確認・保存せず、sanitized な issue / failure 分類だけを見る。
- reviewer issue による repair 後も既存 validation / normalization が維持されているか確認する。

出してはいけないもの:

- raw prompt。
- raw provider response。
- raw reviewer response。
- provider endpoint。
- model 実値。
- secret / token / credential。
- local path。
- stack trace。

real `local_llm` / OpenAI / TradingView 実体依存の再現確認は manual smoke に限定し、required check には入れない。percentage stop / oscillator strategy の品質確認も同じ扱いとし、required check では fake provider output、deterministic fallback、reviewer issue、validation / repair classification を確認する。TradingView compile automation、自動保存、backtest / AI summary 自動連鎖は未実装として扱う。

確認順:

1. backend / frontend が起動しているか確認する。
2. seed 済みか確認する。
3. `POLARIS_E2E_BASE_URL` を指定している場合は接続先を確認する。

## 9. Strategy proposal local_llm が timeout になる

StrategyLab で `STRATEGY_PROPOSAL_PROVIDER=local_llm` を使い、UI に `provider status: timeout / reason: timeout / latency: timeout` が出る場合は、local LLM provider が応答完了前に backend の timeout guard に到達した状態として扱う。

確認順:

1. local LLM process が起動しているか確認する。
2. proposal 専用の endpoint / model / timeout / max output 設定が local env にあるか確認する。ただし実値を docs、PR、screenshot、log へ残さない。
3. default timeout profile は通常運用向けに少し余裕を持たせている。それでも重い local model や長文 context model の manual smoke で timeout する場合は、`STRATEGY_PROPOSAL_LOCAL_LLM_TIMEOUT_PROFILE=long_context` を明示し、必要な timeout を profile 上限内で設定する。
4. env を変更した後は backend / frontend dev process を再起動し、StrategyLab で `ストラテジーを提案` を再実行する。
5. UI / API response では sanitized `provider_observation.status`、`invalid_reason`、`latency_bucket` だけを見る。
6. raw prompt、raw provider response、endpoint、model 実値、secret、local path、stack trace を保存・貼付しない。
7. timeout は `required_field_missing` retry の対象外であり、backend は自動 retry / polling / silent stub fallback を行わない。

long_context profile は local_llm の timeout guard を opt-in で広げるためのものであり、openai_api、Web search、deep research、auto Pine generation、auto save を有効化しない。real local_llm 実体依存の確認は manual smoke に限定し、required check には入れない。

## 10. Strategy proposal local_llm が schema_invalid になる

StrategyLab で `STRATEGY_PROPOSAL_PROVIDER=local_llm` を使い、UI に `provider status: invalid_response / reason: schema_invalid / latency: slow` が出る場合は、local_llm から応答は返っているが `strategy_proposal_candidates` schema に通っていない状態として扱う。

確認順:

1. `STRATEGY_PROPOSAL_PROVIDER=local_llm` が明示設定されているか確認する。未指定 default は `stub`。
2. local LLM process が起動しているか確認する。
3. proposal 専用の endpoint / model / timeout / max output 設定が local env にあるか確認する。ただし実値を docs、PR、screenshot、log へ残さない。
4. backend / frontend dev process を再起動し、StrategyLab で `ストラテジーを提案` を再実行する。
5. UI / API response では sanitized `provider_observation.status`、`invalid_reason`、`latency_bucket` だけを見る。
6. raw prompt、raw provider response、endpoint、model 実値、secret、local path、stack trace を保存・貼付しない。
7. required test では real local_llm を叩かず、mock / fake response で code fence、前後説明文、root metadata 欠落、string array、enum 表記揺れ、重要 field 欠落を確認する。

現行実装の境界:

- local_llm prompt は JSON object のみ、英語 key 固定、array field は配列、`source_type=web` 不使用を要求する。
- candidate card に表示される title / summary / logic / caution / suggested spec などの string value は日本語で返すよう要求する。schema key、enum、`source_type` は英語固定値のまま扱う。
- code fence や前後説明文が混ざる軽微な response は、JSON object / array を抽出して parse する。root array 内の candidate object と nested array fields も delimiter と string escape を追跡して扱う。
- root metadata 補完、string array の配列化、enum 表記揺れの snake_case 化など安全な normalization は行う。
- common alias normalization として、`name` / `description` / `type` / `entry` / `exit` / `riskManagement` / `strengths` / `weaknesses` / `indicators` / `natural_language_spec` などを exact schema key へ寄せる。
- candidate が `candidate` / `proposal` / `strategy` wrapper 内に返った場合は、wrapper 内の field が canonical candidate shape に近いときだけ unwrap する。
- `required_field_missing` の場合は、sanitized `missing_required_fields` / count / affected candidate count だけを確認する。field value、candidate本文、raw response は保存しない。
- StrategyLab UI では `AI候補の形式が不完全だったため取得できませんでした。もう一度お試しください。` のような利用者向け message と sanitized provider status / reason / latency だけを表示する。raw response、欠落 field の値、candidate 本文は表示しない。
- `backtest_cautions`、`uncertainty`、`suggested_pine_constraints` は固定 fallback を補えるが、title / summary / entry / exit / risk / suggested natural language spec などの中核本文は backend が生成して補わない。
- required field 欠落は local_llm 内で最大 1 回だけ bounded retry する。retry prompt に raw response は入れず、missing field names と required key / non-empty scalar / array の境界だけを使う。
- title / summary / entry / exit / risk / suggested natural language spec などの重要本文が欠けている場合は、provider が usable candidate を返せていないものとして invalid response のまま扱う。
- 自動 fallback、openai_api、Web search、deep research、auto Pine generation はこの切り分け対象外。

詳細は `docs/運用ドキュメント/11_Strategy_proposal品質評価運用.md` を参照する。


## 11. Strategy proposal が rate limited になる

StrategyLab で候補取得または Codex CLI manual JSON import 時に 429 / `RATE_LIMITED` になる場合は、短時間に proposal provider call または proposal history write が続いたため、backend の in-memory rate guard が抑止した状態として扱う。

確認順:

1. ブラウザで候補取得ボタンを連打していないか確認する。
2. 少し時間をおいて再試行する。
3. reverse proxy / tunnel 配下の場合、複数利用者が同じ request IP bucket を共有していないか確認する。
4. trusted forwarded IP opt-in を使う場合は、proxy が forwarded header を上書きまたは制御していることを確認する。default では forwarded header を信頼しない。
5. forwarded header の先頭値が valid IPv4 / IPv6 ではない場合、backend は request IP に fallback する。client が任意文字列を先頭に入れられる構成では opt-in しない。
6. local dev で guard 境界を確認する場合だけ、rate guard の env 設定を見直す。実値は docs / PR / screenshot に残さない。
7. blocked request は proposal history に保存されないことを確認する。
8. Codex CLI manual import の 429 では `provider_mode=manual_import` として扱われ、blocked request は raw JSON text を parse / validation / 保存しない。
9. 429 response は retry_after / limit / window / provider mode / key source 程度の sanitized metadata に限定され、actual IP、forwarded header value、内部 key、raw prompt / raw response / raw JSON text / endpoint / model 実値は含めない。

現行の rate guard は per-process の軽量 guard であり、multi-process 環境の厳密な abuse prevention ではない。`openai_api` や Web search / deep research を導入する場合は、DB / job / user scope を含む別設計を行う。

## 12. Codex CLI manual JSON import が失敗する

StrategyLab で Codex CLI manual JSON import が失敗する場合は、Codex CLI の実行ではなく、貼り付けた JSON の parse / validation / candidate count / schema 境界を確認する。

確認順:

1. 貼り付けた text が JSON object として閉じているか確認する。
2. root が `schema_name=strategy_proposal_candidates`、`schema_version=1.0`、`input`、`candidates`、`disclaimer` を持つか確認する。
3. `candidates` が配列であり、10 件以下か確認する。
4. 各 candidate が required field を持つか確認する。
5. enum が仕様値に収まっているか確認する。
6. `research_basis.source_type=web` を使っていないか確認する。Web search / deep research は未実装である。
7. Web検索付き prompt option を使った場合でも、import schema は通常の `codex_cli_manual` と同じである。URL、citation、`source_type=web`、Web検索済み flag を JSON に追加すると validation failure になる可能性がある。
8. file import の場合も、frontend が text として読み込んだ内容を backend に送るだけであり、multipart upload や backend CLI 実行ではないことを確認する。
9. 429 / `RATE_LIMITED` の場合は、短時間の import write が rate guard で抑止された状態である。少し時間をおいて再実行し、blocked import が proposal history に保存されていないことを確認する。
10. error 表示は sanitized reason だけを見る。

出してはいけないもの:

- raw Codex output。
- Codex CLI に渡した raw prompt。
- provider endpoint。
- model 実値。
- secret / token / credential。
- local path。
- stack trace。
- candidate free text の raw dump。

import が成功した場合でも、proposal は検証候補であり投資助言ではない。候補選択は title / natural language spec 反映だけであり、Pine generation / save / backtest / AI summary は自動起動しない。

## 13. Proposal history が多くて目的の run を探しにくい

StrategyLab の提案履歴 section で、保存済み proposal run が増えて見つけにくい場合は、filter / search / pagination を使う。

確認順:

1. provider filter で `stub` / `local_llm` / `codex_cli_manual` を切り替える。
2. failed run を見たい場合は status filter を failed にする。
3. 選択済み candidate を探す場合は selected filter を selected にする。
4. candidate title / summary / strategy_type などから探す場合は search を使う。
5. 目的の run が見つからない場合は pagination の次ページを確認する。
6. detail を開いた後に候補を選択しても、title / natural language spec へ反映するだけで、Pine generation / save / backtest / AI summary は自動起動しない。

履歴 list は sanitized summary を返す。user_hint 全文、raw prompt、raw provider response、raw Codex output、endpoint、model 実値、secret、local path、stack trace、candidate 自由文本文は list に出さない。

run が見つからない場合:

- archived filter が `active` の場合、archived run は通常 list から除外される。
- `archived=archived` または `archived=all` に切り替えて確認する。
- archive は削除ではない。archived run は detail 確認と unarchive ができる。
- archived run から candidate selection をしても、自動 unarchive はしない。
- provider event log は proposal run の archive と独立しているため、event log 側の確認結果は archive filter の影響を受けない。

## 14. Strategy proposal provider event log を確認する

Strategy proposal の候補取得、Codex CLI manual import、rate limited、retry、validation failure の切り分けでは、proposal history と provider quality trend に加えて sanitized provider event log を確認する。

確認順:

1. まず UI / API response の sanitized `provider_observation`、error code、rate limited details を確認する。
2. 該当時刻付近の provider event を `event_type`、`provider_name`、`status`、`invalid_reason`、`proposal_run_id` で絞る。
3. proposal run がある event は `proposal_run_id` から history detail へ進み、候補内容は detail API / UI で確認する。
4. rate limited event は proposal run が作られないため、`rate_limited=true`、`rate_limit_key_source`、`provider_mode` を見る。
5. retry event は `retry_reason`、`retry_succeeded`、missing field count などの sanitized metadata だけを見る。

event log に出ない / 出してはいけないもの:

- raw prompt。
- raw provider response。
- raw Codex output。
- endpoint / model 実値。
- secret / token。
- local path。
- stack trace。
- user_hint 全文。
- candidate 自由文本文。
- actual IP / forwarded header value / internal rate-limit key。

event log は provider 運用観測用であり、投資判断、candidate ranking、Strategy / StrategyVersion 保存、Pine generation 自動起動には使わない。
