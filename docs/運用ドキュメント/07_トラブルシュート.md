# 北極星 トラブルシュート

更新日: 2026-05-13
分類: 運用ドキュメント

## 1. 目的

本資料は、ローカル開発・確認時によくある失敗の切り分け入口である。機能別の詳細 runbook がある場合は、該当 docs を優先する。

## 2. 起動できない

確認順:

1. PostgreSQL / Redis が起動しているか確認する。
2. `.env` が `.env.example` から作られているか確認する。
3. `pnpm install` が完了しているか確認する。
4. backend / frontend の dev process が既に起動して port を占有していないか確認する。
5. DB migration と Prisma generate が実行済みか確認する。

必要なら次をやり直す。

```bash
pnpm run down
pnpm run up
cd backend
pnpm exec prisma validate
pnpm exec prisma migrate deploy
pnpm exec prisma generate
pnpm exec prisma db seed
```

## 3. Prisma / DB で詰まる

- drift が出た場合は、repo の Prisma schema と migrations を正本として扱う。
- Windows で `query_engine-windows.dll.node` の file lock が出る場合は、backend / frontend dev process を止めてから再実行する。
- seed が失敗する場合は、migration 適用状況と DB 接続先を確認する。
- 本番相当データを local troubleshooting に使わない。

## 4. 日本語 JSON が文字化けする

PowerShell から日本語 JSON を送る場合は、文字列 body ではなく UTF-8 byte array に変換して送る。詳細は `docs/walkthrough.md` の PowerShell 手順を参照する。

確認観点:

- `Content-Type` に `charset=utf-8` を付ける。
- `-Body` には UTF-8 byte array を渡す。
- UI 経由の日本語入力と shell 経由の encoding 問題を分けて扱う。

## 5. TradingView webhook が届かない / 落ちる

確認順:

1. webhook URL が外部から到達可能か確認する。
2. token の渡し方を確認する。
3. `trigger_price` が number になっているか確認する。
4. `webhook_receipts` の auth / parse / symbol resolution / dedupe を確認する。
5. `alert_events` と `ai_jobs` を確認する。

詳細は `docs/運用ドキュメント/03_TradingView_webhook運用.md` と `docs/32.北極星 TradingView実送信 webhook運用手順（MVP）.md` を参照する。

## 6. CSV import が failed になる

確認順:

1. 対応形式の CSV か確認する。
2. header と data row があるか確認する。
3. required columns が揃っているか確認する。
4. `parse_error` と補助説明文を読む。
5. 最新 import が failed でも、過去 parsed import が残っていないか確認する。

詳細は `docs/運用ドキュメント/04_CSV取込運用.md` を参照する。

## 7. AI summary が unavailable / failed になる

確認順:

1. provider 設定を確認する。
2. local LLM を使う場合は provider process が起動しているか確認する。
3. `AI_ENABLE_STUB_FALLBACK` の設定を確認する。
4. `ai_jobs` の status と sanitized error を確認する。
5. 同一 input の failed job がある場合、自動 retry ではなく手動生成または別タスクで判断する。

詳細は `docs/運用ドキュメント/05_AI_provider運用.md` を参照する。

## 8. Browser smoke が落ちる

確認順:

1. backend / frontend が起動しているか確認する。
2. seed 済みか確認する。
3. `POLARIS_E2E_BASE_URL` を指定している場合は接続先を確認する。
4. locator が文言変更に依存していないか確認する。
5. 実行系操作や外部 API に依存する scenario を smoke に入れていないか確認する。

## 9. Strategy proposal local_llm が schema_invalid になる

StrategyLab で `STRATEGY_PROPOSAL_PROVIDER=local_llm` を使い、UI に `provider status: invalid_response / reason: schema_invalid / latency: slow` が出る場合は、local_llm から応答は返っているが `strategy_proposal_candidates` schema に通っていない状態として扱う。

確認順:

1. `STRATEGY_PROPOSAL_PROVIDER=local_llm` が明示設定されているか確認する。未指定 default は `stub`。
2. local LLM process が起動しているか確認する。
3. proposal 専用の endpoint / model / timeout / max output 設定が local env にあるか確認する。ただし実値を docs、PR、screenshot、log へ残さない。
4. backend / frontend dev process を再起動し、StrategyLab で `ストラテジーを提案` を再実行する。
5. UI / API response では sanitized `provider_observation.status`、`invalid_reason`、`latency_bucket` だけを見る。
6. raw prompt、raw provider response、endpoint、model 実値、secret、local path、stack trace を保存・貼付しない。
7. required test では real local_llm を叩かず、mock / fake response で code fence、前後説明文、root metadata 欠落、string array、enum 表記揺れ、重要 field 欠落を確認する。

現行実装の境界:

- local_llm prompt は JSON object のみ、英語 key 固定、array field は配列、`source_type=web` 不使用を要求する。
- code fence や前後説明文が混ざる軽微な response は、最初の JSON object / array を抽出して parse する。
- root metadata 補完、string array の配列化、enum 表記揺れの snake_case 化など安全な normalization は行う。
- title / summary / entry / exit / risk / suggested natural language spec などの重要本文が欠けている場合は、provider が usable candidate を返せていないものとして invalid response のまま扱う。
- retry、自動 fallback、openai_api、Web search、deep research、auto Pine generation はこの切り分け対象外。

詳細は `docs/運用ドキュメント/11_Strategy_proposal品質評価運用.md` を参照する。

## 10. docs-only PR の注意

- 実装変更を混ぜない。
- local path、secret、token、shared_secret、API key の実値を残さない。
- `git diff --check` を必ず実行する。
- 旧 docs の削除や大量 rename は別 PR で判断する。
