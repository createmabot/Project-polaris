# 北極星 Tailwind化・UI構造安定化方針（P3）

更新日: 2026-05-09

## 1. 目的

- Tailwind 化の目的を整理する
- UI 構造安定化の目的を整理する
- 見た目の全面刷新ではなく、段階的な保守性向上を優先する
- Tailwind 導入 PR と画面移行 PR を分け、差分と回帰確認を小さく保つ

## 2. 現行UI課題

### 2.1 inline style の偏在

2026-05-09 時点の `frontend/src/pages/*.tsx` では、主要画面に inline style が多い。

- `StrategyVersionDetail.tsx`: 246
- `BacktestDetail.tsx`: 90
- `ComparisonDetail.tsx`: 90
- `SymbolDetail.tsx`: 60
- `Home.tsx`: 56
- `StrategyVersionList.tsx`: 51
- `StrategyLab.tsx`: 49
- `WatchlistManage.tsx` / `PositionsManage.tsx`: 各 19

補足:

- まず `StrategyVersionDetail` から手を付けると差分が大きくなりやすい
- 初回移行は、影響が大きく外部依存が少ない `Home` を優先する方が安全

### 2.2 画面ごとの重複

複数画面で次の表現が重複している。

- 画面外枠: `padding`, `maxWidth`, `margin`, `fontFamily`
- 戻るリンク行
- section 見出し + border / background 付き card
- primary / secondary button
- 空状態 / loading 状態 / error 状態
- link 色と装飾
- key-value 表示

### 2.3 構造の揺れ

特に次の揺れがある。

- card の背景色 / border / radius が画面ごとに異なる
- section ごとの余白と見出し階層が揺れる
- 状態表示のパターンが画面ごとに独自実装になっている
- `Home` / `SymbolDetail` / `ComparisonDetail` / `BacktestDetail` で「情報 card の並び」は似ているが共通 component になっていない

## 3. Tailwind 化の目的

- UI の一貫性向上
- inline style 削減
- 画面改修速度向上
- card / button / section / layout の統一
- 将来の responsive 対応や見た目改善の土台作り

補足:

- 今回の主目的は見た目の全面刷新ではない
- まずは UI 構造を安定させ、component と test の変更コストを下げる

## 4. 対象画面比較

| 候補 | 価値 | コスト | リスク | 初回向きか |
|---|---:|---:|---:|---|
| Home | 高 | 低〜中 | 低 | はい |
| SymbolDetail | 高 | 中 | 中 | 次点 |
| Watchlist / Positions | 中 | 低〜中 | 低 | Home 後に向く |
| StrategyLab | 高 | 中〜高 | 中〜高 | まだ早い |
| BacktestDetail | 高 | 高 | 高 | 初回は重い |
| ComparisonDetail | 中〜高 | 高 | 高 | 初回は重い |
| StrategyVersionDetail | 高 | 非常に高い | 非常に高い | 後回し |

比較のポイント:

- `Home` は MVP 入口であり、Playwright PoC でも既に触っている
- `SymbolDetail` は `Home` の次の主要導線で、同系統の card / section が多い
- `Watchlist / Positions` は CRUD 管理画面として form / list の共通化に向く
- `StrategyLab` / `BacktestDetail` / `ComparisonDetail` / `StrategyVersionDetail` は状態分岐が多く、初回で触るには重い

## 5. 初回移行対象

推奨:

- `Home`

理由:

- MVP 入口である
- Playwright local-only PoC の対象である
- card / section / link / button の共通化効果が高い
- 外部依存が少ない
- 既存 Vitest / Playwright / backend E2E と組み合わせて回帰確認しやすい

## 6. 共通UI component 方針

### 6.1 最初に作る候補

- `AppLayout`
- `PageHeader`
- `SectionCard`
- `TextLink`
- `EmptyState`
- `LoadingState`
- `ErrorState`
- `PrimaryButton`
- `SecondaryButton`

### 6.2 後回しにする候補

- `PillToggle / SegmentedControl`
- `DataList / KeyValueRow`
- `StatusBadge`
- `AlertSummaryCard`
- `AiSummaryCard`
- `ReferenceBreakdown`

### 6.3 方針

- 最初から全部共通化しない
- `Home` で複数回出る構造だけを最小抽出する
- 過剰抽象化を避け、2 画面目で再利用が見えた component だけを残す

## 7. Tailwind 導入方式

### 案1: Tailwind を導入し、既存画面を段階的に `className` 化

- 推奨
- 差分を小さくしやすい
- 既存画面を 1 画面ずつ確認できる

### 案2: 共通 component を先に作り、その中だけ Tailwind を使う

- UI 構造安定には有効
- ただし初回設計が重くなりやすい

### 案3: 画面全面を一気に Tailwind 化

- 非推奨
- 差分が大きすぎて回帰確認が難しい

## 8. Tailwind 導入順序

推奨順:

1. Tailwind 設定追加のみ
2. 共通 UI component の最小土台
3. `Home` の最小置換
4. Playwright / Vitest / build 確認
5. `SymbolDetail` へ拡張
6. `Watchlist / Positions` へ拡張
7. `StrategyLab` / `BacktestDetail` / `ComparisonDetail` / `StrategyVersionDetail` は後続判断

補足:

- Tailwind 導入 PR と `Home` 移行 PR は分ける
- `Home` 全面移行を Tailwind 設定追加 PR と同時にやらない

## 9. テスト方針

PR #210 の方針を引き継ぐ。

- Tailwind class を test しない
- layout / color / spacing を test しない
- `role` / heading / link / button / URL / stable text を確認する
- 既存の `Home → SymbolDetail` Playwright PoC は維持する
- `Home` 移行時は `frontend/src/pages/Home.test.tsx` と `frontend/e2e/home-symbol-detail.spec.ts` を維持し、必要最小限でだけ調整する
- Visual regression / screenshot comparison はまだ導入しない

責務分担:

- Playwright:
  - 実ブラウザでの主要導線
  - 画面遷移
  - ボタン / link 操作
  - API 反映後の表示
- frontend Vitest:
  - 文言
  - 空状態
  - helper
  - URL 生成
  - props 分岐
- backend E2E:
  - API 契約
  - 保存導線
  - queue / summary / collector flow

## 10. 今回やらないこと

- Tailwind 導入
- CSS 改修
- UI 実装
- component 作成
- Playwright spec 追加
- Visual regression
- CI 導入

## 11. 次PR候補

候補1:

- Tailwind 設定追加 + 最小 global CSS 整理

候補2:

- 共通 UI component 最小土台追加

候補3:

- `Home` の `SectionCard` / `PageHeader` 化

推奨:

- 次 PR は Tailwind 設定追加だけ、または Tailwind 設定 + 最小 `AppLayout` / `SectionCard` まで
- `Home` 全面移行はその次

## 12. 判断

- Tailwind 化の初回対象は `Home`
- `SymbolDetail` はその次
- `Watchlist / Positions` は共通 form / list パターン整理後に進める
- `StrategyLab` / `BacktestDetail` / `ComparisonDetail` / `StrategyVersionDetail` は、構造整理コストが高いため後回しにする
- browser-based E2E の対象拡張は、この UI / Tailwind 方針に沿って 2 本目以降を判断する
