# 北極星 Tailwind化・UI構造安定化方針（P3）

更新日: 2026-05-09

## 1. 目的

- Tailwind 化の目的を、見た目刷新ではなく保守性向上として整理する
- UI 構造安定化の対象範囲と順序を固定する
- 画面導線 / IA / ナビゲーション変更と Tailwind 化の前後関係を明確にする
- `docs/45` の browser-based E2E 方針と矛盾しない frontend test 方針を維持する
- `docs/47` の画面導線 / IA / ナビゲーション方針に沿って、触る画面から段階移行する

本ドキュメントで固定する前提は次のとおり。

- 全画面を先に Tailwind 化してから導線変更する方針は採らない
- Tailwind 設定追加は先に進めてよい
- Home / SymbolDetail など画面単位の Tailwind 移行は、画面導線 / IA / ナビゲーション方針整理の後に行う
- 共通 UI component は、導線変更で実際に触る画面から段階的に抽出する

## 2. 現行UI課題

現行 frontend では、特に次の課題がある。

- inline style が多い
- 画面ごとに余白、外枠、section card、button、戻るリンク行の実装が重複している
- `Home` `SymbolDetail` `ComparisonDetail` `StrategyLab` `BacktestDetail` `StrategyVersionDetail` で card / section / link / button 表現が揺れている
- empty / loading / error state の構造が画面ごとに揺れている
- 画面導線やナビゲーションの見直しが入る前に内部 component を抽出しすぎると、あとで分解し直す可能性が高い

補足:

- `Home` と `SymbolDetail` は MVP 入口導線として重要で、今後も画面構造の整理対象になりやすい
- `StrategyLab` `BacktestDetail` `ComparisonDetail` `StrategyVersionDetail` は状態分岐が多く、初回の UI 構造安定化対象としては重い

## 3. Tailwind 化の目的

Tailwind 化の目的は次のとおり。

- UI の一貫性向上
- inline style 削減
- 画面改修速度向上
- card / button / section / layout の統一
- 将来の responsive 対応と見た目改善の土台作り

ただし、今回の基準点は「見た目の全面刷新」ではない。

優先するのは次の順序である。

1. Tailwind 設定を入れられる状態にする
2. 画面導線 / IA / ナビゲーション方針を整理する
3. その導線変更で触る画面から Tailwind 化と共通 component 化を進める

|---|---:|---:|---:|---|

| SymbolDetail | 高 | 中 | 中 | Home の次に向く |
| Watchlist / Positions | 中 | 低〜中 | 低 | Home / SymbolDetail 後に向く |
| StrategyLab | 高 | 中〜高 | 高 | まだ早い |
| BacktestDetail | 高 | 高 | 高 | まだ早い |
| ComparisonDetail | 中〜高 | 中〜高 | 中〜高 | まだ早い |
| StrategyVersionDetail | 中 | 高 | 高 | まだ早い |

判断:

- 初回の画面単位移行候補は引き続き `Home` だが、着手前に導線方針整理を挟む
- `SymbolDetail` は `Home` に続く導線先として次点
- `Watchlist / Positions` は CRUD 管理画面として共通 form/list の効果が高いが、入口導線よりは後
- 状態分岐の多い画面は、共通土台ができるまで後回しにする

## 5. 初回移行対象

推奨:

- 画面単位の初回移行対象は `Home`

ただし、前提条件を置く。

- `Home` の Tailwind 移行は、画面導線 / IA / ナビゲーション方針整理の後に行う
- `Home` 全面移行を Tailwind 設定追加 PR と同時には行わない

理由:

- `Home` は MVP 入口であり、既存 Playwright PoC の対象でもある
- card / section / link / button の共通化効果が高い
- 一方で、導線変更予定がある状態で先に `SectionCard` などを大量抽出すると、後から再設計コストが出る

## 6. 共通UI component方針

### 6.1 先に検討する候補

導線方針整理の前後をまたいでも比較的壊れにくい最小候補は次のとおり。

- `AppLayout`
- `PageHeader`
- `Navigation`
- `TextLink`

これらは、ページ外枠、戻る導線、上位ナビゲーション、共通リンク表現として先に整理しやすい。

### 6.2 導線変更後に抽出する候補

画面内部構造に強く依存するものは、導線変更後の構造が固まってから抽出する。

- `SectionCard`
- `EmptyState`
- `LoadingState`
- `ErrorState`
- `PrimaryButton`
- `SecondaryButton`
- `DataList / KeyValueRow`
- `StatusBadge`
- `AlertSummaryCard`
- `AiSummaryCard`
- `ReferenceBreakdown`

理由:

- これらは画面内部の section 境界や情報階層の影響を受けやすい
- 導線変更前に抽出すると、抽象化が早すぎて逆に手戻りが増える

### 6.3 方針

- 最初から全部共通化しない
- 導線変更で触る画面から、実際に重複が確認できたものだけを抽出する
- 過剰抽象化を避ける

## 7. Tailwind導入方式

### 案1: Tailwind を導入し、既存画面を段階的に `className` 化

- 推奨
- 差分を小さく分けやすい
- Tailwind 設定追加 PR と画面移行 PR を分離しやすい

### 案2: 共通 component を先に作り、その中だけ Tailwind を使う

- 一部有効
- ただし、導線変更前にやりすぎると抽象化コストが高い

### 案3: 画面全面を一気に Tailwind 化

- 非推奨
- 差分が大きく、導線変更予定と競合しやすい
- test 調整と画面確認のコストが上がる

推奨方針:

- Tailwind 導入 PR と Home 移行 PR は分ける
- まず Tailwind 設定追加のみ、または Tailwind 設定追加 + 最小 global CSS 整理まで
- その後に画面導線 / IA / ナビゲーション方針整理を行う
- 画面単位移行は、その導線変更で触る画面から始める

## 8. Tailwind導入順序

推奨順:

1. Tailwind 設定追加のみ、または Tailwind 設定追加 + 最小 global CSS 整理
2. 画面導線 / IA / ナビゲーション方針整理
3. 導線変更で最初に触る画面向けに `AppLayout` / `PageHeader` / `Navigation` / `TextLink` を最小導入
4. `Home` の最小置換
5. `Playwright` / `Vitest` / build 確認
6. `SymbolDetail` へ拡張
7. `Watchlist / Positions` へ拡張
8. `StrategyLab` / `BacktestDetail` / `ComparisonDetail` / `StrategyVersionDetail` は後続判断

明示する判断:

- 「全画面 Tailwind 化してから導線変更」は採らない
- `Home` 全面移行はまだ行わない
- 2026-05-09 時点で Tailwind 設定追加と最小 global CSS 整理までは完了
- 2026-05-09 時点で `AppLayout` / `PageHeader` / `Navigation` / `TextLink` の最小土台も `Home` / `SymbolDetail` にだけ適用済み
- `SideRail` の最小 PoC は `Home` / `SymbolDetail` にだけ追加済み
- `SideRail` は `/api/home` 再利用、タブ切り替え、折りたたみ、`/watchlist` `/positions` への補助導線までに留める
- `SideRail` の CRUD モーダル、`SectionCard` など画面内部 component は未実装
- 2026-05-09 時点で `Home` の最小移行は完了し、`Home` 本体から watchlist / positions の詳細一覧を外して `SideRail` に役割を寄せた
- 2026-05-09 時点で `SideRail` の CRUD モーダル化も最小実装まで完了した
- 次 PR は `SymbolDetail` の最小移行、または `SideRail` 再取得最適化判断へ進む

## 9. テスト方針

`docs/45` の方針を引き継ぐ。

- Tailwind class を test しない
- layout / color / spacing を test しない
- role / heading / link / button / URL / stable text を見る
- Playwright は実ブラウザ導線確認に限定する
- Vitest は文言、空状態、helper、URL 生成、props 分岐を維持する
- backend E2E は API 契約と保存導線を担保する

追加判断:

- 既存 `Home → SymbolDetail` Playwright PoC は維持する
- 導線変更が入った場合は、PoC spec を必要最小限で調整する
- 2 本目以降の Playwright spec 追加は、Tailwind 化または UI 構造安定化後に判断する
- Visual regression / screenshot comparison はまだ入れない

## 10. 今回やらないこと

- Tailwind 導入
- CSS 改修
- UI 実装
- component 作成
- Playwright spec 追加
- Visual regression
- CI 導入
- API 改修
- DB 改修
- backend 改修
