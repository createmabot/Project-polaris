# 北極星 ホームデータ供給仕様（MVP）

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVPにおいて、**ホーム画面を成立させるためのデータ供給仕様** を定義するものである。

企画資料や画面仕様では、ホームは以下の価値を担う。

- 毎日5分で状況把握する
    
- マーケット全体の変化を俯瞰する
    
- 発火アラートとその意味づけを短時間で確認する
    
- 気になる銘柄を銘柄詳細へつなぐ起点になる
    

一方で、ホームで扱う「マーケット概況」「注目イベント」「朝サマリ / 夜サマリ」は、画面だけ定義しても成立しない。  
このため本書では、**どのデータを、どこから、どの頻度で、どの責務で供給するか** を整理する。

---

## 2. 対象範囲

MVPホームで対象にするのは以下。

1. market_overview
    
2. watchlist_symbols
    
3. positions
    
4. recent_alerts
    
5. daily_summary
    
6. key_events
    

対象外とするものは以下。

- ポートフォリオ寄与度分析
    
- 高度スクリーニング連携
    
- 過去相場との類似局面提示
    
- マルチマーケット最適化
    
- リアルタイム秒単位更新
    

---

## 3. ホーム画面の責務

ホームは、詳細分析の場ではなく、**状況把握の入口** として設計する。  
そのため、ホームの責務は以下に限定する。

### 3.1 表示責務

- 主要指数・為替・セクター・監視対象の現況を一覧で見せる
    
- 直近発火したアラートを時系列で見せる
    
- 当日の意味づけをAI要約で見せる
    
- 注目イベントを短く整理して見せる
    
- 次の行動先として銘柄詳細へ誘導する
    

### 3.2 非責務

- 詳細チャート分析
    
- 財務諸表の深掘り
    
- 比較分析の完結
    
- ノート編集の完結
    
- ルール検証の実行
    

---

## 4. ホームレスポンスの基本形

ホームは原則として `GET /api/home` で取得する。

### 4.1 返却項目

- `market_overview`
    
- `watchlist_symbols`
    
- `positions`
    
- `recent_alerts`
    
- `daily_summary`
    
- `key_events`
    

### 4.2 Query

- `date`: 任意
    
- `summary_type`: `morning | evening | latest`
    

### 4.3 設計方針

- 画面初期表示で必要なものは BFF 的にまとめて返す
    
- ホーム専用 view model を組み立てて返す
    
- 個別の元データテーブルをそのまま返さない
    
- 一部データが欠けても、ホーム全体は表示可能にする
    

---

## 5. データ供給全体像

ホームデータは、単一ソースから直接返すのではなく、以下の供給系を組み合わせて構成する。

### 5.1 供給系の区分

1. 静的・準静的データ
    
    - users
        
    - symbols
        
    - watchlists
        
    - watchlist_items
        
    - positions
        
2. 定期取得データ
    
    - market_snapshots
        
3. 外部イベント起点データ
    
    - alert_events
        
    - external_references
        
4. AI生成データ
    
    - ai_jobs
        
    - ai_summaries
        
5. ホーム組立ロジック
    
    - market_overview 生成
        
    - key_events 生成
        
    - daily_summary 選択
        

### 5.2 依存関係の考え方

- `market_overview` は主に `market_snapshots` から組み立てる
    
- `watchlist_symbols` は `watchlists / watchlist_items / symbols / market_snapshots` から組み立てる
    
- `positions` は `positions / symbols / market_snapshots` から組み立てる
    
- `recent_alerts` は `alert_events` を主とする
    
- `daily_summary` は `ai_summaries` を主とし、生成素材に `market_snapshots / alert_events / external_references` を使う
    
- `key_events` は `alert_events / external_references` を主に組み立てる

## 6. データ項目ごとの供給仕様

## 6.1 market_overview

### 目的

ホーム上部で、相場全体の温度感を短時間で把握するための概況データを返す。

### 構成要素

- indices
    
- fx
    
- sectors
    

### 主な供給元

- `market_snapshots`
    

### 補助情報

- `symbols` または固定コードマスタ
    

### 生成方針

- snapshot_type ごとに対象を分類する
    
- 最新の有効 snapshot を採用する
    
- `indices / fx / sectors` の3ブロックに分けて返す
    
- セクターは MVP では主要セクターのみでよい
    

### 最小返却項目

各要素は以下を基本とする。

- `code`
    
- `display_name`
    
- `price`
    
- `change_value`
    
- `change_rate`
    
- `as_of`
    

### 更新頻度

- 朝サマリ用: 当日朝の基準時点で1回以上
    
- 日中表示用: 必要に応じて数回更新してもよい
    
- MVPでは厳密リアルタイムは要求しない
    

### フォールバック

- 一部カテゴリが欠けても他カテゴリは返す
    
- `as_of` を必ず返し、鮮度低下を UI で表現可能にする
    
- MVP 現行契約では `stale` フラグは追加せず、全欠損時は `sectors: []` を返す（部分成立優先）

### 6.1.1 `market_overview.sectors` 正本化設計（2026-04 固定）

`sectors` は現在 placeholder（`[]`）を許容している。理由は、現行 Home 実装では
セクター値の正本となる snapshot 供給が未整備で、推測分類を避けているため。

#### 正本候補比較

- A: `market_snapshots(snapshot_type=sector)` を正本にする
  - 長所: 価格・騰落を snapshot で一貫管理できる
  - 短所: 表示名・表示順を別で補う必要がある
- B: `symbols` に `sector/industry` 属性を持たせ再構成する
  - 長所: 銘柄起点集計には有利
  - 短所: 銘柄マスタ拡張と分類運用が必要で MVP には重い
- C: 固定コードマスタ + `market_snapshots(snapshot_type=sector)` の組み合わせ
  - 長所: 最小で安定した表示名・表示順を確保できる
  - 短所: コードマスタ維持が必要（ただし件数は少数）

#### MVP 採用案（固定）

MVP では **C** を採用する。

1. セクター値（`price/change_value/change_rate/as_of`）は `market_snapshots(snapshot_type=sector)` を正本にする
2. 主要セクターの表示名・表示順は固定コードマスタで補完する
3. Home は完成済み view model を返すのみ（セクター詳細や銘柄内訳は非責務）

#### 最小返却項目（`market_overview.sectors[]`）

- `code`
- `display_name`
- `price`
- `change_value`
- `change_rate`
- `as_of`

#### 欠損 / stale 方針（MVP）

- 一部セクター欠損: 部分成立で返す（欠損コードは省略可）
- 全件欠損: `sectors: []`（Home 全体は成立）
- stale 判定は `as_of` ベースで UI 表示可能にし、failure にはしない

#### seed / snapshot 供給の最小基準（次回実装用）

- 主要セクター 3〜5 件で Home 目視確認を成立させる
- 例:
  - `snapshot_type=sector, target_code=TOPIX_TRANSPORT`
  - `snapshot_type=sector, target_code=TOPIX_ELECTRIC`
  - `snapshot_type=sector, target_code=TOPIX_BANKS`
  - `snapshot_type=sector, target_code=TOPIX_RETAIL`
  - `snapshot_type=sector, target_code=TOPIX_INFO_TECH`

## 6.2 watchlist_symbols

### 目的

監視対象の中で、今日まず見るべき銘柄一覧を返す。

### 主な供給元

- `watchlists`
    
- `watchlist_items`
    
- `symbols`
    
- `market_snapshots`
    
- 必要に応じて `alert_events`
    

### 生成方針

- user の watchlist を取得する
    
- symbol 基本情報を付与する
    
- 最新 snapshot から価格情報を補う
    
- 直近 alert があれば最新 alert 状態を付ける
    
- 将来的には priority や通知重要度で並び替える
    

### 最小返却項目

- `symbol_id`
    
- `display_name`
    
- `tradingview_symbol`
    
- `latest_price`
    
- `change_rate`
    
- `latest_alert_status`
    
- `user_priority`
    

### 並び順

MVPでは以下の優先順を推奨する。

1. user_priority 高
    
2. 直近 alert 発火あり
    
3. change_rate 絶対値大
    
4. display_name

## 6.3 positions

### 目的

保有銘柄の現況をホーム上で簡易確認できるようにする。

### 主な供給元

- `positions`
    
- `symbols`
    
- `market_snapshots`
    

### 生成方針

- `positions.status = open` を主対象とする
    
- symbol 基本情報を付与する
    
- 最新 snapshot から現在値と騰落率を補う
    

### 最小返却項目

- `position_id`
    
- `symbol_id`
    
- `display_name`
    
- `quantity`
    
- `average_cost`
    
- `latest_price`
    
- `unrealized_pnl` または `unrealized_pnl_rate`
    
- `status`
    

### 補足

MVPでは厳密なポートフォリオ分析は行わず、簡易な保有状況確認に留める。

## 6.4 recent_alerts

### 目的

直近の発火アラートを時系列で確認できるようにする。

### 主な供給元

- `alert_events`
    
- `symbols`
    
- 必要に応じて `ai_summaries`
    

### 生成方針

- user に紐づく直近 alert を取得する
    
- `triggered_at` 降順で並べる
    
- unresolved でも表示候補に残せるようにする
    
- `related_summary_id` があれば紐付ける
    

### 最小返却項目

- `id`
    
- `symbol_id`
    
- `symbol`
    
- `alert_type`
    
- `alert_name`
    
- `timeframe`
    
- `trigger_price`
    
- `triggered_at`
    
- `processing_status`
    
- `related_summary_id`
    

### 件数上限

- MVPでは 10〜30 件程度を推奨
    
- 詳細一覧は別 API でページング取得する
    

---

## 6.5 daily_summary

### 目的

当日の全体像を、短時間で読める自然言語要約として返す。

### 主な供給元

- `ai_summaries`
    

### 生成素材

- `market_snapshots`
    
- `alert_events`
    
- `external_references`
    

### 対象 summary_scope

MVPでは以下を想定する。

- `home_daily_morning`
    
- `home_daily_evening`
    
- `home_daily_latest`
    

### Query との対応

- `summary_type = morning` → `home_daily_morning`
    
- `summary_type = evening` → `home_daily_evening`
    
- `summary_type = latest` → 当日利用可能な最新 summary を返す
    

### 最小返却項目

- `id`
    
- `title`
    
- `body_markdown`
    
- `generated_at`
    
- `status`
    
- `insufficient_context`
    

### 生成責務

daily_summary は画面読み出し時に毎回生成するのではなく、**事前生成を原則** とする。

### 事前生成タイミング

- 朝サマリ: 当日朝の基準時刻以後
    
- 夜サマリ: 当日引け後の基準時刻以後
    
- latest: 既存 summary の再利用を優先し、必要時のみ再生成
    

### フォールバック

- 生成済みが無ければ、直近の `latest` を返してもよい
    
- それも無ければ `status = unavailable` で空表示にせず理由を返す
    
- 材料不足時は `insufficient_context = true` を返し、本文にも不足を明示する
    

---

## 6.6 key_events

### 目的

今日の注目ポイントを、ニュース一覧ではなく短いイベント列として返す。

### 主な供給元

- `alert_events`
    
- `external_references`
    
- 必要に応じて将来の決算イベントソース
    

### MVPでの生成方針

MVPでは専用テーブルを持たず、ホーム組立時に **軽量派生データ** として作る。

### イベント候補

- 重要 alert 発火
    
- 決算発表
    
- 適時開示
    
- 大きな価格変動の背景ニュース
    
- 翌営業日注目の予定イベント
    

### 最小返却項目

- `event_type`
    
- `symbol_id` または `target_code`
    
- `title`
    
- `summary_text`
    
- `event_time`
    
- `importance`
    
- `source_refs`
    

### 生成優先順位

MVPでは以下を推奨する。

1. 当日発火した高重要度 alert
    
2. 当日 published_at の external_reference
    
3. 翌営業日に影響しうる予定イベント
    

### 重複排除ルール

- 同一 `symbol_id + source_url` は原則1件に寄せる
    
- 同一テーマの類似ニュースは1件に集約して summary_text へ要約する
    
- 同じ alert を繰り返し見せないよう `dedupe_key` を利用して抑制できる
    

### 件数上限

- MVPでは 5〜10 件程度
    

### フォールバック

- 十分な候補が無ければ空配列を返してよい
    
- ただし daily_summary 側で「本日は注目イベントが少ない」ことを補足できるのが望ましい

## 7. market_snapshots の供給仕様

## 7.1 位置づけ

`market_snapshots` はホームの概況表示を成立させるための基礎テーブルである。  
MVPではホーム向けに、指数・為替・セクター・主要銘柄のスナップショットを保持する。

## 7.2 保存対象

MVPでは以下を対象とする。

- 主要指数
    
- 主要為替
    
- 主要セクター
    
- watchlist / positions に含まれる主要銘柄
    

## 7.3 主なカラム利用

- `snapshot_type`
    
- `target_code`
    
- `snapshot_date`
    
- `snapshot_timeframe`
    
- `price`
    
- `change_value`
    
- `change_rate`
    
- `volume`
    
- `extra_json`
    
- `created_at`
    

## 7.4 取得頻度の考え方

MVPでは以下のような粗い更新でもよい。

- 朝: 寄り前または寄り後に1回
    
- 日中: 必要なら数回
    
- 夜: 引け後に1回
    

重要なのはリアルタイム性そのものではなく、**ホームで見せる概況として破綻しない鮮度** を維持することである。

## 7.5 symbol FK を必須にしない理由

指数や為替、セクターは symbols テーブルで厳密管理しない場合があるため、MVPでは柔軟性を優先し、`target_code` ベースでの保存を許容する。

## 8. daily_summary 生成仕様

## 8.1 生成単位

daily_summary は **営業日単位 + summary_type 単位** で生成する。

### キー例

- `market_day = 2026-03-15`
    
- `summary_type = morning`
    

## 8.2 入力素材の優先順位

1. `market_snapshots`
    
2. `alert_events`
    
3. `external_references`
    

### 理由

- market_overview が無いと相場全体の説明が弱くなる
    
- alert が無いと監視対象の文脈が弱くなる
    
- references が無いと「なぜ動いたか」の説明が弱くなる
    

## 8.3 不足時の扱い

素材不足は failure ではなく **部分成立** として扱う。

### 例

- snapshots はあるが references が無い
    
- alerts はあるが重要イベントが少ない
    
- 監視銘柄は静かで market_overview だけ返せる
    

この場合も summary を生成し、`insufficient_context = true` を付与する。

## 8.4 再生成方針

- 初回生成後、同一 `market_day + summary_type` は再利用を基本とする
    
- 追加材料が大きく増えたときだけ再生成候補にする
    
- 再生成時は上書きか版管理かを将来検討してよいが、MVPでは最新1件でもよい

## 9. 営業日・時刻ルール

## 9.1 基本原則

- 保存時刻は UTC を正本とする
    
- 表示時刻は Asia/Tokyo を標準とする
    
- `triggered_at` と `received_at` は分けて保持する
    

## 9.2 ホーム集約の単位

ホームの日次集約は **日本市場営業日** を基準にする。

### 含意

- 単なるカレンダー日付ではなく営業日概念を使う
    
- 土日祝や休場日は、直近営業日との関係を意識して扱う
    
- 翌営業日朝サマリは、前営業日引け後の重要イベントを再掲してよい
    

## 9.3 朝サマリの範囲

MVPでは以下を目安にする。

- 前営業日引け後〜当日朝の重要イベント
    
- 当日朝時点での market_overview
    
- 必要に応じて overnight の主要変化
    

## 9.4 夜サマリの範囲

MVPでは以下を目安にする。

- 当営業日の日中変化
    
- 当日発火した alert
    
- 当日公開された主要 reference
    
- 引け後速報で翌営業日に効く情報
    

## 9.5 latest の扱い

`latest` は営業日区分と独立した新しい summary_type ではなく、**その時点で最も表示価値の高い既存 summary を返す選択モード** とする。

## 10. ジョブ設計

ホーム成立のため、少なくとも以下のジョブを想定する。

## 10.1 market snapshot 取得ジョブ

### 役割

- 市場概況に必要な snapshot を収集して保存する
    

### 出力

- `market_snapshots`
    

## 10.2 alert 後続処理ジョブ

### 役割

- webhook受信後に alert を整形し、必要に応じて summary 生成候補を立てる
    

### 出力

- `alert_events`
    
- 必要なら `ai_jobs`
    

## 10.3 external reference 収集ジョブ

### 役割

- alert や対象銘柄に紐づく reference を収集する
    

### 出力

- `external_references`
    

## 10.4 daily summary 生成ジョブ

### 役割

- 朝 / 夜の summary を事前生成する
    

### 入力

- `market_snapshots`
    
- `alert_events`
    
- `external_references`
    

### 出力

- `ai_summaries`

## 11. API組立ルール

## 11.1 GET /api/home の内部組立順

1. user を確定する
    
2. watchlists / positions を取得する
    
3. market_snapshots から market_overview を組み立てる
    
4. watchlist_symbols / positions に価格情報を補う
    
5. recent_alerts を取得する
    
6. `summary_type` に応じて daily_summary を選ぶ
    
7. key_events を組み立てる
    
8. 最終レスポンスへ詰める
    

## 11.2 一部欠損時のレスポンス方針

- `market_overview` 欠損 → 空ではなく stale 情報や空カテゴリを返す
    
- `recent_alerts` 欠損 → 空配列を返す
    
- `daily_summary` 欠損 → unavailable 状態で理由を返す
    
- `key_events` 欠損 → 空配列を返す
    

ホーム全体を 500 にしやすい構造は避ける。

## 12. UIで期待する表示制御

本資料はバックエンド中心だが、供給仕様として最低限以下を前提にする。

### 12.1 鮮度表示

- `generated_at`
    
- `as_of`
    
- `stale`
    

のいずれかで、データ鮮度を UI 表示可能にする。

### 12.2 不足表示

- `insufficient_context = true` の場合は本文中でも不足を明示する
    
- データゼロを正常系として扱う箇所は空状態UIを用意する
    

### 12.3 遷移導線

- key_event から symbol 詳細へ遷移できる
    
- recent_alerts から alert 詳細または symbol 詳細へ遷移できる
    
- watchlist_symbols / positions から symbol 詳細へ遷移できる

## 13. MVPでの割り切り

MVPでは以下を割り切る。

- key_events は専用テーブル化しない
    
- market_overview は完全リアルタイムを求めない
    
- セクター粒度は最小限でよい
    
- latest summary は既存 summary の再利用中心でよい
    
- 材料不足時でも部分成立を優先する

## 16. 結論

北極星MVPのホームは、単なる画面ではなく、  
**market_snapshots・alert_events・external_references・ai_summaries を組み合わせて成立する状況把握ハブ** として設計する。

そのため、MVPでは以下を固定する。

- 概況は `market_snapshots` を正本にする
    
- 発火情報は `alert_events` を正本にする
    
- 背景材料は `external_references` を補助根拠にする
    
- 朝サマリ / 夜サマリは `ai_summaries` の事前生成を原則にする
    
- 日次集約は日本市場営業日基準で扱う
    
- 不足時は failure ではなく部分成立として返す
    

この方針により、ホームを「毎日5分で状況把握する」ための安定した起点画面として実装しやすくする。
