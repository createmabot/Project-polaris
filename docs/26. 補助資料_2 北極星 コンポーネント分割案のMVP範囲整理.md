# 北極星 コンポーネント分割案のMVP範囲整理

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVPにおいて、**コンポーネント分割案のうち、どこまでをMVP範囲として実装するか** を明確化するための補助資料である。
※「22.コンポーネント分割案」と「17.画面別入出力サンプル」のズレを埋める形で、**MVPで本当に作るページ・Section・部品** を固定しています。

既存の「コンポーネント分割案（MVP）」では、Page / Section / Domain Component / Primitive UI / ViewModel の分割方針が整理されている。  
一方で、画面別入出力サンプルやスプリント計画と突き合わせると、以下のような論点が残る。

- MVPの正式なページ数は何画面か
    
- ComparisonPage はMVPに含めるか
    
- BacktestComparisonPage を独立ページとして作るか
    
- Backtest比較は機能として必要だが、どのUI形態で提供するか
    
- 再利用部品として定義されているもののうち、MVP時点で本当に必要なものは何か
    

そのため本資料では、**ページ単位・セクション単位・ドメイン部品単位** でMVP範囲を固定し、実装順とPhase 2送りの対象を明確にする。

---

## 2. 基本方針

### 2.1 MVPでは「使える導線」を優先する

MVPでは、ページ数や部品数を増やすことよりも、以下の3導線を成立させることを優先する。

- webhook → 要約 → ホーム
    
- 銘柄詳細 → 論点 → ノート
    
- 自然言語 → Pine → CSV → 総評
    

### 2.2 独立ページは最小限にする

MVPでは、画面遷移や状態管理が複雑になりやすい独立ページを増やしすぎない。  
特に、**BacktestComparisonPage のように機能としては必要だが、独立ページにしなくても成立するもの** は、まず既存ページ内のブロックで提供する。

### 2.3 共通化よりも責務の明瞭さを優先する

MVPでは、部品を増やして抽象化しすぎるより、  
「どの画面の、どの役割の部品か」が明確であることを重視する。

### 2.4 画面一覧とコンポーネント分割案の整合を取る

MVPの正式なルーティング単位は、画面別入出力サンプルに合わせて固定する。  
コンポーネント分割案側がそれより広い場合は、Phase 2候補として扱う。

---

## 3. MVPの正式ページスコープ

MVPで正式に扱うページは以下の6つとする。

1. HomePage
    
2. SymbolDetailPage
    
3. NoteDetailPage
    
4. ComparisonPage
    
5. StrategyLabPage
    
6. BacktestDetailPage
    

### 3.1 含める理由

- **HomePage**: 毎日5分で状況把握する中心導線
    
- **SymbolDetailPage**: 銘柄研究の起点
    
- **NoteDetailPage**: 仮説の保存と更新に必須
    
- **ComparisonPage**: MVPで定義済みの2銘柄比較導線
    
- **StrategyLabPage**: 自然言語→Pine の起点
    
- **BacktestDetailPage**: CSV取込後の結果確認とAI総評に必須
    

### 3.2 MVPから外す独立ページ

以下はMVPでは**独立ページとしては作らない**。

- BacktestComparisonPage
    

### 3.3 判断理由

Backtest比較そのものはMVP価値に含まれるが、  
独立ページを増やすと以下の負担が増える。

- ルーティング追加
    
- comparison data 専用 state の拡張
    
- 画面遷移設計の追加
    
- 比較専用 header / empty / error / loading の増加
    
- 比較画面専用テストの増加
    

MVPではこれを避け、**BacktestDetailPage 内の comparison block** として最小構成で提供する。

## 4. MVPページごとの実装境界

## 4.1 HomePage

### MVPで含めるSection

- HomeHeaderSection
    
- MarketOverviewSection
    
- WatchlistSection
    
- PositionsSection
    
- RecentAlertsSection
    
- DailySummarySection
    
- KeyEventsSection
    

### MVPで含める理由

ホームはMVP価値の中核であり、既存のホームデータ供給仕様でも成立条件が整理されている。  
そのため、表示ブロックは基本的にフルで含める。

### MVPで削るもの

- 高度フィルタバー
    
- 多段ソート
    
- 監視リストの複雑な一括操作
    
- 高頻度リアルタイム更新
    

---

## 4.2 SymbolDetailPage

### MVPで含めるSection

- SymbolHeaderSection
    
- SymbolChartSection
    
- SymbolReferencesSection
    
- ThesisCardSection
    
- LatestNoteSection
    
- CompareActionSection
    

### MVPで含める理由

銘柄研究の価値は、  
「価格 / 参考情報 / AI論点 / ノート導線」が一画面でつながることにあるため、  
この単位は削りにくい。

### MVPで削るもの

- 高度なチャート描画カスタムUI
    
- 比較候補の大量管理
    
- 参考情報の複雑なクロスフィルタ
    

---

## 4.3 NoteDetailPage

### MVPで含めるSection

- NoteHeaderSection
    
- NoteEditorSection
    
- NoteReviewScheduleSection
    
- NoteRevisionSection
    

### MVPで含める理由

ノートは北極星の知見資産なので、単なるメモ入力ではなく revision と review schedule まで含めて成立させる。

### MVPで削るもの

- 高度なテンプレート挿入
    
- コメント / 共同編集
    
- リッチテキストの複雑装飾
    

---

## 4.4 ComparisonPage

### MVPで含めるSection

- ComparisonHeaderSection
    
- ComparisonMetricsSection
    
- ComparisonSummarySection
    

### MVPで含める理由

画面別入出力サンプルではMVP画面に含まれており、  
企画上も銘柄比較は中核価値に近い。したがって、**独立ページとして残す**。

### MVPでの割り切り

- 2銘柄比較に限定する
    
- 比較指標は主要項目に絞る
    
- AI比較サマリは1ブロックに留める
    
- 複数候補保存や履歴管理は後回しにする
    

---

## 4.5 StrategyLabPage

### MVPで含めるSection

- StrategyInputSection
    
- NormalizedRuleSection
    
- PineWarningsSection
    
- PinePreviewSection
    
- BacktestCreateSection
    

### MVPで含める理由

自然言語→Pine→CSV取込導線を成立させるうえで、この5ブロックは最小構成で必要である。

### MVPで削るもの

- 高度な入力テンプレートライブラリ
    
- 条件式GUIビルダー
    
- 複数バージョンの並列編集
    

---

## 4.6 BacktestDetailPage

### MVPで含めるSection

- BacktestHeaderSection
    
- BacktestMetricsSection
    
- TradeSummarySection
    
- BacktestReviewSection
    
- ImportFilesSection
    
- BacktestComparisonActionSection
    
- BacktestComparisonInlineSection
    

### 補足

既存の分割案では `BacktestComparisonActionSection` までが明示されていたが、  
MVPでは比較機能を実際に使える形にするため、**BacktestComparisonInlineSection** を追加定義する。

### BacktestComparisonInlineSection の役割

- 選択した比較対象 run との主要差分を表示する
    
- 最低限の比較メトリクスを見せる
    
- AI比較総評を1ブロックで見せる
    

### MVPで含める子部品

- MetricComparisonTable
    
- TradeoffFlagList
    
- SummaryPanel
    

### MVPで削るもの

- 独立した BacktestComparisonHeaderSection
    
- 比較専用ページ遷移前提のUI
    
- 複数runの高度比較切り替え
    
- 比較条件保存

## 11. ルーティング方針

MVPでは以下のルートのみを正式採用する。

- `/home`
    
- `/symbols/:symbolId`
    
- `/notes/:noteId`
    
- `/comparisons/:comparisonId`
    
- `/strategy-lab`
    
- `/backtests/:backtestId`
    

### MVPで採用しないルート

- `/backtest-comparisons/:comparisonId`
    

### 代替方針

Backtest比較は `/backtests/:backtestId` 内で compare target を選び、  
同一ページ内で inline 表示する。

## 12. テスト観点

## 12.1 MVPで最低限必要な画面テスト

- HomePage 初期表示
    
- SymbolDetailPage 主要ブロック表示
    
- Note 保存と再表示
    
- StrategyLabPage で normalized rule / Pine が見える
    
- BacktestDetailPage で metrics / imports / review が見える
    
- BacktestDetailPage で comparison inline が表示される
    
- ComparisonPage で2銘柄比較が見える
    

## 12.2 テストを簡略化できる点

BacktestComparisonPage を作らないことで、以下を削減できる。

- 専用ルーティングテスト
    
- 専用空状態テスト
    
- 専用エラー状態テスト
    
- comparison page 固有の遷移テスト

## 14. 結論

北極星MVPのコンポーネント分割では、  
**Page / Section / Domain Component / Primitive UI / ViewModel** の基本方針は維持しつつ、  
MVP範囲は以下のように整理する。

- 正式なMVPページは6画面とする
    
- ComparisonPage はMVPに含める
    
- BacktestComparisonPage はMVPでは独立ページにしない
    
- Backtest比較は BacktestDetailPage 内の inline block で最小成立させる
    
- 再利用部品は summary / alert / reference / backtest metrics を優先する
    

この整理により、画面一覧・コンポーネント分割案・スプリント計画の整合を保ちながら、  
MVP実装のスコープを過不足なく固定しやすくする。
