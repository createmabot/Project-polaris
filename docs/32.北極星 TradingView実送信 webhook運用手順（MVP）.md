# 北極星 TradingView実送信 webhook運用手順（MVP）
更新日: 2026-05-07

## 1. 目的

本ドキュメントは、TradingView から北極星へ webhook を実送信する際の運用手順を固定する runbook です。

役割分担は次のとおりです。
- TradingView: alert 条件の設定、一次監視、webhook 送信
- 北極星: webhook 受信、symbol 解決、references 収集、Alert summary 生成、履歴管理

注意:
- secret / token / shared_secret / API key の実値は docs / PR / ログに残さない
- 実送信の初回疎通確認は token のみを推奨する
- shared_secret は追加の整合確認が必要な場合だけ使う

## 2. endpoint と認証

### 2.1 endpoint
- 正式 endpoint: `POST /api/integrations/tradingview/webhook`
- 互換 endpoint: `POST /api/webhooks/tradingview/webhook`

### 2.2 token
- 次のどちらかで送信する
  - Query: `?token=<WEBHOOK_TOKEN>`
  - Header: `Authorization: Bearer <WEBHOOK_TOKEN>`
- 現行実装では token は必須

### 2.3 shared_secret
- payload の `shared_secret` は任意
- `shared_secret` を payload に入れた場合だけ、`webhook_tokens.sharedSecretHash` と照合する
- `shared_secret` が未指定の場合は `not_provided` 扱いで処理継続
- `shared_secret` が不一致の場合は `failed_shared_secret_mismatch` で拒否される

## 3. TradingView alert message JSON

### 3.1 初回疎通確認用テンプレート

初回の疎通確認は token のみを使い、まず webhook 受信から `alert_event` 作成まで通ることを確認する。

```json
{
  "alert_name": "WATCH_BTCUSDT_realtime_webhook_test",
  "alert_type": "technical",
  "tradingview_symbol": "{{exchange}}:{{ticker}}",
  "symbol": "{{ticker}}",
  "market_code": "CRYPTO",
  "timeframe": "{{interval}}",
  "triggered_at": "{{time}}",
  "trigger_price": {{close}},
  "condition_summary": "watch signal: realtime TradingView webhook smoke test. Evaluate whether this is a monitoring signal and whether available background references support it."
}
```

### 3.2 shared_secret を使う場合の例

```json
{
  "alert_name": "WATCH_BTCUSDT_realtime_webhook_test",
  "alert_type": "technical",
  "tradingview_symbol": "{{exchange}}:{{ticker}}",
  "symbol": "{{ticker}}",
  "market_code": "CRYPTO",

  "shared_secret": "<YOUR_SHARED_SECRET>"

```

注意:
- `<YOUR_SHARED_SECRET>` は実値を書かない
- 実送信前に token と shared_secret の両方が正しいことを確認する

## 4. TradingView placeholder の注意点

### 4.1 `triggered_at`

次の書き方で ISO datetime として正常展開されることを実送信で確認済み。

```json
"triggered_at": "{{time}}"
```

### 4.2 `trigger_price`

現行 schema では `trigger_price` は number 型を期待する。

OK:

```json
"trigger_price": {{close}}
```

NG:

```json
"trigger_price": "{{close}}"
```

理由:
- クォートすると TradingView 展開後に string になる
- Polaris webhook schema は `trigger_price` を number として検証する
- string で届くと payload validation で落ちる

失敗時の観測例:
- `authResult: success`
- `parseResult: success_json`
- `symbolResolutionResult: null`
- `dedupeResult: null`
- `errorReason: Missing required payload fields`
- `alertEvent` は作成されない

## 5. symbol 解決と dedupe

### 5.1 symbol 解決順
1. `tradingview_symbol`
2. `market_code + symbol`
3. `symbol`
4. `symbol_code`
5. `display_name`

### 5.2 dedupe 基準
- dedupe は `tradingview_symbol + alert_name + timeframe + triggered_at + trigger_price` を基準にする
- 重複時は `accepted=true` かつ `status=duplicate_ignored`

## 6. 実送信前チェック

### 6.1 Polaris 側
- backend / frontend が起動している
- TradingView から到達可能な URL がある
  - tunnel
  - reverse proxy
  - deploy 環境
- 対象 symbol が DB に登録されている
- `tradingview_symbol` が TradingView 側と一致している

### 6.2 TradingView 側
- webhook URL が正しい
- token が正しい
- `trigger_price` をクォートしていない
- 初回確認では shared_secret を省略するか、正しい値で設定する

## 7. 受信後の確認手順

### 7.1 webhook_receipts
確認する項目:
- `authResult`
- `parseResult`
- `symbolResolutionResult`
- `dedupeResult`
- `alertEventId`

### 7.2 alert_events
確認する項目:
- `alert_name`
- `symbol_id`
- `triggered_at`
- `received_at`
- `processing_status`

### 7.3 ai_jobs
確認する項目:
- `collect_references_for_alert`
  - `status`
  - `saved_count`
  - `skipped_count`
  - `source_breakdown`
  - `diagnostics`
- `generate_alert_summary`
  - `status`
  - `model_name`
  - `retry_count`
  - `error_message`

### 7.4 API
- `GET /api/alerts/:alertId`
- `GET /api/alerts/:alertId/summary`
- `GET /api/home`
- `GET /api/symbols/:symbolId`

### 7.5 UI
- Home の `recent_alerts`
- SymbolDetail の `recent_alerts`
- Alert summary 本文
- `insufficient_context`

## 8. 失敗時の切り分け

### 8.1 `failed_shared_secret_mismatch`
- payload に `shared_secret` を入れた場合だけ発生する
- secret の実値はログに出さず、設定ミスの有無だけ確認する

### 8.2 `Missing required payload fields`
- 必須項目不足
- 型不一致
- 代表例:
  - `trigger_price` を string で送っている

### 8.3 `symbol unresolved`
- `tradingview_symbol` と DB 登録値の不一致
- `market_code` / `symbol` の組み合わせ不一致

### 8.4 `duplicate_ignored`
- 同じ alert 条件を短時間に再送している

### 8.5 summary failed
- `GET /api/alerts/:alertId/summary` の `latest_job.error_message` を確認する
- provider の生エラー全文や secret は返さない

## 9.1 references あり銘柄での追加実送信メモ

2026-05-08 に `2148` で TradingView 実送信 1 件を確認した。

確認条件:
- symbol: `2148`
- alert_name: `WATCH_2148_references_live_check`
- triggered_at: `2026-05-08T06:18:00Z`
- trigger_price: `1334`

確認できたこと:
- `webhook_receipts`
  - `authResult: success`
  - `parseResult: success_json`
  - `symbolResolutionResult: success`
  - `dedupeResult: success_inserted`
- `alert_events`
  - `processing_status: completed`
- `ai_jobs`
  - `collect_references_for_alert: succeeded`
  - `generate_alert_summary: succeeded`
- Home / SymbolDetail の `recent_alerts` 反映

観測結果:
- `collect_references_for_alert` は `saved_count = 5`
- `source_breakdown = { news: 5 }`
- 今回の alert では `disclosure / earnings` direct type は保存されなかった
- diagnostics
  - `disclosure.reason: tdnet_rows_exist_but_no_symbol_match`
  - `earnings.reason: tdnet_rows_exist_but_no_symbol_match`
- Alert summary は生成成功し、`insufficient_context = false`
- summary context / 本文には references が反映されたが、type は `news` のみだった

運用上の見方:
- references あり銘柄でも、alert 発火日と collector lookback の範囲によっては `disclosure / earnings` direct type が入らないことがある
- 2026-05-08 の `2148` では、背景材料は Google News RSS 側の適時開示記事で補われた
- `disclosure / earnings` direct type まで alert 実送信で確認するには、開示日の近傍で再確認するか、lookback 方針の見直しを別タスクで検討する

## 10. secret を残さない運用ルール

- docs / PR / チケット / ログに token / shared_secret の実値を書かない
- payload サンプルは `<YOUR_SHARED_SECRET>` を使う
- query token を含む URL を貼らない
- DB 確認時に secret 列をそのまま表示しない
