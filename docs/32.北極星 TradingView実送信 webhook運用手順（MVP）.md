# 北極星 TradingView実送信 webhook運用手順（MVP）
更新日: 2026-05-01

## 1. 目的
本書は、TradingView から北極星への webhook 実送信を運用者が再現できるように、手順と確認観点を固定するための手順書です。  
機密情報（token / shared_secret / API key）は画面共有、ログ、docs、PR本文に残さない運用を前提とします。

## 2. 対象 endpoint と認証

### 2.1 endpoint
- 推奨: `POST /api/integrations/tradingview/webhook`
- 後方互換: `POST /api/webhooks/tradingview/webhook`

### 2.2 認証方式
- token は次のいずれかで送信可能
  - Query: `?token=<WEBHOOK_TOKEN>`
  - Header: `Authorization: Bearer <WEBHOOK_TOKEN>`
- `shared_secret` は payload 内の `shared_secret` で照合
- 実運用では token + shared_secret 併用を推奨

## 3. TradingView alert message JSON テンプレート
TradingView の Alert message 欄に貼る標準テンプレート例です。  
`<YOUR_SHARED_SECRET>` は実値を記載せず、実運用でのみ置換してください。

```json
{
  "event_id": "{{ticker}}-{{interval}}-{{time}}-{{close}}",
  "alert_name": "MA25 breakout",
  "alert_type": "technical",
  "tradingview_symbol": "{{exchange}}:{{ticker}}",
  "symbol": "{{ticker}}",
  "market_code": "JP_STOCK",
  "timeframe": "{{interval}}",
  "triggered_at": "{{time}}",
  "trigger_price": "{{close}}",
  "condition_summary": "close crossed above MA25",
  "shared_secret": "<YOUR_SHARED_SECRET>"
}
```

注意:
- TradingView placeholder の可用性は TradingView 側仕様に依存するため、実際に使える placeholder は要実機確認。
- `triggered_at` は webhook 実装で日時として解釈可能な値を送る。
- `alert_name` / `alert_type` / `timeframe` は省略しない。

## 4. symbol 解決と dedupe の前提

### 4.1 symbol 解決優先順
1. `tradingview_symbol` 完全一致
2. `market_code + symbol` 一致
3. `symbol` 一致
4. `symbol_code` 一致
5. `display_name` 一致

### 4.2 dedupe の基本
- dedupe は `tradingview_symbol + alert_name + timeframe + triggered_at + trigger_price` をベースに判定
- 重複時は `accepted=true` だが `status=duplicate_ignored`

## 5. ローカル疑似送信と実TradingView送信の違い

### 5.1 ローカル疑似送信
- `localhost` へ PowerShell / curl で直接 POST 可能
- アプリ側の parse / symbol resolution / queue 連携の一次確認に有効

### 5.2 実TradingView送信
- TradingView から到達可能な URL が必要
- ローカル端末で受ける場合は次のいずれかが必要
  - tunnel（例: ngrok / Cloudflare Tunnel）
  - reverse proxy
  - deploy 環境
- 外部公開 endpoint では HTTPS を推奨

## 6. 運用チェックリスト

### 6.1 alert 設定前
- [ ] 対象 symbol が北極星に登録済み
- [ ] `tradingview_symbol` が TradingView 設定値と一致
- [ ] webhook token が有効
- [ ] `shared_secret` が一致
- [ ] endpoint URL を `/api/integrations/tradingview/webhook` に設定

### 6.2 alert 送信直後（0-1分）
- [ ] webhook response が `accepted=true`
- [ ] `webhook_receipts` で auth/parse/symbol resolution が成功
- [ ] `alert_events` が作成されている
- [ ] Home の `recent_alerts` に反映される
- [ ] `ai_jobs` が作成される（collect references / generate summary）

### 6.3 数分後（1-5分）
- [ ] references 収集結果が確認できる
- [ ] alert summary が succeeded、または failed 理由が追跡可能
- [ ] SymbolDetail の recent alerts に反映される

## 7. 失敗時の切り分け手順

### 7.1 auth failure（401）
- token の送信有無（Query / Bearer）を確認
- token 有効状態を確認
- payload の `shared_secret` が一致しているか確認

### 7.2 parse failure（400）
- JSON 構文エラーを確認
- 必須フィールド欠落（`alert_name`, `alert_type`, `timeframe`, `triggered_at`）を確認
- Content-Type と body 形式の整合を確認

### 7.3 symbol unresolved
- `tradingview_symbol` と DB 登録値の差異を確認
- `market_code` / `symbol` の組み合わせが一致するか確認
- watchlist / positions 管理画面で symbol 登録内容を確認

### 7.4 duplicate ignored
- TradingView 側で同一条件の再送がないか確認
- dedupe 対象項目（symbol / alert_name / timeframe / triggered_at / trigger_price）を確認

### 7.5 summary failed
- `GET /api/alerts/:alertId/summary` の `latest_job.error_message` を確認
- local LLM provider 設定・接続可否を確認
- references 収集失敗の連鎖有無を確認

## 8. 確認用 API 手順（secret 非表示運用）

### 8.1 webhook response
- 成功例:
  - `accepted: true`
  - `status: received` または `duplicate_ignored`

### 8.2 alert 単位の確認
- `GET /api/alerts/:alertId`
  - `processing_status`
  - `symbol`
  - `triggered_at`

### 8.3 summary ジョブ確認
- `GET /api/alerts/:alertId/summary`
  - `latest_job.status`
  - `latest_job.error_message`
  - `summary.status`

### 8.4 画面反映確認
- `GET /api/home`
  - `recent_alerts` 反映を確認
- `GET /api/symbols/:symbolId`
  - `recent_alerts` / `latest_processing_status` を確認

## 9. secret を漏らさない運用ルール
- docs、PR、チケット、スクリーンショットに token / shared_secret の実値を残さない
- 実行ログ共有時は Query token を必ずマスクする
- payload サンプルは `<YOUR_SHARED_SECRET>` のようにプレースホルダ化する
- DB 確認時に token / shared_secret の列を select しない

## 10. 補足（現行実装仕様との整合）
- webhook は token（Query/Bearer）に対応
- payload `shared_secret` 照合に対応
- unresolved / duplicate は受信自体は受理しつつ status で判別
- summary 失敗時は `latest_job.error_message` で原因追跡可能
