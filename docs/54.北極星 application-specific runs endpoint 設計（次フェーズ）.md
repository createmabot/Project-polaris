# 北極星 application-specific runs endpoint 設計（次フェーズ）

更新日: 2026-05-12

## 1. 目的

Symbol Strategy Application 配下の run 履歴を取得する endpoint 方針を整理する。

今回の目的は、既存 `GET /api/symbols/:symbolId/strategy-applications` の latest run summary と、application 単位の any run 履歴検索を分離し、次の最小実装へ進む場合の候補を絞ることである。

この文書は docs-only の設計整理であり、今回 API / backend / frontend / DB / Prisma schema / test は変更しない。

## 2. 現在地

- P3 は完了扱いである。
- Symbol Strategy Application / Run / Backtest report の基本関係は docs/52 を正本とする。
- 現行 DB には `SymbolStrategyApplication` と `SymbolStrategyApplicationRun` が存在する。
- 現行 `GET /api/symbols/:symbolId/strategy-applications` は application list 用 endpoint として、`latest_run`、`latest_backtest_report`、`latest_reports_by_source`、pagination meta を返す。
- 現行 list endpoint の `run_type` / `run_status` filter は latest run 基準である。
- CSV import run は `Backtest` / `BacktestImport` / `SymbolStrategyApplicationRun` を作る。
- internal backtest run は `InternalBacktestExecution` / `SymbolStrategyApplicationRun` を作り、succeeded execution の report conversion 後に `Backtest` を link できる。
- internal backtest report では `BacktestImport` を作らない。

## 3. endpoint 第一候補

第一候補は次とする。

```text
GET /api/symbol-strategy-applications/:applicationId/runs
```

目的:

- application 配下の run 履歴を時系列で取得する。
- latest run だけでなく、過去 run を `run_type` / `run_status` で検索する。
- CSV import run、internal backtest run、report conversion 後の Backtest link を同じ run 履歴上で確認できるようにする。
- `SymbolDetail` の application row を肥大化させず、必要になった時点で run history 表示へ接続する。

## 4. 既存 list endpoint との役割分離

### 維持する endpoint

```text
GET /api/symbols/:symbolId/strategy-applications
```

役割:

- SymbolDetail の saved application list 用。
- application summary、strategy、strategy version、latest run summary、latest backtest report summary、CSV / internal latest pair を返す。
- `run_type` / `run_status` は latest run 基準の filter として維持する。
- 未指定時は `status=active` の application list を返す。

この endpoint では、any run 履歴検索を扱わない。

### 新設候補 endpoint

```text
GET /api/symbol-strategy-applications/:applicationId/runs
```

役割:

- application-specific な run history 用。
- `run_type` / `run_status` は application 配下の runs[] そのものに対する filter とする。
- 過去に failed run があるか、CSV import run と internal backtest run がどう並んだか、report conversion 後にどの Backtest へ link したかを確認する。

## 5. query 候補

初回候補は以下に絞る。

| query | allowed value | 既定 | 意味 |
|---|---|---|---|
| `run_type` | `csv_import` / `internal_backtest` | 未指定 | application 配下の runs[] を run type で絞る |
| `run_status` | `queued` / `running` / `succeeded` / `failed` / `canceled` | 未指定 | application 配下の runs[] を run status で絞る |
| `page` | positive integer | `1` | pagination page |
| `limit` | positive integer | `20` | pagination limit。上限は既存 list endpoint と揃え、初回は `50` を候補にする |
| `sort` | `created_at` / `updated_at` | `created_at` | run history の並び替え対象 |
| `order` | `asc` / `desc` | `desc` | run history の並び順 |

判断:

- `status` は application parent の status と混同しやすいため、この endpoint では `run_status` に限定する。
- `report_source` は run type と重複しやすいため初回 query には入れない。report 有無は response の linked backtest summary で判断する。
- `strategy_id` / `strategy_version_id` は application が既に固定しているため入れない。
- `backtest_id` / `execution_id` での逆引きは初回対象外にする。

## 7. CSV import run の扱い

CSV import run は次の関係で返す。

- `run_type = csv_import`
- `status = succeeded` または `failed`
- `backtest_id` は原則あり
- `backtest_import_id` はあり
- `internal_backtest_execution_id` は null
- `linked_backtest` は BacktestDetail link 用 summary
- `linked_backtest_import` は parse status / parse error / import metadata の summary
- `linked_internal_backtest_execution` は null

parse failed の場合:

- run は `failed`
- linked backtest は `import_failed` など既存 Backtest status を返す
- linked import は `parse_status=failed` と `parse_error` を返す
- BacktestDetail へ遷移できるかどうかは既存 UX に従い、runs endpoint は link 情報を隠さない

## 8. internal backtest run の扱い

internal backtest run は次の関係で返す。

- `run_type = internal_backtest`
- `status = queued|running|succeeded|failed|canceled`
- `internal_backtest_execution_id` はあり
- `backtest_import_id` は null
- `linked_internal_backtest_execution` は status / requested / started / finished / error の summary

report conversion 前:

- `backtest_id` は null
- `linked_backtest` は null
- succeeded execution であれば既存 report conversion endpoint の候補になり得る

report conversion 後:

- `backtest_id` はあり
- `linked_backtest` は BacktestDetail link 用 summary
- `linked_backtest_import` は null
- `SymbolStrategyApplicationRun.internalBacktestExecutionId` と `backtestId` は併存する

## 9. report conversion 後の backtest link

report conversion 後の Backtest link は runs endpoint でも返す方針にする。

理由:

- run history を見た時点で、internal execution と作成済み Backtest report の対応を確認できる。
- `BacktestDetail` は個別 report detail として維持する。
- runs endpoint は report 本文や metrics 詳細を背負わず、link summary に留める。

CSV import 由来 report:

- `linked_backtest.execution_source = tradingview`
- `linked_backtest_import` あり
- parsed summary 本体をどこまで返すかは初回は最小にする。必要なら `parsed_summary_preview` を後続候補とする。

internal backtest 由来 report:

- `linked_backtest.execution_source = internal_backtest`
- `linked_backtest_import = null`
- `linked_internal_backtest_execution` あり
- result summary / artifact pointer の詳細は BacktestDetail または internal execution detail に寄せる。

## 10. application-specific reports endpoint との境界

reports endpoint 候補:

```text
GET /api/symbol-strategy-applications/:applicationId/reports
```

役割候補:

- application 配下の Backtest report 一覧を source 別、metrics preview 付きで取得する。
- CSV import report と internal backtest report の比較 UX を主目的にする。
- run がない report、または report aggregation を扱う必要が出た場合に検討する。

runs endpoint との境界:

- runs endpoint は実行履歴が主語。
- reports endpoint は検証レポート一覧が主語。
- runs endpoint は queued / running / failed / canceled も返す。
- reports endpoint は backtest report が存在するものに絞る。
- runs endpoint の `linked_backtest` は link summary までとし、metrics 横並びや AI summary 比較は reports endpoint または BacktestDetail 側に寄せる。

初回判断:

- まず runs endpoint を優先する。
- reports endpoint は、SymbolDetail または BacktestDetail 以外で「同一 application の report 一覧 / 比較」を本格表示する段階まで保留する。

## 11. DB / Prisma schema 判断

初回の `GET /api/symbol-strategy-applications/:applicationId/runs` は、既存 schema で実現可能と判断する。

利用する既存 relation:

- `SymbolStrategyApplication.runs`
- `SymbolStrategyApplicationRun.backtest`
- `SymbolStrategyApplicationRun.backtestImport`
- `SymbolStrategyApplicationRun.internalBacktestExecution`
- `SymbolStrategyApplication.symbol`
- `SymbolStrategyApplication.strategyRule`
- `SymbolStrategyApplication.strategyRuleVersion`

今回の判断:

- DB migration は不要。
- Prisma schema change は不要。
- `run_type` / `run_status` / `created_at` / `updated_at` の初回検索は既存 index で開始できる。
- 大量データ時のみ、`applicationId + runType + createdAt`、`applicationId + status + createdAt`、latest/cache/read model を後続判断にする。

## 12. error / validation 方針候補

- application が存在しない場合は `404 NOT_FOUND`。
- `run_type` の invalid value は `400 VALIDATION_ERROR`。
- `run_status` の invalid value は `400 VALIDATION_ERROR`。
- `page` / `limit` が positive integer でない場合は `400 VALIDATION_ERROR`。
- `limit` は上限超過時に `400 VALIDATION_ERROR`。
- `sort` / `order` の invalid value は `400 VALIDATION_ERROR`。

## 13. 最小実装候補

次に実装へ進む場合は、候補を以下 4 つに絞る。

1. `GET /api/symbol-strategy-applications/:applicationId/runs` の read-only backend API を追加する。
2. `run_type` / `run_status` / `page` / `limit` / `sort` / `order` の validation と pagination meta を実装する。
3. response に application summary、runs[]、linked backtest / import / internal execution summary を返す。
4. `SymbolDetail` にはすぐ新規 UI を足さず、まず API contract と docs / manual API 確認に留める。

やらないこと:

- 既存 `GET /api/symbols/:symbolId/strategy-applications` の response meaning 変更。
- latest run filter を any run filter に変えること。
- DB migration。
- Prisma schema change。
- new report comparison screen。
- application-specific reports endpoint の同時実装。
- frontend の大きな表示追加。
- Visual regression。

## 14. 確認観点

docs-only の今回確認:

- 既存 list endpoint の latest run filter と any run 履歴検索の役割が分離されている。
- CSV import run、internal backtest run、report conversion 後の Backtest link の返し方が分かる。
- application-specific reports endpoint との境界が分かる。
- DB / Prisma schema 変更が初回不要である判断が明記されている。
- 次の最小実装候補が 2〜4 個に絞られている。

実装に進む場合の確認:

- `GET /api/symbol-strategy-applications/:applicationId/runs` が active / archived application のどちらでも read-only に取得できる。
- `run_type=csv_import` で CSV import run だけを返す。
- `run_type=internal_backtest` で internal backtest run だけを返す。
- `run_status=queued|running|succeeded|failed|canceled` が run history 基準で効く。
- CSV import run では linked backtest import summary が返る。
- internal backtest report conversion 後は linked backtest summary と linked internal execution summary が同じ run に併存する。
- internal backtest report では linked backtest import summary が null である。

## 15. application-specific reports endpoint との更新境界

application-specific reports endpoint の詳細設計は docs/55 を正本とする。

更新後の境界:

- runs endpoint は実行履歴が主語であり、report が存在しない queued / running / failed / canceled run も扱う。
- reports endpoint は検証 report 一覧 / 比較 UX が主語であり、Backtest report が linked された run を中心に扱う。
- CSV import report と internal backtest report の source / metrics / linked run 整理は reports endpoint に寄せる。
- runs endpoint の `linked_backtest` は link summary に留め、metrics 横並び比較や BacktestDetail link 一覧は reports endpoint または BacktestDetail 側に寄せる。
