# 北極星 AI summary 自動生成運用設計（次フェーズ）

更新日: 2026-05-13

## 1. 目的

本ドキュメントは、Backtest report の AI summary を「ユーザーが押したら作る」状態から、どのタイミングで自動生成・再生成・失敗管理するかを整理する運用設計である。

今回は CSV import report auto enqueue の初回実装範囲を正本化する。DB migration、Prisma schema 変更、internal backtest report conversion 後 enqueue、表示起点 enqueue、batch / scheduled job、polling 本格化は行わない。

## 2. 前提

- P3、UI foundation completion pass、IA / navigation debt cleanup、Report comparison UX phase 2、AI summary / artifact operations phase は完了扱い。
- AI summary の状態は既存 `ai_jobs` と `ai_summaries` を正本にする。
- Backtest AI summary は `summary_scope=backtest_review`、`target_entity_type=backtest` を基本にする。
- provider 境界は Home / Symbol / Comparison / Backtest と同じ `HOME_AI_PROVIDER=stub|local_llm|openai_api` を利用する。
- `AI_ENABLE_STUB_FALLBACK=false` が既定であり、provider failure は `ai_jobs=failed` として扱う。
- secret / token / API key / local path の実値を docs、PR、log に残さない。

## 3. 自動生成対象

| 対象 | 初回方針 | 理由 |
|---|---|---|
| CSV import report | 対象候補 | parsed summary が揃った直後に input が安定するため |
| internal backtest report | 対象候補 | report conversion 完了後に result_summary / artifact_pointer が固定されるため |
| BacktestDetail 個別 report AI summary | 表示・手動生成は維持 | 個別 report detail の確認場所として既存責務を維持するため |
| ApplicationDetail report history | 自動生成はしない | report history の入口であり、詳細は BacktestDetail に送るため |
| BacktestComparisonDetail | 対象外 | 保存済み pairwise comparison の再訪画面であり、report summary 自動生成とは別責務のため |
| AI summary 同士の比較 | 対象外 | 今回は自動生成 policy のみを決め、比較 UX は後続判断にするため |

## 4. CSV import report と internal backtest report の input 差

### CSV import report

- report 作成元は TradingView CSV import。
- 主 input は `BacktestImport`、`parsedSummaryJson`、import file metadata、同一 Backtest 内の parsed imports、comparison diff。
- TradingView 実行結果の文脈を含める。
- parsed summary がない場合は、summary 生成は自動 enqueue しない。
- parse failed import は保存するが、自動生成対象にはしない。過去 parsed import があれば手動生成では既存ロジックに従い部分成立を許容する。

### internal backtest report

- report 作成元は internal backtest execution の report conversion。
- `BacktestImport` は作成されない。
- 主 input は `Backtest.strategySnapshotJson.result_summary`、`artifact_pointer`、`internal_backtest_execution_id`、strategy snapshot。
- `artifact_pointer` は metadata であり、artifact file read / download / diff は行わない。
- `result_summary` が不足する場合は deterministic fallback として「入力不足」を structured_json に残す方針にし、自動 enqueue は見送る。

## 5. 自動生成タイミング候補の比較

| 候補 | 初回採用 | 判断 |
|---|---|---|
| CSV import 完了時 | 条件付き候補 | `parse_status=parsed` になった時点で input が安定する。初回自動化するなら第一候補。 |
| BacktestImport parsed summary 作成後 | 条件付き候補 | CSV import 完了時と同義。parse failed は対象外。 |
| internal backtest report conversion 完了時 | 条件付き候補 | internal report の input が固定される。CSV とは別系統なので初回実装では同時に入れない。 |
| BacktestDetail 初回表示時 | 見送り | page view で provider cost / latency が発生し、duplicate enqueue を避けにくい。 |
| ApplicationDetail report履歴表示時 | 見送り | 履歴閲覧だけで大量 enqueue される可能性がある。 |
| ユーザー操作時のみ | 現状維持 | 既存 manual generate button を維持する。 |
| batch / scheduled job | 後続判断 | retry / cost cap / 対象選定が必要で初回には重い。 |

初回実装する場合の推奨は「report 作成直後に enqueue するかどうか」を 1 系統だけで試すこと。優先順位は次の通り。

1. CSV import parsed report 作成直後。
2. internal backtest report conversion 完了直後。

理由は、CSV import はユーザー操作直後に parsed summary が明確で、既存 manual generate endpoint との比較がしやすいため。internal backtest は report conversion の完了タイミングが別であり、CSV と同時に入れると failure / retry の切り分けが難しくなる。

## 6. 手動生成との共存

- 既存 `POST /api/backtests/:backtestId/summary/generate` と BacktestDetail の `AI総評を生成` button は維持する。
- 自動生成済みの場合も、手動再生成は後続で許可候補とする。ただし初回は同一 `inputSnapshotHash` の重複生成を避け、既存 summary を返す。
- failed の場合は手動 retry を許可する方針にする。
- stale summary は `inputSnapshotHash` が変わった場合に stale とみなす。初回 UI では stale 表示を必須にしない。
- duplicate enqueue は、同一 target + summary_scope + inputSnapshotHash の succeeded summary、または queued/running job がある場合に防止する。
- BacktestDetail は自動生成の有無に関わらず、生成済み summary を read-only に表示し、未生成時は manual button を表示する。

## 7. job / retry / failure 方針

### status 遷移

`ai_jobs.status` は既存どおり次を正とする。

- `queued`
- `running`
- `succeeded`
- `failed`

初回自動生成でも、状態保存先は既存 `ai_jobs` に寄せる。新規 table は作らない。

### idempotency

- idempotency key 相当は `summary_scope + target_entity_type + target_entity_id + inputSnapshotHash` とする。
- `succeeded` summary が同一 hash で存在する場合は、新規 provider 呼び出しを行わない。
- `queued` または `running` job が同一 key で存在する場合は、新規 enqueue しない。
- `failed` job は手動 retry または明示 retry job で再試行可能にする。

### retry

- 初回自動生成では provider retry を増やさない。
- 既存 provider boundary の retry / fallback 設定に従う。
- automatic retry は最大 1 回を上限候補にし、導入する場合は別 PR で docs と実装を同時に固定する。
- timeout / provider error は `ai_jobs.errorMessage` に保存するが、UI には sanitized summary のみ出す。

### failure 表示

- UI に出す error は `AI summary generation failed` 相当の短い説明に留める。
- provider endpoint、API key、raw prompt、local path、stack trace は UI / docs / PR に出さない。
- 詳細切り分けは internal logs と `ai_jobs` の sanitized fields で行う。

## 8. provider / cost / latency 方針

- `stub`: test / local deterministic 確認用。自動生成でも cost は発生しないが、本番品質判断には使わない。
- `local_llm`: 既定 provider。自動生成の初回候補は local_llm 前提で低頻度に限定する。
- `openai_api`: 明示設定時のみ利用。自動生成対象を広げる前に cost cap / rate limit / opt-in 条件を設計する。
- fallback: `AI_ENABLE_STUB_FALLBACK=true` の場合のみ stub fallback を許可する。既定は失敗を failed job として残す。
- 生成が遅い場合、BacktestDetail は polling を本格化せず、手動再読み込みまたは後続の read-only status 表示で扱う。
- 自動生成による過剰コストを避けるため、初回は report 作成直後の 1 系統、parsed / completed report、同一 hash 重複防止、provider failure 時の無限 retry 禁止を条件にする。

## 9. 画面別運用方針

### BacktestDetail

- 個別 report の AI summary 表示と manual generation の責務を持つ。
- 自動生成導入後も manual button は残す。
- queued / running / failed の永続表示を行うには、既存 `GET /api/backtests/:backtestId` の optional read-only 拡張で latest job status を返す必要がある。今回は実装しない。

### ApplicationDetail

- report history の入口に留める。
- row では summary 有無や generation status を本格表示しない。
- 将来表示する場合は、既存 reports endpoint の optional read-only field として `ai_summary_status` 程度に限定する。
- ApplicationDetail 表示を契機に enqueue しない。

### BacktestComparisonDetail

- 保存済み pairwise comparison の AI summary 表示を担当する。
- Backtest report の AI summary 自動生成対象にはしない。
- AI summary 同士の比較は対象外。

## 10. 初回実装する場合の最小候補

今回の初回実装は、次の 1 系統に限定する。

1. CSV import parsed report 作成直後の enqueue。

internal backtest report conversion 完了直後の enqueue は後続候補として残す。

実装条件:

- DB migration / Prisma schema 変更なし。
- 既存 `ai_jobs` / `ai_summaries` / provider boundary を利用する。
- 既存 manual generate endpoint を壊さない。
- duplicate enqueue guard を先に入れる。
- provider error / timeout は failed job として保存し、UI には sanitized error のみ表示する。
- 既存 `backtest-ai-summary.e2e.test.ts` は manual generation の回帰として維持し、CSV import auto enqueue は `backtest-import.e2e.test.ts` と `symbol-strategy-applications.e2e.test.ts` で固定する。

## 10-1. CSV import report auto enqueue first implementation

今回の実装範囲:

- direct route `POST /api/backtests/:backtestId/imports` と application 起点 route `POST /api/symbol-strategy-applications/:applicationId/csv-import` の両方で、CSV parse が `parsed` になり、`BacktestImport.parsedSummaryJson` と Backtest report が紐づいた直後に Backtest AI summary generation を起動する。
- 生成経路は既存 `generate_backtest_review_summary`、`ai_jobs`、`ai_summaries`、Home AI provider boundary を再利用する。
- 新規 table、DB migration、Prisma schema 変更は行わない。
- `requestPayload.trigger=csv_import_auto`、`source_import_id`、`input_snapshot_hash` を `ai_jobs.requestPayload` に保存し、運用上の識別に使う。
- 同一 `inputSnapshotHash` の succeeded summary が既にある場合は自動 enqueue しない。
- 同一 `inputSnapshotHash` の queued / running job が既にある場合は自動 enqueue しない。
- 同一 `inputSnapshotHash` の failed job が既にある場合は自動 retry しない。手動生成 endpoint による retry 余地を維持する。
- CSV parse failed import は保存するが、自動 enqueue しない。
- 自動 enqueue / generation が失敗しても CSV import response は壊さず、import / report 保存を優先する。
- 既存 `POST /api/backtests/:backtestId/summary/generate` と BacktestDetail の手動生成 button は維持する。
- BacktestDetail / ApplicationDetail 表示起点での enqueue は行わない。
- read-only status API / UI の追加は今回は行わない。`GET /api/backtests/:backtestId` は従来どおり `ai_review.status=available|unavailable` を返す。

実装上の注意:

- 自動起動は CSV import report 系だけに限定し、direct Backtest import と application 起点 CSV import の parsed report 作成直後を同じ対象として扱う。
- internal backtest report は今回対象外であり、report conversion 後 enqueue は後続候補のまま残す。
- provider failure / timeout の扱いは既存 summary generation と同じく `ai_jobs=failed` に寄せる。
- docs / PR / log に secret、token、API key、local path の実値を残さない。

## 12. 今回の結論

CSV import parsed report 作成直後の auto enqueue を初回実装として固定する。

この PR では、backend guard、既存 manual generate との共存、CSV import failure 非波及、既存 test、docs 更新を同時に扱う。status 表示改善、internal backtest report conversion 後 enqueue、表示起点 enqueue、batch / scheduled job は後続判断とする。
