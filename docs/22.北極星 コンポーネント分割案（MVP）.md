# 北極星 コンポーネント分割案（MVP）

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVPにおいて、フロントエンド実装を進めるための **コンポーネント分割方針** を定義するものである。

これまでに、画面一覧、画面別入出力サンプル、API JSON サンプル、実装チケット、スプリント計画、DBマイグレーション順序まで整理してきた。

一方で、画面仕様が固まっても、フロント実装では以下の問題が起きやすい。

- 画面ごとに巨大コンポーネントができる
    
- APIレスポンス整形が画面内に散らばる
    
- AI summary 表示ロジックが各画面に重複する
    
- table / card / status 表示がバラバラになる
    
- 再利用すべき部品と画面専用部品が混ざる
    

そのため本資料では、MVPで無理なく実装・保守できるよう、  
**画面コンテナ / セクション / 再利用 UI / データ変換レイヤー** に分けて整理する。

---

## 2. 基本方針

### 2.1 画面 = 1コンポーネントにしない

ページコンポーネントは、

- データ取得
    
- セクション構成
    
- 画面遷移  
    に責務を絞る。
    

表示の細部は下位コンポーネントへ分離する。

### 2.2 APIレスポンスを UI に直接渡さない

各画面で **view model へ整形** してから描画する。

### 2.3 再利用単位は「見た目」だけでなく「意味」で切る

たとえば単なる Card だけでなく、

- AlertCard
    
- SummaryPanel
    
- MetricComparisonTable
    
- ReferenceList  
    のように、業務意味を持った部品も定義する。
    

### 2.4 AI表示系は共通化する

北極星では AI summary の表示が多いため、

- title
    
- confidence
    
- insufficient_context
    
- payload の要点表示  
    を共通化する。
    

### 2.5 MVPでは汎用化しすぎない

最初から design system を作り込みすぎず、  
明確に再利用されるものだけ共通化する。

- ReferenceItem
    
- NoteEditorForm

例:

## 19. MVPで避けたい分割

### 19.1 汎用化しすぎた form builder

最初から複雑な汎用 form 化はしない。

### 19.2 1画面1巨大コンテナ

HomePage や StrategyLabPage にロジックを詰め込まない。

### 19.3 AI summary ごとの完全個別描画

SummaryPanel をベースにし、種類差分だけ下位で吸収する。

### 19.4 table を完全汎用化しすぎる

MVPでは domain-specific table の方が実装しやすい。

---

## 20. コンポーネント設計上の重要判断

### 20.1 SummaryPanel を共通中核にする

北極星では AI summary が多いため、最重要共通部品にする。

### 20.2 ReferenceList と AlertCard を domain component として扱う

単なるリストではなく、意味のある再利用単位だから。

### 20.3 view-model を先に作る

UI の安定性は API ではなく view-model で担保する。

### 20.4 Strategy / Backtest は専用部品で切る

複雑なため、汎用部品に寄せすぎない。

## 23. 結論

北極星MVPのコンポーネント分割では、  
**Page / Section / Domain Component / Primitive UI / ViewModel** の5層で責務を分けるのが実装しやすい。

そのうえで、MVPでは以下を重視する。

- 画面コンポーネントを巨大化させない
    
- APIレスポンスは view-model へ整形してから使う
    
- AI summary 表示は共通化する
    
- Strategy / Backtest は専用部品で扱う
    
- 再利用性よりも責務の明瞭さを優先する
    

この分割方針により、北極星のフロント実装を無理なく進めやすい状態にする。
