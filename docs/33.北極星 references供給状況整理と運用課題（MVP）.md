# 北極星 references供給状況整理と運用課題（MVP）
更新日: 2026-05-01

## 1. 目的

本資料は、実データ運用時の `external_references` 供給状況を観測し、`news / disclosure / earnings` の偏り、0件時の画面挙動、AI要約への影響、運用上の確認手順を固定するための補助資料である。

今回の対象は観測と整理であり、references 収集ロジックの全面改修や新規外部API導入は含まない。

## 2. 現行実装の要点

### 2.1 DBモデル

references は `external_references` テーブルに保存される。

- `symbol_id`
- `alert_event_id`
- `reference_type`
- `title`
- `source_name`
- `source_url`
- `published_at`
- `summary_text`
- `metadata_json`
- `dedupe_key`
- `relevance_score`

`reference_type` は現行実装では文字列で保持し、実質的に次の3種を使う。

- `news`
- `disclosure`
- `earnings`

### 2.2 取得経路

`backend/src/references/collector.ts` の現行実装では、source は次の3経路で構成される。

- `news`
  - Google News RSS search
  - `source_name = google_news_rss`
- `disclosure`
  - TDnet 日次開示一覧
  - `source_name = tdnet_disclosure`
- `earnings`
  - TDnet 日次開示一覧から決算系タイトルを抽出
  - `source_name = tdnet_earnings`

### 2.3 収集ジョブの入口

- TradingView webhook 後:
  - `collect_references_for_alert`
  - `generate_alert_summary`
- Symbol AI論点カード生成時:
  - 既存 `external_references` を選択して使用
- Comparison AI比較総評生成時:
  - 各 symbol の `related_references` を合算して使用

## 4. 取得経路ごとの現状

### 4.1 news

- 取得想定:
  - Google News RSS
- 観測結果:
  - 実DBに 6件存在
- 特徴:
  - 最も供給されやすい
  - 銘柄名ベース検索のため、一般ニュース寄りになることがある

### 4.2 disclosure

- 取得想定:
  - TDnet 日次開示一覧
- 実装:
  - symbol code / tradingview symbol / display name を使って TDnet 行を照合
- 観測結果:
  - 実DBでは 0件
- 0件の解釈:
  - collector 未実装ではない
  - 観測期間に該当開示がなかった、または照合条件に一致しなかった可能性がある

### 4.3 earnings

- 取得想定:
  - TDnet 日次開示一覧から決算関連タイトルを抽出
- 実装:
  - `決算短信` `四半期決算` `業績予想` `業績修正` などのキーワード判定
- 観測結果:
  - 実DBでは 0件
- 0件の解釈:
  - collector 未実装ではない
  - 観測期間の対象銘柄に決算系開示がなかった、または title 判定に入らなかった可能性がある

### 5.1 SymbolDetail

- AI:
  - `POST /api/symbols/:symbolId/ai-summary/generate` は references 0件でも実行可能
  - `generation_context_json.reference_count` には実際の件数が残る

### 5.2 Comparison

- API:

- 画面:
  - 0件時は symbol card 側で空状態を表示

- AI:
  - `POST /api/comparisons/:comparisonId/generate` は references 0件でも実行可能
  - `generation_context_json.reference_count` に比較時点の合算件数が残る

### 5.3 Alert summary

- API:
  - `GET /api/alerts/:alertId/summary`
  - `latest_job` で最新の `generate_alert_summary` 状態を確認できる
- 収集:
  - webhook 後に `collect_references_for_alert` が走る
  - `response_payload.saved_count / skipped_count / ref_ids` が `ai_jobs` に残る
- AI:
  - references 不足時でも summary 生成自体は試行される
  - provider failure 時は `latest_job.error_message` で追跡する

### 5.4 Home AIデイリーサマリー

- Home 画面は `daily_summary.insufficient_context` を表示できる
- ただし、references 数から機械的に `insufficient_context=true` を立てているわけではない
- 実運用では `body_markdown` の内容と references の実数を併せて確認する必要がある

## 6. 実観測で分かった注意点

### 6.1 `insufficient_context` は reference_count 0 と常に一致しない

実DBでは、`generation_context_json.reference_count = 0` の symbol thesis summary が複数存在したが、`structured_json.insufficient_context` は `false` のものがあった。

これは現時点では次の意味を持つ。

- `insufficient_context` は AI 出力依存であり、reference count から自動補正されていない
- 「references が足りないか」を判定するには、次を併せて確認する必要がある
  - 画面上の reference 内訳
  - `generation_context_json.reference_count`
  - `structured_json.insufficient_context`

### 6.2 Comparison は片側だけ references が少なくても生成できる

- 比較総評は、比較対象ごとの均等性を保証していない
- 片側だけ references 0件でも、もう片側の references と snapshot に寄って summary が出る可能性がある

### 6.3 alert summary の失敗は references 不足とは限らない

観測時点では最新 `generate_alert_summary` job は failed だったが、原因は `LocalLLM connection failed` であり、references 収集失敗そのものではなかった。

### 7.3 DB直接確認（secret非参照）

銘柄別件数:

```sql
select
  s.symbol_code,
  s.display_name,
  er.reference_type,
  count(*) as ref_count,
  max(er.published_at) as latest_published_at
from external_references er
left join symbols s on s.id = er.symbol_id
group by s.symbol_code, s.display_name, er.reference_type
order by s.symbol_code, er.reference_type;
```

直近ジョブ:

```sql
select
  job_type,
  status,
  created_at,
  completed_at,
  error_message
from ai_jobs
where job_type in (
  'collect_references_for_alert',
  'generate_alert_summary',
  'generate_symbol_thesis_summary',
  'generate_comparison_summary'
)
order by created_at desc
limit 20;
```

## 8. 実運用での確認ポイント

## 12. 2026-05-02 追加整理: TDnet disclosure / earnings が 0件のときの切り分け

- 0件 reason 一覧

  - `tdnet_no_file_for_date`
  - `tdnet_parse_zero_rows`
  - `tdnet_rows_exist_but_no_symbol_match`
  - `tdnet_symbol_match_but_no_earnings_title`
  - `tdnet_no_matching_disclosure_in_lookback`
  - `tdnet_no_matching_earnings_in_lookback`
- collector 診断で残す情報
  - `dates_checked`
  - `fetched_dates`
  - `fetch_failed_dates`
  - `no_file_dates`
  - `parse_zero_row_dates`
  - `rows_parsed`
  - `symbol_matches`
  - `earnings_candidates`
  - `returned_count`
  - `reason`
  - `checked_dates`
  - `per_date[]`
- 運用上の見方
  - `tdnet_fetch_failed`: 403/timeout/一時障害などで一覧取得自体が失敗している
  - `tdnet_no_file_for_date`: 対象日の一覧ファイルが存在しない。土日祝や対象日未掲載の可能性が高い
  - `tdnet_parse_zero_rows`: HTML は取得できたが、既存 parser が row を 0 件しか取れていない。TDnet HTML 構造変更の疑いがある
  - `tdnet_rows_exist_but_no_symbol_match`: 一覧 row は取れているが、symbol code / display name / tradingview symbol の照合で落ちている
  - `tdnet_symbol_match_but_no_earnings_title`: symbol 一致 row はあるが、現行 earnings keyword に入っていない
  - `tdnet_no_matching_disclosure_in_lookback`: disclosure 対象期間内に該当開示がなかった
  - `tdnet_no_matching_earnings_in_lookback`: earnings 対象期間内に決算系タイトルがなかった
- 2026-05-02 の 1 回確認結果
  - 対象 URL: `I_list_001_20260501.html`
  - HTTP status: `200`
  - parsed row count: `100`
  - `7203` / `6758` 該当 row: `0`
  - 解釈:
    - 少なくともこの 1 日については `tdnet_parse_zero_rows` ではない
    - `7203` / `6758` の `disclosure` / `earnings` 0件は、「その日の TDnet 一覧に対象銘柄がいない」可能性が高い
    - lookback 日数、照合条件、earnings keyword のどこで 0 件になったかは diagnostics で継続確認する
