# 北極星 MVP後ロードマップ・バックログ整理

更新日: 2026-05-08

## 1. 目的

本ドキュメントは、北極星MVP完了後の開発再開にあたり、MVP後バックログを優先順位付きで整理するための正本である。

目的は次のとおり。

- MVP完了後に何から進めるかを明確にする
- 不具合修正、運用品質改善、データ供給改善、UI改善、拡張機能を分類する
- 優先度を `P1 / P2 / P3` で揃える
- 次スプリント候補を決める
- MVP完了済みの主要導線を壊さないよう、実装前の判断基準を残す

## 2. 前提

- 北極星MVPは完了済み。
- `docs/37` で総合判定 `OK`、`MVP完了可`、`NGなし` を記録済み。
- `docs/38` でMVP完了報告を記録済み。
- P1完了整理は `docs/40` `docs/41` `docs/42` を正本とする。
- P2完了整理とP3着手判断は `docs/44.北極星 MVP後P2完了整理とP3着手判断.md` を正本とする。
- 今後の作業はMVP後フェーズとして扱う。
- 新機能実装に入る前に、対象タスクが `P1 / P2 / P3` のどこに属するかを確認する。
- docs は引き続き正本として扱う。

## 3. 優先度の定義

| 優先度 | 定義 | 判断基準 |
|---|---|---|
| P1 | MVP直後に先に片付けるべき運用品質・既知要確認 | MVP完了時点の要確認に直結する、または次の実装判断の前提になる |
| P2 | 主要導線の品質向上・データ供給改善 | MVP blockerではないが、日常運用の精度・信頼性を上げる |
| P3 | 拡張機能・中長期改善 | MVP後の価値拡張だが、P1/P2の整理後でよい |

## 5. P1詳細

### 5.1 Home card の表示件数・並び順・snapshot依存の仕様化

背景:

- MVP受入時、watchlist / positions の API CRUD は成立した。
- ただし、追加した temp symbol が Home の watchlist / positions card で即時視認できることは未再現だった。
- 原因候補は、表示件数、並び順、snapshot有無、seed/実データ混在である。

整理したいこと:

- Home card は何件まで表示するか
- 並び順は何を正とするか
- snapshot がない symbol を表示対象に含めるか
- Homeで見えない場合でも、管理画面とSymbolDetail導線が成立していればOKとするか
- `watchlist / positions API CRUD 成立` と `Home card即時視認` を同一判定にしないこと
- 仕様固定後の正本 docs は `docs/40.北極星 Home card 表示件数・並び順・snapshot依存仕様（MVP後P1）.md` とする

成果物:

- docs追記または新規docsで仕様固定
- 必要なら frontend 表示文言の軽微修正
- 必要なら E2E / frontend test の期待値更新

### 5.2 Rule Lab日本語入力のUI経路再確認

背景:

- PowerShell 経由で日本語JSONを送ると、encoding由来で文字化けし Pine generate failed になるケースがある。
- walkthrough には UTF-8 バイト列で送る手順が追記済み。
- MVP完了判定上は blocker ではないが、UI経路では実利用に耐えるか再確認したい。

整理したいこと:

- UIから日本語自然文を入力した場合に strategy / version 作成が正常に通るか
- Pine generate が成功するか
- 文字化け警告が出る場合、ユーザーに理解できる文言か
- shell経路の制約とUI経路の通常利用を docs 上で明確に分けるか

成果物:

- UI経路再確認メモ
- 必要なら walkthrough の補足修正
- 必要なら StrategyLab 文言テスト追加のタスク化

### 5.3 docs全体の文字化け完全棚卸しの範囲決め
- 棚卸し範囲と優先順位の正本 docs は docs/42.北極星 docs文字化け棚卸し範囲整理（MVP後P1）.md とする

背景:

- 主要runbookはMVP受入に耐えている。
- 一方で、過去docsや一部実装反映差分docsには文字化けが残っている。
- PowerShell 既定表示で文字化けして見える場合でも、ファイル実体が壊れているとは限らない。
- 全量修復に入る前に、修復対象と優先順位を決める必要がある。

分類方針:

| 分類 | 対象例 | 対応方針 |
|---|---|---|
| A. 現行runbook | README / walkthrough / docs/32 / docs/34 / docs/35 / docs/36 / docs/37 / docs/38 | UTF-8 明示読み取りと GitHub 表示で実体確認し、実体として壊れている場合のみ優先修正 |
| B. 正本仕様 | docs/1 / docs/2 / docs/3 / docs/16 / docs/17 / docs/29 / docs/30 | 現行実装判断に使うものから順に、UTF-8 明示読み取りで再確認したうえで修正。`docs/29` は可読、`docs/30` は参照注意付きで後続 docs と実装を正本参照先に寄せる |
| C. 履歴資料 | 古い補助資料 / 旧スプリント資料 | 参照頻度が低ければ後回し |

成果物:

- 棚卸し対象リスト
- 修復優先度
- 修復する場合の文字コード方針
- 旧資料を修復しない場合の扱い
