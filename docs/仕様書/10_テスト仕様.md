## 1. 目的

本資料は、現行テスト方針の入口である。個別テストの詳細は test files と Playwright spec を正本として扱う。

## 2. backend tests

- route / API behavior は backend e2e tests で固定する。
- Symbol Strategy Application / Backtest import / AI summary は、該当 e2e test を優先して確認する。
- Internal backtest Stage 2C cleanup 後は `/api/internal-backtests` route、application internal start / report conversion、worker、service、audit script が存在しないため、410 behavior / worker / audit tests は削除済み。
- historical `execution_source=internal_backtest` report の read-only 表示は Backtest snapshot ベースで確認する。DB execution relation / artifact table を required test 前提にしない。
- validation error、pagination、CSV import、Backtest report read-only、Backtest AI summary は backend tests で確認する。
- Backtest AI summary manual generation は、未生成時の生成と生成済み summary の明示再生成を確認する。`force=true` では同一 input snapshot の保存済み summary があっても新しい summary を作り、`force` 未指定では duplicate guard により既存 summary を返すことを backend test で確認する。BacktestDetail では `AI総評を再生成` button が表示され、押下だけで `/summary/generate` に `{ force: true }` を送るが、表示だけでは provider call を起動しないことを frontend test で確認する。
- Strategy proposal benchmark result recording は、real provider 依存 test ではなく、stub / fake data による schema、sanitization、output path validation を確認する。
- Strategy proposal Codex CLI manual JSON import は、real Codex CLI 依存 test ではなく、fake `strategy_proposal_candidates` JSON fixture で prompt endpoint、import validation、candidate count、history 保存、selection 非回帰、raw output 非露出を確認する。
- Strategy proposal Codex CLI Web検索付き prompt option は、real Codex CLI / Web search 依存 test ではなく、`web_search_prompt=false` では既存 prompt のまま、`web_search_prompt=true` では Web 検索利用時の注意文だけが追加され、import endpoint / schema / provider name / selection flow が変わらないことを確認する。
- SymbolDetail 起点の Codex CLI strategy proposal は、real Codex CLI / Web search 依存 test ではなく、symbol context がある場合に銘柄調査付き prompt guidance へ分岐し、Web検索必須、対象銘柄の基本情報 / 価格傾向 / 出来高 / volatility / 決算・業績 / news / events / sector 確認、銘柄に合う可能性、backtest caution / uncertainty の記載を要求することを確認する。schema は `strategy_proposal_candidates` v1.0 のまま、`source_type=web`、URL / citation 保存、raw prompt / raw output 保存、auto Pine / auto save / auto backtest / AI summary 自動連鎖がないことを確認する。
- Strategy proposal history full management は、provider / status / selected / search / pagination の backend behavior、safe list serialization、detail / selection 非回帰、provider quality trend 非回帰を fake history data で確認する。
- Strategy proposal sanitized provider event log persistence は、real provider 依存ではなく fake / mock data で provider generation success / failure、local_llm retry、proposal generation rate limited、Codex CLI import success / failure / rate limited、event read API pagination / filter、raw prompt / raw provider response / raw Codex output / endpoint / model 実値 / local path / stack trace / user_hint 全文 / candidate free text 非保存を確認する。
- Strategy proposal history soft archive は、new run が active で作成されること、default / `archived=active` list が archived run を除外すること、`archived=archived` / `archived=all` filter、archive / unarchive idempotency、archived detail、archived run からの candidate selection、provider event log 非影響、hard delete 不在、raw leakage boundary を backend tests で確認する。
- StrategyVersion annotation / lineage は、annotation upsert、label / note / favorite が versions list に返ること、`favorite=true` filter、annotation label / note を含む `q` 検索、lineage nodes / edges、latest_backtest_metrics、lineage response に raw natural_language_rule / raw Pine script / raw CSV / raw import text が含まれないこと、annotation 更新で Pine generation / backtest / AI summary / application apply / proposal generation が起動しないことを backend e2e で確認する。annotation sanitizer は改行、URL、endpoint、model、secret、token、credential、local path、stack trace 風文字列を保存・返却しないことを確認する。
- Pine generation LLM-first は、real `local_llm` / `openai_api` / TradingView compile に依存せず、fake provider output と deterministic baseline / fallback fixture で確認する。
- Pine generation backend tests は、StrategyVersion generate context、`JP_STOCK|US_STOCK` と `D|1D|4H|1H` の扱い、unsupported value の warning / assumption、job start endpoints、status polling endpoint、backend stage、generator -> reviewer -> repair pipeline、LLM output の最小 validation、retryable invalid output / reviewer issue の bounded repair、reviewer issue priority selection、warning-only / priority 0 issue の non-blocking success、repair prompt へ渡す selected issue 最大 3 件、`repair_attempts` / `invalid_reason_codes` / `pine_reviewer_issues`、provider failure の sanitized failure を確認する。
- local_llm Pine generation timeout は `PINE_GENERATION_LOCAL_LLM_TIMEOUT_MS` の既定値と上限 clamp を fake fetch で確認する。Strategy proposal の timeout 設定とは別に扱い、real local_llm timeout を required check にしない。
- natural language rule rewrite timeout は `RULE_REWRITE_LOCAL_LLM_TIMEOUT_MS` として Pine generation timeout とは別に扱う。rewrite provider failure は fake provider error で `provider_timeout` / `provider_invalid_response` などの sanitized reason を確認し、raw endpoint、model 実値、secret、token、local path、stack trace を API / UI に出さないことを確認する。real local_llm timeout は required check にしない。
- dev 起動時の local LLM preflight は、`local_llm` provider が有効な場合に疎通確認だけを行う。Ollama / local LLM server を自動起動せず、未起動時は sanitized message で停止する。required check では real local_llm 実体に依存せず、preflight script の分岐と provider failure 分類を fake / mock で確認する。
- Pine generation tests では deterministic reviewer が明らかな syntax / style / safety issue を検出すること、AI reviewer provider boundary が structured issue だけを返すこと、既存 normalization / validation が維持されることを fake / deterministic data で確認する。unsupported function alias、setupActive premature reset、stopLossPrice scope / guard、ADX / DMI unsafe pattern、below-vs-crossunder、overlay oscillator plot、narrative comment は reviewer issue として確認対象にする。
- Pine semantic reviewer hardening は、provider reviewer failure が deterministic pass 時に non-blocking であること、deterministic blocking issue は provider reviewer failure 時も repair / failure になること、generic setup-state variable、Donchian current-bar self-reference、entry-time ATR not persisted を fake / deterministic tests で確認する。
- Pine generation tests では raw prompt、raw provider response、raw reviewer response、endpoint、model 実値、secret、local path、stack trace が API response / persisted note / UI に出ないことを確認する。real `local_llm` / OpenAI / TradingView 実体依存 test は required check に入れない。
- Prisma schema validation は `pnpm --filter backend exec prisma validate` を使う。

## 3. frontend tests

- 画面ごとの rendering / interaction intent は page tests で固定する。
- Tailwind class の詳細は原則 test しない。
- route / link / stable text / read-only display / mutation intent を優先する。
- ApplicationDetail は read-only history、SymbolDetail は application 操作導線として分けて確認する。
- StrategyLab の Codex CLI manual JSON import UI は、prompt 作成、JSON paste / file text import、複数候補 card 表示、candidate selection、sanitized error 表示を確認する。real Codex CLI は frontend test の dependency にしない。
- StrategyLab の proposal history management UI は、provider / status / selected filter、search、pagination、empty / loading / error、filtered detail からの candidate selection を確認する。raw prompt / raw provider response / raw Codex output / endpoint / model 実値 / local path / stack trace を UI に出さないことも確認する。
- StrategyLab の proposal history soft archive UI は、default active filter、archive action で active list から隠れること、archived filter で表示されること、unarchive で active に戻ること、all filter、archived detail / selection flow、hard delete UI がないことを確認する。
- SymbolDetail から StrategyLab への symbol context handoff は、`この銘柄でストラテジー提案` link が `symbol_id` / `symbol_code` / `symbol_name` / `market` / `timeframe` / `return_to` を渡すこと、StrategyLab が実ブラウザの search params から context banner / prefill / return link を表示すること、proposal request / Codex CLI prompt request に `symbol_code` を渡すこと、表示だけで proposal generation や downstream flow を起動しないことを確認する。
- StrategyLab の symbol context apply CTA は、symbol context 付きで保存済み version がある場合だけ `この銘柄に適用` を表示すること、明示クリックで既存 `/api/symbols/:symbolId/strategy-applications` を呼ぶこと、表示・proposal generation・candidate selection・save 完了だけでは自動適用しないこと、duplicate / failure が section-local の sanitized message に閉じることを確認する。
- SymbolDetail application row には `改善版を作る` を表示しないことを確認する。SymbolDetail は適用済み strategy の概要、最新の読める検証結果、StrategyVersionDetail link、run / report 履歴、CSV import、archive / restore の入口に留める。改善版作成は BacktestDetail application report の `この検証結果をもとに改善版を作る` を主導線として確認する。
- BacktestDetail application report の `この検証結果をもとに改善版を作る` は、`symbol_strategy_application` がある場合だけ表示され、明示クリックで既存 `/api/strategy-versions/:versionId/clone` を呼び、clone 後に StrategyVersionDetail へ `source_backtest_id` と `/backtests/:backtestId` return path を渡すことを frontend fake / mock test で確認する。表示だけでは clone、Pine generation、save、backtest、AI summary generation、apply を起動しないこと、失敗表示が sanitized section-local message に閉じることも確認する。
- StrategyVersionDetail の `source_backtest_id` context は、既存 `GET /api/backtests/:id` を read-only fetch し、source report metadata、主要 metrics、保存済み AI summary excerpt / key points、関連 report note、改善メモ textarea、`LLMで新しいルール本文を作る` button を表示することを確認する。暫定の deterministic draft button と改善メモを Pine 修正依頼へ直接反映する button は表示しない。fetch failure は base version view / rule edit / Pine flows を壊さず、raw CSV、raw import text、raw prompt、raw provider response、raw Codex output、endpoint / model / secret / token / local path / stack trace を表示しない。
- Backtest AI summary は `backtest_review_summary` schema v1.0 を維持したまま、bodyMarkdown に改善向け section（問題の切り分け、改善仮説、自然言語ルール改善案、Pine修正依頼に入れるべきではない注意、次に試す検証案を含む）を持つことを backend test で確認する。low trade count / low PF / large drawdown などでは、entry / exit / risk management の見直しや再検証 action が自然言語ルール改善向けに `payload.next_actions` / `payload.overall_view` / optional `payload.rule_refinement_candidates` に入ること、CSV全文、取込本文、raw prompt、endpoint、model 実値、secret、local path、stack trace が出ないことを確認する。
- StrategyVersionDetail の source backtest 改善メモは、保存済み AI summary の `structured_json.payload.rule_refinement_candidates` を優先し、なければ `next_actions`、`overall_view`、`risks`、`strengths` を body excerpt より優先して使うことを frontend test で確認する。field がない場合は body excerpt fallback を維持する。`LLMで新しいルール本文を作る` は明示クリック時だけ rewrite draft endpoint を呼び、response draft を natural_language_rule textarea に反映する。押下だけでは save / Pine generation / Pine revision regeneration / backtest / apply を起動しないこと、backend test では保存済み rule と sanitized source backtest context を使い、raw CSV、raw import text、raw prompt、provider response、endpoint、model 実値、secret、token、local path、stack trace を response に出さないことを確認する。rewrite draft が保存済み rule と実質同一の場合は success response として返さず、natural_language_rule 保存、Pine generation job、backtest、apply を起動しないことも確認する。
- StrategyVersionDetail の rule / Pine 操作は、`ルール本文を保存`、`保存済みルールから Pine を作り直す`、`修正依頼をもとに Pine を再生成` の label と helper text を確認する。保存 button は natural_language_rule PATCH のみを呼び、Pine generation を起動しないこと、Pine 作り直し button は generation job endpoint を呼ぶこと、`source_pine_script_id` がない場合は修正再生成 button が disabled で理由が表示されること、Pine 作り直しは既存 Pine 継承ではなく保存済み自然言語ルールから新しい Pine を生成する操作として説明されること、Pine 修正再生成は既存ルールの意図を保った実装修正向けであり、strategy logic の変更は自然言語ルール本文へ反映する説明を確認する。基本情報と `この version を複製する` は折りたたみの低優先表示で、forward validation note UI は表示しないことを確認する。
- StrategyVersionLineageTree は共通 component として、StrategyVersionList / StrategyDetail / StrategyVersionDetail で表示されることを frontend tests で確認する。StrategyVersionList では履歴ツリー section、subtree centered layout、single chain の横伸び、branch parent の中央配置、missing parent の root 扱い、複数 root の縦配置、return query 付き StrategyVersionDetail link、latest metrics badge、zoom in / out / reset、favorite toggle が annotation PATCH を呼ぶこと、保存後に list / lineage cache を更新すること、label が card と tree node に表示されること、favorite filter が URL query と API path に反映されることを確認する。StrategyDetail では lineage API fetch と node link を確認する。StrategyVersionDetail では improvement context でも tree が表示され、current node に `現在表示中` が出ることを確認する。favorite / label / note 操作や tree 表示だけで detail 遷移以外の Pine generation / rewrite / save / backtest / AI summary / apply が起動しないことも確認する。
- StrategyVersion clone endpoint は、元 version に latest PineScript がある場合に clone 先へ sanitized reference copy を作成し、clone 先の `/pine` response が `parent_pine_script_id` / `source_pine_script_id` として元 PineScript id を返すことを backend test で確認する。元 version に PineScript がない場合は clone 自体は成功し、clone 先 PineScript record は作られず、`/pine` は既存の unavailable 挙動を維持することを確認する。clone 時に LLM / Pine generation / repair / reviewer が起動しないこと、元 `generationNoteJson` の raw prompt / provider response / reviewer response を丸ごとコピーしないことも確認する。
- StrategyLab / StrategyVersionDetail の Pine generation UI は、market / timeframe 選択肢、TradingView chart timeframe 前提の helper text、LLM-first / validation / bounded repair の sanitized 表示、job start、status polling、backend stage に基づく progress indicator を確認する。SSE / WebSocket / streaming は test dependency にしない。endpoint / model / raw prompt / raw provider response / raw reviewer response は UI に出さない。real TradingView compile、real local_llm、auto Pine、auto save、auto backtest、AI summary 自動生成は test dependency にしない。

## 4-1. optional benchmark

## 5. docs-only PR

- docs-only PR は最低限 `git diff --check` を確認する。
- changed docs に local path / secret 実値が残っていないことを確認する。
- PR本文は ASCII English only、docs本文は UTF-8 日本語を維持する。
- 実装変更がない場合、build / e2e を無理に実行しない。
- release acceptance / stabilization の docs-only PR では `docs/運用ドキュメント/10_release_acceptance_checklist.md` と README / `docs/0` / `docs/57` / walkthrough の導線が矛盾しないことを確認する。
