# 北極星 データモデル 現行仕様

更新日: 2026-05-25
分類: 仕様書

## 1. 目的

本資料は、現行データモデルの読み方を仕様として整理する。詳細なカラム定義は Prisma schema を正本とし、本資料では主要概念、関係、扱い方を固定する。

## 2. 正本

- DB schema の正本: `backend/prisma/schema.prisma`
- migration の正本: `backend/prisma/migrations/`
- Symbol Strategy Application / run / report 関係: `docs/52.北極星 Symbol Strategy Application DB・API設計（P3）.md`

## 3. 主要モデル

- `Symbol`: 銘柄。watchlist / position / alert / application の起点。
- `StrategyRule`: strategy の親概念。
- `StrategyRuleVersion`: strategy version。market / timeframe / Pine / validation / application の起点。
- `SymbolStrategyApplication`: symbol に strategy version を適用した親。`active` / `archived` status を持つ。
- `SymbolStrategyApplicationRun`: application 配下の run。`csv_import` / `internal_backtest` の履歴を持つ。
- `Backtest`: CSV import または internal backtest 由来の report detail。
- `BacktestImport`: CSV import 専用の import record。internal backtest report では作成しない。
- `InternalBacktestExecution`: internal engine execution。succeeded execution は Backtest report に変換できる。
- `InternalBacktestExecutionArtifact`: internal engine artifact の参照情報。
- `AiJob` / `AiSummary`: AI summary generation の状態と生成物。
- `PineGenerationJob`: Pine generation / regeneration の job progress と sanitized failure metadata。

## 4. application / run / report 関係

- `SymbolStrategyApplication` は symbol と strategy version の適用単位である。
- `SymbolStrategyApplicationRun` は application 配下の実行履歴であり、CSV import と internal backtest を同じ履歴として扱う。
- CSV import run は `BacktestImport` と `Backtest` を link する。
- internal backtest run は `InternalBacktestExecution` を link し、report conversion 後に `Backtest` も link する。
- `internalBacktestExecutionId` と `backtestId` は、internal backtest report conversion 後に同一 run 上で併存できる。

## 5. report 関係

- CSV import report は `BacktestImport` と `Backtest` を持つ。
- internal backtest report は `BacktestImport` を持たず、`strategySnapshotJson.result_summary` / `artifact_pointer` / `internal_backtest_execution_id` を使う。
- Internal backtest backend deprecation Stage 2A では、internal backtest 系 model / column / relation は削除しない。Stage 2B は read-only legacy として keep、Stage 2C は data audit 後に drop 可否を判断する。
- `execution_source=internal_backtest` の historical Backtest report は、DB cleanup 後も読める必要がある。`SymbolStrategyApplicationRun.internalBacktestExecutionId` を drop する場合は、snapshot に残る `internal_backtest_execution_id` / `result_summary` / `artifact_pointer` だけで履歴説明が足りるかを先に確認する。
- `latest_backtest_report` は latest report run の linked backtest を表す。
- `latest_reports_by_source` は source別 latest report pair を表す read-only summary であり、`latest_backtest_report` の意味を変えない。
- report comparison helper は現行データから表示する。comparison entity と metrics normalization table は追加しない。

## 6. status の扱い

- application status は `active` / `archived`。
- run status は `queued` / `running` / `succeeded` / `failed` / `canceled`。
- CSV import parse result は `BacktestImport.parseStatus` と `Backtest.status` に反映する。
- internal backtest execution status は execution と application run の双方で参照される。

## 7. 今回正本化しないもの

- metrics normalization table。
- comparison entity。
- hard delete。
- StrategyRuleMetadata table。

## 7-1. Pine generation job model 方針

Pine generation は job-based progress を持つ。`PineGenerationJob` は StrategyVersion の Pine generation / regeneration request ごとに作成され、queued / running / succeeded / failed と sanitized stage history を保持する。

保存するもの:

- `strategy_version_id` / request kind。
- job status / current stage / stage history。
- result pine script id。
- sanitized error code / message。
- failed job の切り分け用 sanitized `error_details_json`。内容は `invalid_reason_codes` と `pine_reviewer_issues` の code / severity / repair_hint に限定する。

保存しないもの:

- raw prompt。
- raw provider response。
- raw reviewer response。
- provider endpoint。
- model 実値。
- secret / token / credential。
- local path。
- stack trace。

Pine generation job は進捗確認と失敗分析のための operational metadata であり、TradingView compile automation、auto save、auto backtest、AI summary 自動生成を意味しない。

## 8. Strategy proposal history / lineage model 方針

LLM strategy proposal の proposal history / selected proposal lineage は、`StrategyProposalRun` と `StrategyProposalCandidate` の additive model として実装済み。既存 table への必須 column 追加、StrategyRuleVersion への relation、Strategy / StrategyVersion 自動保存は行わない。

追加済み model:

- `StrategyProposalRun`: 1 回の proposal generation request / provider observation / status を保存する親。
- `StrategyProposalCandidate`: run 内の normalized candidate を保存する子。

保存方針:

- run には request parameters、bounded / sanitized user_hint、sanitized provider metadata、sanitized `provider_observation`、status、candidate count、nullable FK の `selected_candidate_id` を保存する。
- candidate には provider candidate id、rank、normalized candidate JSON、selected_at を保存する。raw provider response は保存しない。
- raw prompt、raw provider response、provider secret、endpoint、model 実値、local path、stack trace は保存しない。
- local_llm failure / invalid response は candidates なしの failed run として、sanitized status / reason のみ保存する方針を第一候補にする。

初回では追加しない relation:

- `StrategyRuleVersion.createdFromProposalCandidateId` 相当。
- `selected_strategy_id` / `selected_strategy_version_id`。

理由:

- proposal selection は StrategyLab input 反映であり、Strategy / StrategyVersion 保存とは別操作である。
- StrategyVersion lineage まで同時に入れると StrategyVersion 作成 API と migration scope が広がる。
- 初回 migration は additive table 追加に限定し、既存 strategy / backtest data への rollback risk を抑える。

## 9. Strategy proposal provider event log model 方針

Strategy proposal の provider event log は、proposal history より低レイヤーの運用観測用 event として扱う。proposal history は「生成結果と候補」を残す責務、provider event log は「provider call / manual import / retry / rate limit / validation failure の発生」を sanitized metadata で残す責務に分ける。

追加 model:

- `StrategyProposalProviderEvent`: provider execution / manual import / rate limit / retry / validation failure の sanitized event。

初回 implementation の data model 方針:

- additive table 追加に限定する。
- `StrategyProposalRun` への relation は nullable とする。proposal run が作成された success / failed provider run / manual import は `proposal_run_id` で紐づけ、rate limited など run が作られない event は null のまま保存できる。
- event は `event_type`、`provider_name`、`provider_mode`、`selected_by`、`status`、`invalid_reason`、`latency_bucket`、`elapsed_ms`、`candidate_count`、`validation_error_count`、retry metadata、rate limit metadata、manual import flag、sanitized `metadata_json` を持つ。
- index は `occurred_at`、`provider_name + occurred_at`、`event_type + occurred_at`、`status + occurred_at`、`proposal_run_id` を第一候補にする。
- event write failure は proposal generation / import 本体を不要に壊さない。実装では sanitized warning に留め、raw error object / stack trace を保存・返却しない。

保存してよいもの:

- provider name / mode / selected_by。
- event type。
- status / invalid reason / latency bucket / rounded elapsed ms。
- candidate count / validation error count。
- retry used / reason / succeeded。
- rate limited flag / rate limit key source。
- manual import flag。
- sanitized metadata: schema_valid、fallback_used、fallback_reason、missing_required_field_count、affected_candidate_count、candidate_count_requested、manual_import_source など enum / count / bucket に限定した値。

保存しないもの:

- raw prompt。
- raw provider response。
- raw Codex output。
- provider endpoint。
- model 実値。
- secret / token / credential。
- local path。
- stack trace。
- user_hint 全文。
- candidate title / summary / suggested_natural_language_spec / entry_logic / exit_logic / risk_management などの自由文本文。
- raw candidateJson / raw inputJson / raw providerObservationJson 全体。
- actual IP / forwarded header value / internal rate-limit key。

初回では追加しないもの:

- provider quality trend の materialized aggregate。
- prompt regression automation。
- benchmark result DB table。
- retention / archive / hard delete。
- StrategyVersion created-from-proposal relation。

## 10. Strategy proposal history soft archive 方針

Strategy proposal history soft archive は、`StrategyProposalRun` を削除せず、通常一覧から隠すための lifecycle state として扱う。archive は「普段見ない proposal run を整理する」ための状態であり、candidate、selection lineage、provider quality trend、provider event log の履歴を消すものではない。

追加候補 field:

- `StrategyProposalRun.archivedAt DateTime? @map("archived_at")`

初回 implementation の data model 方針:

- additive nullable field 追加に限定する。既存 data は `archived_at=null` の active run として扱う。
- `@@index([archivedAt, createdAt])` を候補にし、default active list と archived list の created_at sort を支える。
- archive reason は初回では追加しない。reason は自由文になりやすく、privacy / validation / UI 境界が増えるため後続判断とする。
- hard delete、retention job、export は実装しない。
- `StrategyProposalCandidate` は archive field を持たない。run archive に従って通常 list から隠れるが、detail では candidates を確認できる。
- `StrategyProposalProviderEvent` は archive 対象外とする。event log は運用観測ログであり、proposal run の archive と独立して保持する。event の `proposalRunId` は archived run を指してもよい。

API / UI 上の意味:

- default proposal history list は archived run を除外する。
- `archived=active|archived|all` filter で通常表示、archived のみ、全件を切り替える。
- archived run の detail は確認できる。
- archived run からの candidate selection は許可する。archive は利用禁止ではなく通常一覧から隠すだけの状態であるため。ただし selection しても自動 unarchive はしない。

保存しないもの:

- archive reason 自由文。
- raw prompt。
- raw provider response。
- raw Codex output。
- endpoint / model 実値。
- secret / token / credential。
- local path。
- stack trace。

初回では追加しないもの:

- hard delete。
- retention job。
- export。
- bulk archive / bulk unarchive。
- provider event log archive。
- archive を契機にした provider quality trend 再集計条件の変更。
