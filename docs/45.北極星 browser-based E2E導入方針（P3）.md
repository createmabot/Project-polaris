# 北極星 browser-based E2E導入方針（P3）

更新日: 2026-05-09

## 1. 目的

- browser-based E2E を導入する目的を整理する
- 既存 backend / frontend test と手動 runbook の役割差を明確にする
- 最初の PoC 対象、runner、起動方式、seed 方針を固定する

## 2. 現行テスト構成

### 2.1 backend API E2E

- Fastify + Vitest で API 導線を固定している
- 既に固定されている代表導線:
  - Home / SymbolDetail / Comparison の最小 API flow
  - Backtest import / detail / AI summary
  - Strategy Lab / Pine generate
  - webhook / alert summary / queue handler
  - references collector / TDnet save path

### 2.2 frontend Vitest

- `renderToStaticMarkup` 中心で画面文言、空状態、主要ブロック表示を固定している
- `Home.test.tsx` などで placeholder / populated state / null 表示を固定している
- `StrategyLab.test.tsx` で Rule Lab の主要文言、生成結果、CSV 失敗ガイダンスを固定している

### 2.3 frontend flow test

- `test:e2e:*` script はあるが、実ブラウザ E2E ではない
- 実態は jsdom 上の Vitest flow test で、URL 組み立てや return-flow 契約を固定している
- 例:
  - Backtest list -> detail -> list
  - StrategyVersion list -> detail -> list

### 2.4 walkthrough 手動確認

- `docs/walkthrough.md` に主要手動導線がまとまっている
- 実ブラウザ表示、画面遷移、クリック、外部ツール貼り付け、CSV ファイル選択などは手動 runbook 依存

## 3. browser-based E2Eで確認したいこと

- 実ブラウザでの初期表示
- 画面遷移とリンク導線
- ボタン押下後の画面更新
- API 応答後の UI 反映
- jsdom では拾いにくい実画面の回帰

既存 test との差分:

- backend API E2E は API 契約を守る
- frontend Vitest は文言と単体表示を守る
- browser-based E2E は「画面として本当に一巡できるか」を守る

## 4. 初回PoC対象候補

| 候補 | 対象導線 | 価値 | コスト | 外部依存 | 初回向きか |
|---|---|---:|---:|---|---|
| Home → SymbolDetail | Home 表示、watchlist/positions、SymbolDetail 遷移、references 内訳、recent_alerts | 高 | 低 | なし | はい |
| Rule Lab → Pine生成 → Version Detail | StrategyLab 表示、日本語入力、generate、generated pine、detail 導線 | 高 | 中 | backend 状態と generate API | 次点 |
| Backtest CSV import → Backtest Detail | CSV file input、parse_status、detail 遷移、import 失敗表示 | 高 | 中〜高 | file upload 前提 | 初回は重い |
| Alert summary 表示 | alert detail / summary / latest_job / 本文表示 | 中 | 中 | alert fixture 前提 | 初回は後回し |

## 5. 初回PoC推奨

推奨:

- Home → SymbolDetail

理由:

- MVP の入口導線である
- seed データで成立しやすい
- 外部 API / LLM / TradingView に依存しない
- 手動 runbook での重要度が高い
- 既存 backend E2E と補完関係が明確

## 6. runner比較

| runner | 導入コスト | Vite/TS相性 | Windows / PowerShell | CI適性 | trace / screenshot | 評価 |
|---|---:|---|---|---|---|---|
| Playwright | 中 | 良い | 良い | 良い | 強い | 推奨 |
| Cypress | 中〜高 | 良い | 良い | 良い | あるが trace は弱め | 今回は次点 |
| Vitest + jsdom 継続 | 低 | 良い | 良い | 良い | なし | 実ブラウザ価値は出ない |
| 手動 runbook 継続 | 低 | N/A | 良い | なし | 目視のみ | 補助として継続 |

Playwright を推奨する理由:

- Vite / TypeScript と相性が良い
- local-only から始めやすい
- trace / screenshot / locator が揃っている
- 将来 CI に載せるときの移行負荷が比較的低い

## 7. 推奨構成

- runner: Playwright
- 初期運用: local-only
- 対象: Home → SymbolDetail
- 外部依存: なし
- AI生成: しない
- TradingView: しない
- DB: 既存 seed または E2E 用固定 seed

起動方式の初期案:

1. `docker-compose.yml` で Postgres / Redis を上げる
2. migrate / seed を適用する
3. backend / frontend は手動起動済み前提で E2E を回す

理由:

- 初回 PoC で `webServer` 自動起動や Docker 全自動化まで入れると論点が増える
- まずは導線 1 本の smoke test を成立させる方が安全

## 8. local-only PoC 導入結果

- runner: Playwright
- config: `frontend/playwright.config.ts`
- spec: `frontend/e2e/home-symbol-detail.spec.ts`
- script:
  - root: `pnpm test:e2e:browser`
  - frontend: `pnpm --filter frontend test:e2e:browser`
- 前提:
  - Postgres / Redis 起動済み
  - backend 起動済み
  - frontend 起動済み
  - seed 済み
- PoC の確認範囲:
  - Home 表示
  - watchlist または positions の symbol link 押下
  - SymbolDetail 遷移
  - `現在スナップショット` / `関連参照情報` の表示

## 9. 実行手順

```bash
pnpm run up
cd backend
pnpm exec prisma migrate deploy
pnpm exec prisma generate
pnpm exec prisma db seed
cd ..
pnpm run dev
```

別 terminal:

```bash
pnpm --filter frontend exec playwright install chromium
pnpm test:e2e:browser
```

補足:

- browser install は初回だけ必要
- `webServer` 自動起動は使わない
- Docker 全自動化はしない
- 外部 API / LLM / TradingView は前提にしない
- `POLARIS_E2E_BASE_URL` を与えれば接続先を上書きできる

## 10. 導入時の最小タスク分割

1. Playwright 依存追加と最小設定
2. Home → SymbolDetail の smoke test 追加
3. E2E 用起動手順 docs 追加
4. Tailwind 化を見据えた frontend test 方針整理
5. CI 導入可否判断
6. Rule Lab / Backtest / Alert summary への対象拡張

## 11. Tailwind 化を見据えた frontend test 方針

### 11.1 Tailwind 化で壊れやすい test

Tailwind 化の前後で次に依存する test は増やさない。

- CSS class 名
- inline style
- DOM 階層の細かい構造
- margin / padding / color / border などの見た目
- grid / flex の具体構造
- HTML 全体 snapshot

### 11.2 Tailwind 化後も守る test

優先して守るのは次の確認である。

- 画面見出し
- 主要導線のリンク
- ボタン操作
- URL 遷移
- form 入力と submit
- API 応答後の表示状態
- 空状態 / エラー状態 / loading 状態
- ユーザーが見る安定文言
- `aria role` / accessible name

### 11.3 責務分担

- Playwright:
  - 実ブラウザでの主要導線確認
  - 画面遷移確認
  - ユーザー操作確認
  - API 連携後の表示確認
- frontend Vitest:
  - component の文言固定
  - 空状態
  - helper 関数
  - URL 生成
  - CSV / flow helper
  - props に対する表示分岐
- backend API E2E:
  - API 契約
  - データ保存導線
  - queue / collector / summary などの backend flow

補足:

- Playwright では Tailwind class、色、余白、layout の厳密確認はしない
- Vitest でも Tailwind class 前提の assertion はしない
- backend E2E で UI 見た目は担保しない

### 11.4 data-testid 方針

原則は次の順で locator を選ぶ。

1. role
2. label
3. text
4. href
5. `data-testid`

`data-testid` を使ってよい候補:

- 同名リンクが複数ある領域
- 将来文言が変わりやすいが導線として重要な要素
- file input
- hidden input
- chart placeholder
- TradingView chart 枠

今回は `data-testid` の大量追加はしない。

### 11.5 今回の判断

- Tailwind 化前に Playwright spec を増やしすぎない
- 2 本目以降の Playwright spec 追加は Tailwind 化方針または共通 Layout / Component 方針が固まってから判断する
- 現在の `Home → SymbolDetail` PoC は CSS class に依存していないため維持する
- 次に実装するなら、Tailwind 化または共通 Layout / Component 設計を先に検討する
- browser E2E の対象拡張は Tailwind 化後、または UI 構造が安定した後に進める

## 12. PoC導入後も今回やらないこと

- CI 設定
- `webServer` 自動起動
- Docker 全自動化
- UI 改修
- API 改修
- DB 改修
- LLM 実行
- TradingView 連携
- Rule Lab / Backtest / Alert summary への対象拡張

## 13. 判断

- Playwright の local-only 最小 PoC は導入済み
- 現時点では `Home → SymbolDetail` PoC を維持し、対象拡張は急がない
- 次に進むなら、Tailwind 化または UI 構造安定化を踏まえた frontend test 方針に沿って 2 本目以降の spec を判断する
- CI 導入は PoC 拡張後に判断する
