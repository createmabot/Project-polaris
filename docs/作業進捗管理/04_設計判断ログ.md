# 北極星 設計判断ログ

更新日: 2026-05-16
分類: 作業進捗管理

## 1. 目的

本資料は、横断的な設計判断を短く追えるようにする decision log である。詳細仕様は各仕様書と機能別正本 docs に残し、本資料では判断内容、理由、見送り範囲を記録する。

## 2. docs architecture

判断:

- `docs/仕様書/`、`docs/運用ドキュメント/`、`docs/作業進捗管理/` の 3 分類を現行 docs architecture とする。
- 既存番号 docs は削除・rename しない。
- 新体系から旧 docs へリンクし、旧 docs は履歴資料または詳細正本として扱う。

理由:

- 既存 PR / issue / docs link を壊さないため。
- 大量移動より、読む順番と正本分類を明確にする方が安全なため。

見送り:

- archive directory への一括移動。
- 番号 docs の bulk rename。

## 3. TradingView との役割分担

判断:

- TradingView は表示、監視、一次検証、CSV export を担う。
- 北極星は自然言語変換、履歴保存、report、comparison、AI summary、internal execution result 管理を担う。

理由:

- TradingView 固有のチャート体験と、北極星の履歴・分析体験を分離するため。

見送り:

- TradingView を完全に置き換える本格 chart / execution UX。

## 4. Symbol Strategy Application

- Symbol 起点で strategy version を適用し、application / run / report の関係で履歴を管理する。
- SymbolDetail は overview と入口、ApplicationDetail は application-scoped run / report history を担当する。

理由:

- 銘柄を起点にした検証履歴を辿りやすくするため。

- 両者の metrics root と欠損値の意味を UI / docs で説明する。

理由:

見送り:

## 6. AI summary auto enqueue

判断:

- CSV import parsed report 作成直後、application 起点 CSV import parsed report 作成直後、新規 internal backtest report conversion 完了直後だけを auto enqueue 対象にする。
- BacktestDetail 表示や ApplicationDetail report history 表示を契機に enqueue しない。
- duplicate guard は同一 input snapshot hash の succeeded summary、queued / running job、failed job を対象にする。
- failed job auto retry は行わず、既存 manual generation / regeneration を維持する。
- phase 2 では `GET /api/backtests/:backtestId` に `latest_ai_summary_job` を optional read-only field として追加し、BacktestDetail で最新 job status を表示する。
- `latest_ai_summary_job` は safe な status metadata に限定し、raw request / response payload、provider endpoint、secret、local path、stack trace は返さない。

理由:

- page view 起点では provider cost / latency / duplicate enqueue の制御が難しいため。
- report 作成直後なら input が固定され、低頻度で扱えるため。
- status visibility は provider を呼ばず、既存 `ai_jobs` を読むだけで failed / running の誤読を減らせるため。

見送り:

- scheduled batch。
- automatic retry の本格化。
- polling / live status update。
- AI summary 同士の比較。
- ApplicationDetail row への job status 表示。

## 7. artifact metadata / retention / file access

判断:

- BacktestDetail は artifact pointer metadata summary、raw JSON、absence explanation を担当する。
- BacktestDetail では artifact path 系 metadata を非表示または sanitized 表示にし、raw artifact JSON でも path 系値をそのまま表示しない。
- ApplicationDetail と SymbolDetail は artifact 詳細を持たず、BacktestDetail への入口に留める。ApplicationDetail の report row には artifact path を表示しない。
- BacktestComparisonDetail は artifact diff の将来候補だが、現時点では保存済み pairwise comparison の再訪画面として維持する。
- artifact file access は既存 internal_backtests engine_actual trades / equity JSON read endpoint に限定する。
- artifact path suffix は既知 endpoint に対応する whitelist のみ許可し、UI link は execution ID と既知 route から導く。
- BacktestDetail は download 導線を持たず、metadata 表示に留める。
- download / diff / retention job は現行フェーズでは実装しない。
- PR #334 / #335 により、artifact metadata 正本仕様、retention / file access / diff boundary、BacktestDetail path 非表示、raw JSON path 値 sanitization、ApplicationDetail path 非表示 note まで完了扱いにする。

理由:

- metadata 表示と file access は権限境界、path traversal、audit、retention の設計責務が異なるため。
- frontend に raw path を渡す設計は避ける必要があるため。
- arbitrary route や local path leakage を避ける必要があるため。
- artifact diff は metrics diff / JSON diff / file diff の責務分離が必要なため。

見送り:

- download API。
- signed URL / file token 発行。
- backend proxy。
- permission boundary 本格実装。
- retention job / cleanup job。
- artifact diff / JSON diff。
- audit log 本格化。
- hard delete。

## 8. AI summary comparison UX phase 2

判断:

- phase 2 の AI summary comparison は、既存 summary を read-only に並べて理解する補助に限定する。
- BacktestDetail は個別 report detail と、同一 application 内の current / related AI summary comparison helper を担当する。
- ApplicationDetail は report history の入口であり、詳細比較は BacktestDetail に送る。
- BacktestComparisonDetail は保存済み pairwise comparison の再訪画面として維持し、本格 AI summary comparison / artifact diff は後続候補に留める。
- CSV import report summary と internal backtest report summary は input が異なるため、source 差と欠損の意味を UI / docs で説明する。
- summary missing / failed / stale は provider 再生成や polling ではなく、read-only status / note と手動生成導線で扱う。

理由:

- summary の入力文脈が source ごとに異なり、単純な同一条件比較として扱うと誤読を招くため。
- page view 起点の provider 呼び出しや polling は cost / latency / duplicate control の設計責務が重いため。
- 保存済み comparison と artifact diff は metrics diff / AI generated comparison / file access boundary の判断が必要なため。

見送り:

- comparison entity の追加または拡張。
- metrics normalization table。
- artifact diff / JSON diff / file diff。
- 自動 AI 比較生成。
- BacktestComparisonDetail の本格 AI summary comparison 画面化。

## 13. LLM strategy proposal 初回設計 / 実装

判断:

- LLM strategy proposal は StrategyLab の一時候補表示として扱い、候補選択時に natural language spec へ反映する。
- proposal 選択だけでは Strategy / StrategyVersion 保存、Pine generation、backtest 作成へ自動連鎖しない。
- 初回実装は deterministic stub provider に限定し、DB 保存や Web search / deep research は扱わない。
- 初回実装では `POST /api/strategy-lab/proposals` で構造化候補を返し、StrategyLab で候補を表示する。
- 候補選択時は title と natural language spec へ反映し、古い generated result / backtest / CSV import state は無効化する。
- proposal schema は title / summary / market assumption / timeframe assumption / strategy type / entry / exit / risk / invalidation / strengths / weaknesses / indicators / Pine feasibility / backtest caution / source / confidence / uncertainty / suggested natural language spec を持つ構造化 response にする。
- 投資助言ではなく、検証候補であることを UI / docs で明示する。

理由:

- StrategyLab は既に natural language spec から Pine 生成へ進む作業画面であり、proposal の入口として責務が自然なため。
- 保存や Pine 生成まで自動連鎖すると、ユーザー確認なしに投資戦略が確定したように見えるため。
- Web search / deep research は cost / latency / citation / freshness / safety boundary が重いため、初回実装から分離する必要があるため。

見送り:

- proposal history entity。
- DB migration。
- Web search / deep research の同期実装。
- proposal から StrategyVersion への自動保存。
- proposal から Pine generation への自動実行。
- provider cost cap / rate limit の本格実装。

## 14. LLM strategy proposal provider expansion PR 1

判断:

- provider expansion PR 1 は docs-only で扱い、実装コード、API shape、DB、Prisma schema、test は変更しない。
- stub 以外の provider を導入する前に、provider boundary、schema validation、failure handling、timeout、fallback、cost / rate limit、safety validation を仕様として固定する。
- route 層は request validation、provider response validation、sanitized error、provider metadata 表示の責務を持つ。
- provider adapter は proposal candidate generation のみに責務を限定し、Strategy / StrategyVersion 保存、Pine generation、backtest、AI summary を起動しない。
- Web search / deep research は今回も out of scope とし、`source_type=web` は将来予約値として扱う。
- fallback は silent に行わず、stub fallback を使う場合は opt-in と provider metadata で明示する。

理由:

- LLM provider を増やす前に、投資助言化、raw provider diagnostics leak、cost / latency 増大、schema mismatch による UI 破綻を防ぐため。
- Web search / deep research は citation、freshness、source reliability、job 化、cost cap の設計が必要であり、通常 proposal provider expansion と分ける必要があるため。
- proposal は検証候補の作成補助であり、保存や実行まで自動連鎖するとユーザー確認前に strategy が確定したように見えるため。

見送り:

- `local_llm` / `openai_api` provider 実装。
- Web search / deep research。
- proposal history / selected proposal lineage。
- provider cost cap / rate limit の実装。
- proposal から StrategyVersion 保存、Pine generation、backtest、AI summary への自動連鎖。

## 15. LLM strategy proposal provider boundary 実装

判断:

- deterministic stub provider を provider boundary 配下に分離し、route 層で request validation と response validation を行う。
- 現時点の provider selection は `stub` 固定とし、`local_llm` / `openai_api` は実装しない。
- response validation は enum、candidate count、必須項目、型、format、Web research basis を検査する。投資助言風 wording だけを理由に provider invalid とはしない。0 candidates は既存 UI の EmptyState として表現できるため許容する。
- user input 由来の投資助言風 wording は request validation の `VALIDATION_ERROR` にしない。安全のために検証候補を狭めすぎず、売買推奨ではなく backtest / user review 前提であることを UI / docs で明示する。
- provider output の schema / type / format / candidate count failure は `PROVIDER_INVALID_RESPONSE` として proposal API 内に閉じる。StrategyLab の既存 Pine generation / save / validation flow へ影響させない。

理由:

- provider を増やす前に、LLM output が UI や Pine generation input を壊さない境界を固定するため。
- stub provider でも同じ validation path を通すことで、後続 provider 実装時の責務を明確にするため。

見送り:

- `local_llm` / `openai_api` provider 実装。
- request-time provider selection。
- provider timeout / fallback 実装。
- Web search / deep research。
- proposal history / DB 保存。
- proposal から Pine generation / StrategyVersion 保存への自動連鎖。

## 16. LLM strategy proposal local_llm provider design PR 1

判断:

- local_llm provider implementation の PR 1 は docs-only で扱い、実装コード、API shape、DB、Prisma schema、test は変更しない。
- proposal 専用 provider selector は `STRATEGY_PROPOSAL_PROVIDER=stub|local_llm` とし、未指定 default は `stub` とする。
- local_llm provider は StrategyLab の一時 proposal candidates generation のみに責務を限定する。
- local_llm endpoint / model / timeout / max output は proposal 専用設定で分離する。実値、raw prompt、raw response、provider endpoint、stack trace、credential、local path は UI / docs / PR / response に出さない。
- local_llm output は既存 `strategy_proposal_candidates` schema に合わせ、UI に出す前に route 層の request validation と provider response validation を必ず通す。
- 初回 local_llm 実装では silent stub fallback を行わず、timeout、provider unavailable、malformed JSON、schema / type / format 不正、必須項目欠落、candidate count 不正は provider error とする。
- fallback option は後続候補として残す。実装する場合は opt-in とし、provider metadata で fallback を明示する。

理由:

- deterministic stub から non-stub provider に進む前に、設定境界、失敗時挙動、fallback の有無、raw provider 情報の非表示を固定するため。
- local_llm は環境依存の unavailable / timeout / malformed response が起きやすく、silent fallback すると品質確認と疎通確認が混ざるため。
- proposal は投資助言ではなく検証候補であり、provider を変えても保存や Pine generation へ自動連鎖させない境界を維持するため。

見送り:

- 実装コード変更。
- `openai_api` provider。
- Web search / deep research。
- request-time provider selection。
- proposal history / DB 保存。
- auto Pine generation / auto save。
- DB / Prisma schema change。

## 17. LLM strategy proposal local_llm provider implementation PR 2

判断:

- `STRATEGY_PROPOSAL_PROVIDER=stub|local_llm` による env provider selection を実装する。未指定 default は `stub` のまま維持する。
- local_llm provider は StrategyLab の一時 proposal candidates generation のみに責務を限定する。
- local_llm provider は JSON response を要求し、既存 `strategy_proposal_candidates` schema に合わせて UI に返す前に provider response validation を通す。
- malformed JSON、schema / type / format 不正、必須項目欠落、candidate count 不正、Web search 未実装時の web research basis、provider unavailable、timeout は generic provider failure として扱う。
- user_hint / provider output の投資助言風 wording は wording だけでは reject しない方針を維持する。
- silent stub fallback は行わない。

理由:

- deterministic stub の既存挙動を default として維持しながら、明示 opt-in で local_llm provider を検証できるようにするため。
- local_llm output は malformed / schema mismatch が起き得るため、既存 validation pipeline を必須にして UI 破綻と raw provider 情報露出を避けるため。

見送り:

- `openai_api` provider。
- Web search / deep research。
- request-time provider selection。
- proposal history / DB 保存。
- auto Pine generation / auto save。
- DB / Prisma schema change。
