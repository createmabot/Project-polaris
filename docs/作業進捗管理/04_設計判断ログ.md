更新日: 2026-05-17

## 1. 目的

本資料は、横断的な設計判断を短く追えるようにする decision log である。詳細仕様は各仕様書と機能別正本 docs に残し、本資料では判断内容、理由、見送り範囲を記録する。

## 2. docs architecture

判断:

- `docs/仕様書/`、`docs/運用ドキュメント/`、`docs/作業進捗管理/` の 3 分類を現行 docs architecture とする。
- 既存番号 docs は削除・rename しない。
- 新体系から旧 docs へリンクし、旧 docs は履歴資料または詳細正本として扱う。

理由:

- 既存 PR / issue / docs link を壊さないため。
- 大量移動より、読む順番と正本分類を明確にする方が安全なため。

見送り:

- archive directory への一括移動。
- 番号 docs の bulk rename。

## 6. AI summary auto enqueue

判断:

- CSV import parsed report 作成直後、application 起点 CSV import parsed report 作成直後、新規 internal backtest report conversion 完了直後だけを auto enqueue 対象にする。
- BacktestDetail 表示や ApplicationDetail report history 表示を契機に enqueue しない。
- duplicate guard は同一 input snapshot hash の succeeded summary、queued / running job、failed job を対象にする。
- failed job auto retry は行わず、既存 manual generation / regeneration を維持する。
- phase 2 では `GET /api/backtests/:backtestId` に `latest_ai_summary_job` を optional read-only field として追加し、BacktestDetail で最新 job status を表示する。
- `latest_ai_summary_job` は safe な status metadata に限定し、raw request / response payload、provider endpoint、secret、local path、stack trace は返さない。

理由:

- page view 起点では provider cost / latency / duplicate enqueue の制御が難しいため。
- report 作成直後なら input が固定され、低頻度で扱えるため。
- status visibility は provider を呼ばず、既存 `ai_jobs` を読むだけで failed / running の誤読を減らせるため。

見送り:

- scheduled batch。
- automatic retry の本格化。
- polling / live status update。
- AI summary 同士の比較。
- ApplicationDetail row への job status 表示。

## 15. LLM strategy proposal provider boundary 実装

判断:

- deterministic stub provider を provider boundary 配下に分離し、route 層で request validation と response validation を行う。
- 現時点の provider selection は `stub` 固定とし、`local_llm` / `openai_api` は実装しない。
- response validation は enum、candidate count、必須項目、型、format、Web research basis を検査する。投資助言風 wording だけを理由に provider invalid とはしない。0 candidates は既存 UI の EmptyState として表現できるため許容する。
- user input 由来の投資助言風 wording は request validation の `VALIDATION_ERROR` にしない。安全のために検証候補を狭めすぎず、売買推奨ではなく backtest / user review 前提であることを UI / docs で明示する。
- provider output の schema / type / format / candidate count failure は `PROVIDER_INVALID_RESPONSE` として proposal API 内に閉じる。StrategyLab の既存 Pine generation / save / validation flow へ影響させない。

理由:

- provider を増やす前に、LLM output が UI や Pine generation input を壊さない境界を固定するため。
- stub provider でも同じ validation path を通すことで、後続 provider 実装時の責務を明確にするため。

見送り:

- `local_llm` / `openai_api` provider 実装。
- request-time provider selection。
- provider timeout / fallback 実装。
- Web search / deep research。
- proposal history / DB 保存。
- proposal から Pine generation / StrategyVersion 保存への自動連鎖。
