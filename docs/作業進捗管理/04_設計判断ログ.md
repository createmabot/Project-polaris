# 北極星 設計判断ログ

更新日: 2026-05-21
分類: 作業進捗管理

## 1. 目的

本資料は、横断的な設計判断を短く追えるようにする decision log である。詳細仕様は各仕様書と機能別正本 docs に残し、本資料では判断内容、理由、見送り範囲を記録する。

## 2. docs architecture

判断:

- `docs/仕様書/`、`docs/運用ドキュメント/`、`docs/作業進捗管理/` の 3 分類を現行 docs architecture とする。
- 既存番号 docs は削除・rename しない。
- 新体系から旧 docs へリンクし、旧 docs は履歴資料または詳細正本として扱う。

理由:

- 既存 PR / issue / docs link を壊さないため。
- 大量移動より、読む順番と正本分類を明確にする方が安全なため。

見送り:

- archive directory への一括移動。
- 番号 docs の bulk rename。

## 4. Symbol Strategy Application

判断:

- Symbol 起点で strategy version を適用し、application / run / report の関係で履歴を管理する。
- SymbolDetail は overview と入口、ApplicationDetail は application-scoped run / report history を担当する。
- SymbolDetail は銘柄起点 strategy workflow の入口も担う。`この銘柄でストラテジー提案` は StrategyLab への context handoff、`改善版を作る` は StrategyVersionDetail の clone / improvement flow への入口として扱う。
- StrategyLab / StrategyVersionDetail からの apply は既存 symbol strategy application 作成 API を使う明示操作に限定する。
- 改善版適用時も旧 application は自動 archive せず、run / report history と比較余地を残す。
- SymbolDetail 起点の Codex CLI manual strategy proposal は、通常の汎用 proposal ではなく銘柄調査付き prompt として扱う。symbol context がある場合は Web検索必須の guidance を使い、Codex CLI 側で銘柄の基本情報、価格傾向、出来高、volatility、決算・業績、news、events、sector 環境を確認してから候補を作る前提にする。
- `local_llm` / `stub` は従来どおり symbol_code / market / timeframe を使う軽い context proposal とし、Web検索や最新情報確認を要求しない。

理由:

- 銘柄を起点にした検証履歴を辿りやすくするため。
- latest summary と any-run history の責務を分離するため。
- 実運用では「この銘柄で考える」「この銘柄に適用した戦略を改善する」操作が SymbolDetail 起点になるため。
- DB lineage を急いで追加せず、既存 application / clone / apply API を使った context handoff から始める方が scope と rollback risk を抑えられるため。
- 銘柄固有の最新情報は local_llm では扱いにくく、Codex CLI manual flow ならユーザー操作による Web検索確認を前提にできるため。
- ただし北極星 backend が Web検索や Codex CLI を自動実行すると cost / latency / citation / freshness / raw output 保存の境界が大きく変わるため、初回は prompt guidance と manual JSON import に限定する。

見送り:

- hard delete。
- richer metadata table の即時追加。
- improvement lineage / proposal-to-version relation の即時追加。
- 改善版適用時に旧 application を transaction で archive する replace flow。
- `source_type=web` の解禁、URL / citation 保存、Web research freshness metadata。
- backend job としての Web search / deep research / Codex CLI 自動実行。

- Stage 2C は DB / Prisma / tests cleanup として、internal tables / columns / relations、service / queue / worker / tests、legacy docs の削除を data audit 後に判断する。

- internal report conversion endpoint は Stage 2B で閉じるため、conversion 完了直後の AI summary auto enqueue は新規には使われない。generic Backtest AI summary と既存 summary 表示は維持する。

- 404 unregister より 410 Gone の方が、意図的に閉じた API であることを client / test / operation に示しやすいため。

見送り:

## 15. LLM strategy proposal provider boundary 実装

判断:

- deterministic stub provider を provider boundary 配下に分離し、route 層で request validation と response validation を行う。
- 現時点の provider selection は `stub` 固定とし、`local_llm` / `openai_api` は実装しない。
- response validation は enum、candidate count、必須項目、型、format、Web research basis を検査する。投資助言風 wording だけを理由に provider invalid とはしない。0 candidates は既存 UI の EmptyState として表現できるため許容する。
- user input 由来の投資助言風 wording は request validation の `VALIDATION_ERROR` にしない。安全のために検証候補を狭めすぎず、売買推奨ではなく backtest / user review 前提であることを UI / docs で明示する。

理由:

- provider を増やす前に、LLM output が UI や Pine generation input を壊さない境界を固定するため。
- stub provider でも同じ validation path を通すことで、後続 provider 実装時の責務を明確にするため。

見送り:

- `local_llm` / `openai_api` provider 実装。
- request-time provider selection。
- provider timeout / fallback 実装。
- Web search / deep research。
- proposal history / DB 保存。

## 16. LLM strategy proposal local_llm provider design PR 1

判断:

- local_llm provider implementation の PR 1 は docs-only で扱い、実装コード、API shape、DB、Prisma schema、test は変更しない。
- proposal 専用 provider selector は `STRATEGY_PROPOSAL_PROVIDER=stub|local_llm` とし、未指定 default は `stub` とする。
- local_llm provider は StrategyLab の一時 proposal candidates generation のみに責務を限定する。
- local_llm endpoint / model / timeout / max output は proposal 専用設定で分離する。実値、raw prompt、raw response、provider endpoint、stack trace、credential、local path は UI / docs / PR / response に出さない。
- local_llm output は既存 `strategy_proposal_candidates` schema に合わせ、UI に出す前に route 層の request validation と provider response validation を必ず通す。
- 初回 local_llm 実装では silent stub fallback を行わず、timeout、provider unavailable、malformed JSON、schema / type / format 不正、必須項目欠落、candidate count 不正は provider error とする。
- fallback option は後続候補として残す。実装する場合は opt-in とし、provider metadata で fallback を明示する。

理由:

- rate guard key は user id があれば user、trusted forwarded IP opt-in 時のみ forwarded client IP、通常は request IP、取得不能時は unknown bucket とする。forwarded header は default では信頼せず、opt-in 時も先頭値が valid IPv4 / IPv6 として検証できる場合だけ使う。
- StrategyLab は 429 を短い再試行案内として表示し、provider endpoint / model 実値 / raw diagnostics を表示しない。
- rate guard は accidental load と連打抑止が目的であり、multi-process / distributed production の厳密な abuse prevention ではない。
- silent stub fallback は導入しない。fallback が必要な場合は explicit opt-in と metadata 表示を別設計する。

理由:

- 有料 provider や Web search を増やす前に、同期 proposal API の費用・遅延・連打・失敗境界を固定する必要があるため。
- 個人利用前提では、初回から DB / billing / distributed rate limit を導入するより、env guard と per-process rate guard で事故防止を先に固める方が適切なため。
- reverse proxy / tunnel 配下で `request.ip` が proxy 側に寄る可能性がある一方、client-supplied forwarded header をそのまま信頼すると rate guard 回避につながるため、opt-in、trust boundary 明確化、IP validation を条件に限定する必要があるため。
- rate limit / timeout / max output / retry を backend boundary に置くことで、UI や benchmark 側が raw diagnostics を扱わずに済むため。

見送り:

- per-user billing system。
- distributed rate limit / hard daily cost cap。
- `openai_api` provider。
- Web search / deep research。
- request-time provider selection。
- provider event log persistence。
- benchmark result DB table。
- proposal から Pine generation / save / backtest / AI summary への自動連鎖。

- session 作成は明示操作で行い、既存 active session は再利用する。session 作成だけでは clone、rewrite、Pine generation、backtest、AI summary、application apply を起動しない。
- candidate からの version 作成は明示操作に限定し、既存 StrategyVersion clone と PineScript lineage copy を利用する。作成済み candidate は同じ version を返し、重複 clone を増やさない。
- StrategyVersionDetail の rewrite draft は `refinement_candidate_id` を optional context として受け取れるが、rewrite 結果は textarea draft のみで自動保存しない。
- StrategyOptimizationSessionDetail は比較 board として、source metrics、candidate latest report、diff を read-only に表示する。winner 自動判定、投資助言、売買推奨、raw data 表示は行わない。

## 2026-06-07 Strategy Optimization Session detail URL context follow-up

- candidate create-version の `detail_url` は、source backtest から SymbolStrategyApplication context を解決できる場合、symbol / application / source version / source backtest / refinement candidate / return path を含む full improvement context URL とする。
- これにより candidate 経由の StrategyVersionDetail でも `mode=improve_application` が成立し、検証結果からの改善メモと LLM rewrite 導線が表示される。
- application context が解決できない report では fallback URL を返し、API は落とさない。表示だけで rewrite、save、Pine generation、backtest、apply は起動しない。

## 2026-06-07 Normalized Strategy Spec v1 foundation

- `StrategyRuleVersion.normalizedRuleJson` は、自然言語ルールから生成した `normalized_strategy_spec` v1 の保存先として再定義する。新 table / migration は追加しない。
- `normalized_strategy_spec` v1 の生成は deterministic parser only ではなく、LLM-first + deterministic validator / sanitizer / fallback とする。entry / exit / risk / filters の文脈分離と数値閾値保持は LLM structured extraction を主経路に任せ、invalid output や provider unavailable 時は deterministic parser の暫定 spec に fallback する。spec 生成は明示操作のみで、Pine generation、backtest、AI summary、optimization、application apply は起動しない。
- 初回は deterministic parser で、market `JP_STOCK|US_STOCK`、timeframe `D`、side `long_only`、SMA / EMA / RSI / MACD / ATR / volume average、entry / exit / risk / filters / time exit を抽出する。
- unsupported condition は generation failure にせず、`validation.unsupported_features` と warnings に残す。`validation.supported_for_internal_backtest=false` として、internal backtest engine readiness とは分離する。
- spec 生成はユーザーの明示操作だけで実行し、Pine generation、backtest、AI summary、optimization、application apply は起動しない。raw natural language rule、raw Pine、raw prompt、provider response、endpoint、model 実値、secret、token、local path、stack trace は spec API response に出さない。

## 2026-06-07 Internal Backtest Engine v1

- Internal Backtest Engine v1 は、保存済み `normalized_strategy_spec` v1 と `MarketPriceBar` の `D` bars を使う deterministic MVP として再導入する。
- 実行入口は StrategyVersionDetail の明示操作と `POST /api/strategy-versions/:versionId/internal-backtests` に限定する。SymbolDetail / ApplicationDetail 起点の internal start / report conversion endpoint、execution table、artifact table、worker / queue は復活させない。
- 初回は long-only、single position、next open fill、stop-first same-bar handling、commission / slippage 0、100% equity allocation とする。unsupported feature は warnings / ignored list に残し、hard unsupported は validation error にする。
- 成功時は `Backtest` を `execution_source=internal_backtest` として作成し、`strategySnapshotJson.result_summary` に metrics / trades / equity curve / assumptions / warnings を保存する。`BacktestImport` は作成しない。
- 表示や spec 生成だけで internal backtest、Pine generation、AI summary、optimization、application apply は起動しない。raw CSV、raw import text、raw Pine、raw prompt、provider response、endpoint、model、secret、token、local path、stack trace は response / snapshot に出さない。
