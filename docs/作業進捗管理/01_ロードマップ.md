## 3. 現在の優先軸

現時点では、機能拡張よりも既存導線の読みやすさ、運用性、docs正本化を優先する。

優先するもの:

- 既存主要導線を壊さない小さな改善。
- API / backend / DB / Prisma schema を変えない read-only UX 改善。
- docs の正本分類と参照順の明確化。
- AI summary / artifact / report comparison の運用境界整理。
- Visual regression は optional pilot に限定し、required check 化や対象拡大は後続判断に留める。
- PR #337〜#340 で完了した4領域は、既存導線を壊さない read-only / optional / boundary hardening の範囲として扱う。

## 4. 次フェーズ候補

| 優先度 | 候補 | 位置づけ |
|---|---|---|
| 1 | Release / operations stabilization | required checks、docs-only release acceptance、manual walkthrough、provider failure 時の運用を先に安定化する。 |
| 2 | AI quality / cost operations | auto enqueue 後の cost cap、rate limit、provider opt-in、failure analysis、retry 方針を整理する。 |
| 3 | LLM strategy proposal provider operations / openai_api expansion | provider quality trend aggregation 完了後に、StrategyVersion created-from-proposal relation、history management、benchmark result recording workflow、openai_api provider、sanitized provider event log persistence、request-time provider selection、cost / rate guard を段階判断する。 |
| 4 | Artifact operations phase 2 | download / signed URL / file token、retention、diff は権限境界を固定してから判断する。 |
| 5 | Report comparison phase 3 | read-only helper の利用実績を見て、comparison entity、metrics normalization、自動比較生成の要否を判断する。 |
| 6 | Performance / scale | 大量データ時の index / read model / cache を、実利用のボトルネックに合わせて絞る。 |

継続候補:

- Visual regression expansion は optional pilot の実行負荷と snapshot churn を確認し、本導入可否を再判断する。
- TradingView / Pine workflow 強化は現行導線の運用観測後に優先度を再評価する。
- LLM strategy proposal の StrategyVersion created-from-proposal relation、proposal history filter / pagination / search / retention / full management、benchmark result recording workflow、provider quality trend の range / filter / materialized aggregation、openai_api provider、Web search / deep research、provider cost / rate guard、symbol context 導線、optional fallback metadata は、optional benchmark と history / trend UI の運用観測後に判断する。auto Pine generation / auto save は引き続き対象外とする。
- 旧 docs の注意書き追加、API sample docs と現行 routes / tests の対応、walkthrough 分割は release stabilization の一部として小さく進める。
- StrategyVersionDetail、StrategyLab、SymbolDetail filter の部分的な UI component 適用は、主要導線を壊さない範囲で判断する。

## 5. ロードマップ更新ルール

- 完了扱いにする場合は、完了範囲と未対応範囲を同時に書く。
- 実装を伴う判断は、仕様書または機能別正本 docs にも反映する。
- roadmap には詳細仕様を重複させず、読むべき正本へのリンクを残す。
- 大量 rename / archive 移動は別 PR でリンク影響を確認してから扱う。
