# 北極星 ロードマップ

更新日: 2026-05-13
分類: 作業進捗管理

## 1. 目的

本資料は、北極星の作業ロードマップを現在の docs architecture に合わせて読むための入口である。詳細な積み上げ履歴は `docs/39.北極星 MVP後ロードマップ・バックログ整理.md`、P2 / P3 判断は `docs/44` と `docs/53` を参照する。

## 2. フェーズの大枠

| フェーズ | 状態 | 主な内容 |
|---|---|---|
| MVP | 完了 | Rule Lab、Pine 生成、TradingView CSV import、Backtest Detail、AI summary、comparison の基本導線 |
| MVP後 P1 | 完了 | 運用品質、docs文字化け範囲整理、Rule Lab 日本語入力再確認など |
| MVP後 P2 | 完了 | snapshot stale / market status、CSV import 日本語ヘッダー、references diagnostics など |
| P3 | 完了 | browser-based E2E PoC、Tailwind / IA 土台、Symbol Strategy Application、internal backtest report 化、UI cleanup |
| P3後続 | 進行中 | Application Detail / History、report comparison UX、AI summary / artifact operations、docs architecture refactor |

## 3. 現在の優先軸

現時点では、機能拡張よりも既存導線の読みやすさ、運用性、docs正本化を優先する。

優先するもの:

- 既存主要導線を壊さない小さな改善。
- API / backend / DB / Prisma schema を変えない read-only UX 改善。
- docs の正本分類と参照順の明確化。
- AI summary / artifact / report comparison の運用境界整理。
- Visual regression など重い仕組みは pilot 条件が固まるまで readiness に留める。

## 4. 次フェーズ候補

- 旧 docs の注意書き追加範囲を拡張するか判断する。
- API sample docs と現行 routes / tests の対応を整理する。
- walkthrough を確認観点別に分割するか判断する。
- Visual regression pilot を 1 から 3 個の stable container に限定して再判断する。
- AI summary auto enqueue の status 表示、batch / scheduled job、cost control を判断する。
- StrategyVersionDetail、StrategyLab、SymbolDetail filter の部分的な UI component 適用を判断する。

## 5. ロードマップ更新ルール

- 完了扱いにする場合は、完了範囲と未対応範囲を同時に書く。
- 実装を伴う判断は、仕様書または機能別正本 docs にも反映する。
- roadmap には詳細仕様を重複させず、読むべき正本へのリンクを残す。
- 大量 rename / archive 移動は別 PR でリンク影響を確認してから扱う。
