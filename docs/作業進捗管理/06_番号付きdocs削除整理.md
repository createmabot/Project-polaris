# 北極星 番号付き docs 削除整理

更新日: 2026-05-14
分類: 作業進捗管理

## 1. 目的

本資料は、MVP初期から残っている平置き番号付き docs の削除 / 保留 / 移管判断を記録するための作業台帳である。PR 1 では実削除を行わず、削除前確認 checklist、移管先判断、候補分類、保留理由を固定した。PR 2 では data / API / sample 系のうち、移管済みでリンク影響が小さい docs を削除し、保留対象を記録する。

現行仕様判断では `docs/仕様書/`、運用判断では `docs/運用ドキュメント/`、進捗判断では `docs/作業進捗管理/` を優先する。

## 2. PR 1 の判断

- 番号付き docs は削除しない。
- 番号付き docs の rename / archive 移動もしない。
- 削除候補は分類だけ行う。
- 削除前に、内容が現行 docs へ移管済みか、履歴資料として残す価値があるかを確認する。
- 実削除は PR 2 以降で、分類単位に小さく分けて行う。

## 3. 削除前確認 checklist

各 docs を削除候補にする前に、次を確認する。

- 現行仕様、運用手順、作業進捗のどれに相当する内容か分類した。
- 移管先 docs が存在し、必要な内容がそこに要約または反映済みである。
- 現行仕様と矛盾する古い前提が、削除ではなく注意書きで残すべき履歴価値を持たないか確認した。
- README、docs/0、docs/57、walkthrough、仕様書、運用ドキュメント、作業進捗管理から参照されていないか確認した。
- 実装判断や障害調査で頻出する背景情報が失われないか確認した。
- 削除対象が PR / issue / commit message から参照されていても、現行 docs から代替参照へ辿れることを確認した。
- 削除理由、移管先、保留理由を本資料に記録した。
- docs-only 確認として `git diff --check` と secret / local path scan を実施する。

## 4. 移管先判断

| 移管先 | 対象 | 判断基準 |
|---|---|---|
| `docs/仕様書/` | 現行システム仕様、画面仕様、データ/API/AI/UI仕様 | 現在の実装判断に使う内容。PR履歴や作業経緯は入れない。 |
| `docs/運用ドキュメント/` | 起動、確認、CI、CSV、webhook、AI provider、トラブルシュート | 手順として今後も実行する内容。履歴ではなく再現可能な運用に整える。 |
| `docs/作業進捗管理/` | roadmap、backlog、完了整理、decision log、cleanup判断 | 経緯・判断・残課題として残す内容。仕様本文には入れない。 |
| 残置 | 背景資料、履歴資料、外部連携の参照資料 | 移管すると文脈が欠ける、または履歴資料として読む価値がある内容。 |

## 5. 削除してよい条件

- 内容が現行 docs に移管済みで、削除しても現行仕様・運用・進捗判断に影響しない。
- 古い前提が中心で、注意書き付きで残すより削除した方が誤読リスクを下げられる。
- 参照リンクの代替先が明確で、docs/0 / docs/57 から辿れる。
- 削除理由と移管先を本資料に記録済みである。
- 削除 PR が小さく、review で内容確認できる粒度である。

## 6. 削除しない条件

- 現行仕様の正本または詳細正本として扱っている。
- 運用手順としてまだ使う可能性がある。
- 完了判断、意思決定、受け入れ確認、障害対応の根拠として残す価値がある。
- 移管先が未作成、または移管内容が不十分である。
- 削除により既存リンクが多数切れ、代替導線を用意していない。
- 旧前提ではあるが、注意書きを付けて履歴資料として残す方が安全である。

## 7. PR 分割方針

- PR 1: 本資料を追加し、削除判断 checklist と候補分類を固定する。実削除はしない。
- PR 2: data / API / sample 系の候補を確認し、移管済みなら削除または保留理由を記録する。
- PR 3: screen / UI / task 系の候補を確認し、移管済みなら削除または保留理由を記録する。
- PR 4: progress 系の候補を確認し、現行 `docs/作業進捗管理/` へ集約できるものだけ削除する。
- PR 5 以降: 運用補助、webhook、snapshot、walkthrough 分割など、影響範囲が異なるものを必要に応じて扱う。
