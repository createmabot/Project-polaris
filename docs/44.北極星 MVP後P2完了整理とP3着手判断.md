| 候補 | 目的 | 効果 | コスト感 | 今すぐ着手するか |

| browser-based E2E導入 | 実画面回帰の自動化 | 高い | 高い | まだ早い |

- いきなり大きなUI拡張に入るより、MVP実運用品質の確認を続ける方が安全である。
- 特に `disclosure / earnings` と TradingView 実送信は、既存MVP導線の品質補強として直接価値がある。
- browser-based E2E は有用だが、導入コストがやや高く、P2で残った live 確認の後でよい。

## 6. 次に進める推奨候補

推奨順:

1. `Home` の最小移行、または `SideRail` PoC の優先度判断
2. browser-based E2E の対象拡張判断
3. transaction履歴編集UI
4. 複数ポートフォリオUI
5. 非日本銘柄の本格対応

理由:

- disclosure / earnings live データ追加確認 は 2026-05-08 時点で一次確認を完了した。
- references あり銘柄での TradingView 実送信自体は 2026-05-08 に `2148` で 1 件確認した。
- その後、TradingView payload互換の疑似送信で `disclosure / earnings` direct type が alert summary context / 本文へ入ることを確認した。
- `9929` の保存導線確認は完了した。
- direct type 実送信再確認は、alert 用 lookback が `disclosure=3日`, `earnings=7日` であることと、TradingView 実送信では過去 `triggered_at` を固定できないことから、現時点では実施しない判断に整理した。
- browser-based E2E の local-only PoC 自体は導入済みだが、Tailwind 化前に spec を増やしすぎると UI 改修コストに対して test 修正コストが先に膨らむ。
- `docs/46` に従う Tailwind 設定追加と最小 global CSS 整理は完了した。
- `docs/47` に基づく画面導線 / IA / ナビゲーション方針整理も完了した。
- `AppLayout` / `PageHeader` / `Navigation` / `TextLink` の最小土台を `Home` / `SymbolDetail` にだけ適用済みである。
- `Home` の本格移行と `SideRail` PoC はまだ未着手であり、どちらを先に進めるかを次の判断点に置く。
- `SectionCard` など画面内部 component は、導線変更後の構造に合わせて抽出する。
- docs/29 / docs/30 の個別可読性確認は完了し、`docs/29` は可読、`docs/30` は全文復元せず後続 docs と実装を優先参照する方針を固定した。
- `docs/45` で browser-based E2E の方針整理を完了し、そのまま Playwright local-only の初回 PoC を `Home → SymbolDetail` で導入した。
- 次に見るべき論点は CI ではなく、画面導線 / IA / ナビゲーション方針を固め、その後に PoC 対象を Rule Lab / Backtest へ広げる価値があるかを判断することである。
- browser-based E2E 以降は価値は高いが、導入規模や仕様整理コストが上がる。
