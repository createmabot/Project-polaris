# 北極星 API / ユースケース単位の入出力設計（MVP）

更新日: 2026-03-08

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVP開発に向けて、  
**ユースケース単位で必要となる API の入出力** を整理するものである。

前提として、北極星は以下の役割分担で構成される。

- TradingView: 表示、監視、一次検証
    
- 北極星: 要約、比較、仮説、履歴管理
    

したがって、本ドキュメントでは **北極星側で実装すべき API** を中心に扱う。

---

## 2. 設計方針

### 2.1 設計の基本姿勢

- API は **画面単位** ではなく **ユースケース単位** で切る
    
- MVPでは BFF 的に画面が使いやすいレスポンスを優先する
    
- TradingView の widget 表示データは北極星 API で過度に中継しない
    
- AI生成物は非同期前提とし、状態を返せるようにする
    
- CSV取込や webhook 受信などの外部連携APIは、内部画面APIと責務を分ける
    

### 2.2 API の大分類

MVPでは、APIを以下の9カテゴリに分ける。

1. 認証・ユーザー基盤
    
2. ホーム / 市場概況
    
3. 銘柄研究
    
4. 研究ノート
    
5. 比較
    
6. アラート受信 / イベント処理
    
7. AI要約
    
8. ルール検証
    
9. CSV取込
    

### 2.3 レスポンスの共通方針

レスポンスは基本的に以下の形式を採用する。

```json
{
  "data": {},
  "meta": {},
  "error": null
}
```

エラー時は以下を基本形とする。

```json
{
  "data": null,
  "meta": {
    "request_id": "req_xxx"
  },
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "timeframe is required"
  }
}
```

## 3. ユースケース一覧

MVPで扱う主要ユースケースは以下の通り。

### 3.1 毎日5分で状況把握

- ホーム画面を開く
    
- 監視銘柄一覧を取得する
    
- 発火アラート一覧を取得する
    
- 今日動いた理由のAI要約を取得する
    
- 朝サマリ / 夜サマリを取得する
    

### 3.2 銘柄研究

- 銘柄詳細を開く
    
- AI論点カードを取得する
    
- 関連ニュース/開示の参照情報を取得する
    
- 仮説ノートを取得 / 更新する
    

### 3.3 比較

- 比較セッションを作成する
    
- 比較対象銘柄を設定する
    
- 比較結果を取得する
    
- AI比較総評を生成 / 取得する
    

### 3.4 ルール検証

- 自然言語ルールを登録する
    
- Pine Script を生成する
    
- 検証実行情報を登録する
    
- TradingViewから出力したCSVを取込む
    
- 検証結果を取得する
    
- 検証履歴を参照する
    

### 3.5 イベント処理

- TradingView webhook を受信する
    
- 受信したアラートをイベントとして保存する
    
- AI要約生成ジョブを起動する

### 4.10.5 engine_estimated の実データ接続契約（最小実装）

`summary_kind = engine_estimated` は、`scaffold_deterministic` より一段現実寄りの推定段階として扱う。
ただし本節更新時点では本格シミュレーション（`engine_actual`）ではない。

- 最小対象データ: 日足 OHLCV
- 最小対象 market/timeframe: `JP_STOCK` / `D`
- 取得責務: data source adapter に限定（route/worker は取得責務を持たない）
- provider: `daily_ohlcv / JP_STOCK / D` の最小 real provider を実装し、テスト実行時は deterministic stub を利用する
- 取得失敗時: execution を `failed` とし、`errorCode = DATA_SOURCE_UNAVAILABLE` を返却

`POST /api/internal-backtests/executions` は `strategy_rule_version_id`（実行ルール識別）と
`execution_target.symbol`（実データ取得識別）を分離して扱う。
現在の最小固定条件は以下。

- `engine_config.summary_mode = "engine_estimated"`
- `execution_target.symbol` canonical（`market = JP_STOCK`）は `7203` を正とし、`TYO:7203` は受理後に `7203` へ正規化する
- `execution_target.symbol` canonical（`market = JP_STOCK`）は前後空白を trim し、4桁数字以外は `INVALID_EXECUTION_TARGET`
- `execution_target.symbol`（`engine_estimated` では必須）

`GET /api/internal-backtests/executions/:executionId/result` では、再現性情報として `input_snapshot` または同等責務に
以下の `data_source_snapshot` を保持する前提を固定する。

- `source_kind`
- `market`
- `timeframe`
- `from`
- `to`
- `fetched_at`
- `data_revision`
- `bar_count`

`engine_estimated` の `result_summary.metrics` は、上記で取得した normalized bars から算出した価格要約を返す（`bar_count`, `first_close`, `last_close`, `price_change`, `price_change_percent`, `period_high`, `period_low`, `range_percent`）。
これらは売買シミュレーションの損益指標ではなく、`engine_actual` 導入前の再現可能な要約値として扱う。
empty bars（`bar_count = 0`）は `engine_estimated` の失敗条件ではなく、`succeeded` のまま 0件要約を返す契約とする。

### 4.10.6 result consumer の表示・解釈ルール（固定）

consumer 側は、以下を判定軸として表示を分岐する。

- 実行成否の正本: `GET /api/internal-backtests/executions/:executionId` の `status`
- 失敗理由の判定: `error_code`
- 0件成功判定: `status = succeeded` かつ `result_summary.summary_kind = engine_estimated` かつ `result_summary.metrics.bar_count = 0`
- 再現性確認: `input_snapshot.data_source_snapshot.bar_count`

表示パターンは以下で固定する。

- 通常成功: `status = succeeded` かつ `bar_count > 0`。metrics を通常表示する
- データ0件成功: `status = succeeded` かつ `bar_count = 0`。失敗文言にせず「対象期間のデータがありません」系で表示する
- 取得失敗: `status = failed` かつ `error_code = DATA_SOURCE_UNAVAILABLE`。「データ取得に失敗しました」系で表示する
- その他失敗: `status = failed` かつ上記以外。「内部エラーが発生しました」系で表示する

実装上の判定カテゴリ（共有 helper）は以下を正とする。

- `success_with_data`
- `success_no_data`
- `data_source_unavailable`
- `internal_failure`
- `not_ready`

`summary_kind = scaffold_deterministic` は、現時点では `success_with_data` と同カテゴリで扱う（将来分岐余地は残す）。

表示層向けの最小 view-model helper は、上記カテゴリを以下へ変換する。

- `state_label`
- `is_error`
- `is_empty`
- `can_show_metrics`
- `recommended_message_key`

対応（固定）:

- `success_with_data` → `state_label=success`, `can_show_metrics=true`
- `success_no_data` → `state_label=success_no_data`, `is_empty=true`, `can_show_metrics=false`, `recommended_message_key=internal_backtest.result.success_no_data`
- `data_source_unavailable` → `state_label=error_data_source_unavailable`, `is_error=true`, `recommended_message_key=internal_backtest.result.data_source_unavailable`
- `internal_failure` → `state_label=error_internal`, `is_error=true`, `recommended_message_key=internal_backtest.result.internal_failure`
- `not_ready` → `state_label=not_ready`, `recommended_message_key=internal_backtest.result.not_ready`

`GET /api/internal-backtests/executions/:executionId/result` は `succeeded` 以外では `RESULT_NOT_READY` を返すため、
失敗系表示の一次判定は status API 側で行う。

### 4.10.7 engine_actual 最小契約（最小実装済み）

本節は `engine_actual` の初回スライスを実装反映した契約である。

- 対象スライス（初回）:
  - `market = JP_STOCK`
  - `timeframe = D`
  - 単一銘柄（`execution_target.symbol`）
  - long-only / 1ポジション
- API 追加は行わず、既存の `POST /api/internal-backtests/executions` / `GET /api/internal-backtests/executions/:executionId` / `GET /api/internal-backtests/executions/:executionId/result` を利用する
- `engine_config.summary_mode = "engine_actual"` で最小実装経路へ入る

`engine_actual` の最小 result_summary（実装済み）:

- `summary_kind = engine_actual`
- 最小 metrics:
  - `trade_count`
  - `win_rate`
  - `total_return_percent`
  - `max_drawdown_percent`
  - `holding_period_avg_bars`
  - `first_trade_at`
  - `last_trade_at`

`engine_actual` の最小 artifact（実装済み）:

- trade list / equity curve を後続で参照可能な pointer を返却
- 参照は既存 `artifact_pointer`（`type` / `execution_id` / `path`）で返却
- 初回実装では `path = /internal-backtests/executions/:executionId/artifacts/engine_actual/trades-and-equity`

実装前提（初回実装）:

- シグナル評価: 当日終値確定後
- 約定: 翌営業日始値
- 手数料/スリッページ: `engine_config` 指定値（未指定時 0）
- empty bars/欠損データ: fail へ自動昇格させず no-trade 結果を返す
- no-trade（シグナルなし/約定なし）も `succeeded` とし、`trade_count = 0` 系 summary を返す

非対象（初回）:

- short
- 複数ポジション
- intraday
- 複数銘柄同時実行
- portfolio/corporate action 高度対応

## 6. 入力バリデーション方針

## 6.1 共通

- ID系は必須
    
- 日付は ISO 8601 ベース
    
- timeframe は列挙値制限
    
- 数値項目は型変換せず厳格に扱う
    

## 6.2 研究ノート

- symbol_id 必須
    
- title 必須
    
- thesis_text は空でも可だが、少なくとも1つの仮説系項目入力を推奨
    

## 6.3 比較

- symbol_ids は最低2件必要
    
- 同一銘柄の重複は不可
    

## 6.4 ルール検証

- natural_language_spec 必須
    
- target_market 必須
    
- timeframe 必須
    
- backtest_period_from / to は妥当な順序であること
    

## 6.5 CSV取込

- file_type 必須
    
- 対応拡張子は csv のみ
    
- 文字コードエラー時は parse_status=failed で保持

## 8. 権限と所有権の扱い

MVPでは単一ユーザー利用が前提だが、内部的には owner を user_id で分離する。

### 原則

- notes は owner のみ参照・更新可能
    
- comparisons は owner のみ参照可能
    
- strategies / backtests は owner のみ参照・更新可能
    
- webhook は認証トークンまたは秘密URLで受ける

## 9. エラー設計方針

代表的なエラーコードを以下とする。

- VALIDATION_ERROR
    
- NOT_FOUND
    
- UNAUTHORIZED
    
- FORBIDDEN
    
- CONFLICT
    
- IMPORT_PARSE_ERROR
    
- AI_JOB_FAILED
    
- EXTERNAL_PAYLOAD_INVALID
    

### 例

- symbol_id が存在しない → NOT_FOUND
    
- strategy version と pine script の組み合わせ不整合 → CONFLICT
    
- CSV パース失敗 → IMPORT_PARSE_ERROR
    
- webhook payload 必須項目不足 → EXTERNAL_PAYLOAD_INVALID

## 12. 結論

北極星MVPの API / 入出力設計は、画面そのものではなく、  
**毎日状況把握する / 銘柄研究する / 比較する / ルール検証する** というユースケース単位で構成する。

そのうえで、MVPでは以下を中心に先行実装する。

- ホーム取得 API
    
- 銘柄詳細取得 API
    
- 研究ノート API
    
- TradingView webhook 受信 API
    
- AI要約取得 API
    
- 比較 API
    
- ルール登録 / Pine生成 API
    
- バックテスト取込 API
    

この構成により、北極星の独自価値である **要約・比較・仮説・履歴管理** を支える実装基盤を作る。
