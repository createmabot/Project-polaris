# 北極星 docs正本整理・読む順番（現行）

更新日: 2026-05-15

## 1. 目的

本資料は、MVP初期から積み上がった `docs/` を、現在の作業でどの順に読むべきか、どれを正本として扱うべきか、どれを履歴資料として扱うべきかを整理する。現行 docs 体系は「仕様書」/「運用ドキュメント」/「作業進捗管理」の3分類で扱う。

既存番号 docs の削除・rename は原則避ける。ただし移管済みでリンク影響が小さいものは `docs/作業進捗管理/06_番号付きdocs削除整理.md` に理由と移管先を記録し、小さな PR で削除する。古い前提を含む資料は注意書きと分類で扱い、必要な実装判断は現在の正本 docs に寄せる。

## 2-1. docs体系の3分類

| 分類 | ディレクトリ | 役割 |
|---|---|---|
| 仕様書 | `docs/仕様書/` | 現在のシステム仕様・画面仕様・データ/API/AI/UI component仕様。作業履歴やPR経緯は入れない。 |
| 運用ドキュメント | `docs/運用ドキュメント/` | 開発・起動・テスト・運用・確認・トラブルシュート手順。 |
| 作業進捗管理 | `docs/作業進捗管理/` | ロードマップ、フェーズ完了整理、残課題、decision log。 |

## 4. README / walkthrough の位置づけ

- `README.md` は開発者向け入口。セットアップ、主要コマンド、現行API概要を短く確認する。
- `docs/walkthrough.md` は動作確認と受け入れ確認の正本。仕様判断の入口ではなく、実装後の確認手順として読む。
- `docs/0.目次.md` は docs入口。どの正本を読むかを決めるための index として扱う。
- 詳細仕様の判断は、機能別正本 docs に進む。

## 5. 古い前提を含む資料の扱い

- MVP初期の意思決定と比較できる。

- 後続で正本に昇格させる場合、差分を追いやすい。

2. `docs/39` / `docs/44` / `docs/53` に積み上がった追記の再構成。
