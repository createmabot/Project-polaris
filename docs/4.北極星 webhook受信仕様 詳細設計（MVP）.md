# 北極星 webhook受信仕様 詳細設計（MVP）

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」における **TradingView webhook受信機能** の詳細仕様を定義するものである。

本仕様の対象は、TradingView から送信されるアラートを北極星側で受け取り、イベントとして保存し、その後のAI要約や研究導線に接続する一連の流れである。

MVPにおいて webhook は、以下の3コンセプトのうち特に **毎日5分で状況把握** を支える中核機能である。

- 毎日5分で状況把握
    
- 銘柄研究の高速化
    
- ルール運用の検証
    

---

## 2. webhook機能の目的

TradingView の価格アラート、テクニカルアラート、watchlist alert を北極星へ送信し、以下を実現する。

1. アラートの履歴保存
    
2. 銘柄単位のイベント管理
    
3. 「今日動いた理由」のAI要約生成起点化
    
4. ホーム画面上での注目イベント表示
    
5. 銘柄研究画面への導線生成
    

---

## 3. webhook設計の基本方針

### 3.1 受信は軽く、後段処理は非同期にする

webhook受信時点では、重い処理を行わない。

受信エンドポイントでは最低限のみを行う。

- リクエスト検証
    
- 認証確認
    
- payload保存
    
- symbol解決
    
- alert_event作成
    
- 非同期ジョブ投入
    
- 即時応答
    

### 3.2 TradingView payload は可能な限り原文保存する

後でパース仕様を改善したり、障害調査を行ったりするため、受信したpayloadは生に近い形で保持する。

### 3.3 北極星内の標準イベント形式へ正規化する

TradingView から届く payload は柔軟である一方、北極星側では後続処理を安定化させる必要がある。  
そのため、受信後に内部標準形式へマッピングする。

### 3.4 冪等性を考慮する

同一アラートの重複受信や、再送・重複登録に備え、重複判定キーを持つ。

### 3.5 webhookは外部境界APIとして扱う

通常の画面APIとは別責務とし、バリデーション・認証・ログ・監視を強める。

## 5. webhook受信のシステム境界

## 5.1 外部送信元

- TradingView webhook

## 5.2 北極星側受信対象

- Public webhook endpoint
- 非同期ジョブキュー
- alert_event 永続化
- AIジョブ起動
- 外部参照収集ジョブ起動

## 5.3 処理対象外

## 6. エンドポイント仕様

## 6.1 受信URL

### Endpoint

`POST /api/integrations/tradingview/webhook`

### 用途

TradingViewから送信されるアラート payload を受信する。

### Content-Type

- application/json
- text/plain

### 備考

MVPでは TradingView 側の設定自由度を考慮し、JSON と plain text の両方を許容する。  
ただし、北極星としての推奨は JSON payload とする。

---

## 6.2 認証方式

MVPでは以下の二段構えを採用する。

### 方式A: 秘密URLトークン

受信URLに秘密トークンを含める。

例:  
`POST /api/integrations/tradingview/webhook?token=tv_xxxxx`

#### 特徴

- 設定が簡単
- TradingView側設定が容易
- MVP向き

### 方式B: Payload内 shared_secret

payload内に `shared_secret` を入れ、サーバ側で検証する。

#### 特徴

- URL流出時の保険になる    
- 将来の切替に備えられる

### MVPの採用方針

- URL token 必須
- shared_secret 任意だが推奨
- どちらか片方ではなく、将来的には両方利用可能な設計にする    

MVPでは token は単なる秘密文字列ではなく、**owner_user_id を解決する識別子** として扱う。 webhook 受信時は、token 検証と同時に owner_user_id を確定し、alert_event.user_id に保存する。

### 推奨テーブル案

- webhook_tokens
    - id
    - user_id
    - provider
    - token_hash
    - shared_secret_hash
    - is_active
    - created_at
    - rotated_at

---

## 6.3 許可対象IP

MVPでは allowlist を必須化しない。  
理由は以下。

- 運用環境により設定コストが増える
- MVP段階でまず動作確立を優先する

ただし将来対応のため、IP allowlist を追加しやすい構造にする。

## 7. 推奨payload仕様

TradingViewは alert message を自由に書けるため、北極星側では **推奨標準payload** を定義する。
## 7.1 外部受信 payload v1

TradingView から北極星へ送られる推奨 JSON 形式を定義する。 これは **外部境界の受信仕様** であり、内部保存形式とは分けて扱う。

### 外部受信 payload v1

```json
{
	"provider": "tradingview",
	"shared_secret": "tv_shared_secret_xxx",
	"alert_name": "MA25 breakout",
	"alert_type": "technical",
	"tradingview_symbol": "TSE:7203",
	"symbol": "7203",
	"market_code": "TSE",
	"timeframe": "D",
	"trigger_price": 3021.5,
	"currency": "JPY",
	"triggered_at": "2026-03-11T00:15:00Z",
	"message": "7203 triggered technical alert",
	"condition_summary": "close crossed above MA25",
	"watchlist_name": null,
	"strategy_name": null,
	"metadata": {
		"open": 3001.0,
		"high": 3030.0,
		"low": 2998.0,
		"volume": 18450000
	}
}
```

### 命名統一方針

- `alert_type` を採用する
- `market_code` を採用する
- `trigger_price` を採用する
- `condition_summary` はトップレベルで持つ
- 補助的な追加値は `metadata` に入れる

---

## 7.2 必須項目

MVPで最低限必要とする項目は以下。

- alert_name
- tradingview_symbol または symbol
- alert_type
- timeframe
- triggered_at
### 補足

`trigger_price` はできれば欲しいが、受信不能条件にはしない。

---

## 7.3 任意項目

- shared_secret
    
- market_code
    
- display_name
    
- currency
    
- condition_summary
    
- watchlist_name
    
- strategy_name
    
- message
    

---

## 7.4 plain text payload の扱い

plain text の場合は、以下の扱いとする。

- raw_body をそのまま保存
    
- 構造化パースは最小限
    
- symbol 解決に失敗した場合は `processing_status=needs_review`
    
- UI上で「構造化不足」として表示可能にする
    

MVPでは plain text は互換性維持のため許容するが、実運用では JSON を推奨する。

## 9. symbol解決仕様

Webhook受信後、銘柄を内部 `symbols` テーブルへ解決する必要がある。

## 9.1 解決優先順位

1. tradingview_symbol 完全一致
    
2. market_code + symbol の組み合わせ
    
3. symbol_code 単独一致
    
4. display_name 補助一致（MVPでは非推奨）
    

## 9.2 解決結果パターン

### 成功

- symbol_id を確定
    
- alert_event 保存継続
    

### 候補複数

- `processing_status=needs_review`
    
- raw payload 保存
    
- 管理画面または後続処理対象に回す
    

### 失敗

- `processing_status=unresolved_symbol`
    
- raw payload 保存
    
- 要約処理は起動しない
    

---

## 10. 重複排除・冪等性

同一イベントの重複登録を防ぐため、冪等キーを持つ。

## 10.1 冪等キー候補

以下の組み合わせを候補とする。

- source_type
    
- tradingview_symbol
    
- alert_name
    
- timeframe
    
- triggered_at
    
- trigger_price
    

これをハッシュ化して `dedupe_key` とする。

## 10.2 重複時の方針

### 完全重複

- 新規 alert_event は作らない
    
- 既存イベントを返す
    
- 受信ログのみ残してよい
    

### 近似重複

例: 同じ alert_name だが trigger_price が微差

- 新規イベントとして扱う
    
- ただし UI 側で近接イベントの束ね表示を検討可能
    

---

## 11. 受信後の状態遷移

## 11.1 alert_event.processing_status

MVPでは以下の状態を持つ。

- received
    
- normalized
    
- unresolved_symbol
    
- queued_for_enrichment
    
- enrichment_running
    
- summarized
    
- needs_review
    
- failed
    

## 11.2 基本遷移

### 正常系

received → normalized → queued_for_enrichment → enrichment_running → summarized

### symbol未解決

received → unresolved_symbol

### payload不正

received → failed

### 補足レビュー必要

received → normalized → needs_review

## 12. 受信処理フロー

## 12.1 同期処理

1. token 検証
    
2. body 受信
    
3. content-type 判定
    
4. raw payload 保存
    
5. 最低限バリデーション
    
6. symbol 解決
    
7. alert_event 作成または重複判定
    
8. 非同期ジョブ投入
    
9. 即時レスポンス返却
    

## 12.2 非同期処理

1. 外部参照収集ジョブ起動
    
2. ニュース/開示/決算参照の紐付け
    
3. AI要約生成ジョブ起動
    
4. ai_summary 保存
    
5. alert_event 状態更新
    

---

## 13. バリデーション仕様

## 13.1 リクエストレベル

- HTTP method は POST のみ
    
- token 必須
    
- body 必須
    
- content-type は application/json または text/plain を許可
    

## 13.2 JSON payload レベル

- JSON parse 可能であること
    
- 必須キー不足時は `EXTERNAL_PAYLOAD_INVALID`
    
- triggered_at が日付として解釈不能なら invalid
    

## 13.3 business validation

- symbol 解決不可は受信エラーではなく処理エラー扱い
    
- alert_type は未知値でも raw として受ける
    
- timeframe は未知値でも raw として受けるが warnings を残す

## 15. エラー処理方針

## 15.1 エラー分類

### 認証エラー

- token不一致
    
- shared_secret不一致
    

### 構文エラー

- JSON parse失敗
    
- body空
    

### 業務エラー

- symbol未解決
    
- 必須項目不足
    
- 重複
    

### システムエラー

- DB保存失敗
    
- ジョブ投入失敗
    

## 15.2 返却方針

- 認証・構文エラーは 4xx
    
- 内部エラーは 5xx
    
- symbol未解決は 2xx で受理し、内部状態で unresolved にする
    

### 理由

外部送信元に対しては「配信成功」と「後段処理成功」を分けて考えるため。

## 17. セキュリティ方針

## 17.1 基本方針

- HTTPS 前提
    
- token を長く推測困難にする
    
- 機密情報を payload に含めない
    
- raw payload は監査目的で保存するが、表示時はマスク可能にする
    

## 17.2 リプレイ対策

MVPでは厳密な署名検証は行わないが、以下を行う。

- dedupe_key による重複抑止
    
- triggered_at の過度な古さを検出し warning 化
    
- token ローテーション可能設計
    

## 17.3 レート制限

MVPでも最低限以下を設ける。

- token 単位の burst 制限
    
- IP 単位の上限監視

## 20. MVPで採用する運用ルール

## 20.1 推奨アラートメッセージ形式

MVPでは JSON payload を標準とする。  
plain text は互換モードとする。

## 20.2 TradingView設定ルール

ユーザー側の alert 設定では、可能な限り以下を含める。

- tradingview_symbol
- alert_name
- alert_type
- timeframe
- trigger_price
- triggered_at
- condition_summary

## 20.3 unresolved_symbol の扱い

未解決イベントは捨てずに保存する。  
後で symbol マスタ追加やパーサ改修後に再処理可能にする。

## 20.4 まずは single endpoint で始める

MVPでは、価格アラート・テクニカルアラート・watchlist alert を単一 endpoint で受ける。  
将来 provider や用途ごとに増やせるようにする。

## 20.5 時刻・営業日ルール

MVPでは以下を統一ルールとする。
- 保存時刻は UTC を正本とする
- 表示時刻は Asia/Tokyo を標準とする
- `triggered_at` は受信時に UTC へ正規化して保存する
- ホームの日次集約は「日本市場営業日」を基準にする
- 引け後の重要イベントは当日夜サマリに含め、必要に応じて翌営業日朝サマリにも再掲できる
- `received_at` と `triggered_at` は別カラムで保持する

## 23. 結論

北極星MVPの webhook受信仕様は、TradingView からのアラートを **軽く受け、正規化し、保存し、非同期で価値化する** ことを目的とする。

そのため、MVPでは以下を採用する。

- 単一 webhook endpoint
    
- token ベース認証
    
- JSON payload 推奨
    
- raw payload 保存
    
- symbol 解決と標準形式への正規化
    
- 冪等性確保
    
- 非同期ジョブによる要約生成
    
- unresolved なイベントも破棄せず保存
    

この設計により、北極星の「毎日5分で状況把握」を安定して支える webhook 基盤を構築する。
## 24. 実送信運用固定（2026-05 追記）

- TradingView 実送信の運用手順は `docs/32.北極星 TradingView実送信 webhook運用手順（MVP）.md` を正本とする。
- endpoint は `POST /api/integrations/tradingview/webhook` を推奨（`/api/webhooks/tradingview/webhook` は後方互換）。
- 認証は token（Query/Bearer）を必須とし、shared_secret 併用を推奨する。
- secret実値は docs/ログ/PR本文に残さない。
- failed 切り分けは auth / parse / unresolved_symbol / duplicate_ignored / summary_failed の順で確認する。
