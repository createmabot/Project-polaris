# 北極星 GitHub Projects 用カラム設計案（MVP）

更新日: 2026-03-11

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVP開発において、**GitHub Projects で実装チケットを運用するためのカラム設計案** を定義するものである。

これまでに、以下の実装管理に必要な資料を整理してきた。

- MVPタスク分解
    
- 実装チケット化用の粒度分解
    
- スプリント計画案
    
- 初期開発チェックリスト
    
- DBマイグレーション順序案
    
- コンポーネント分割案
    

一方で、実装チケットが揃っても、運用ボードの列設計が曖昧だと次の問題が起きやすい。

- 何が着手前で、何がレビュー待ちか分からない
    
- backend / frontend / qa の流れが見えない
    
- スプリント対象と未着手 backlog が混ざる
    
- blocked なものが埋もれる
    
- 完了条件を満たしていないのに Done に移る
    

そのため本資料では、MVPにちょうどよい GitHub Projects の列設計と、補助フィールド、ラベル運用、カード移動ルールを整理する。

---

## 2. 基本方針

### 2.1 列は少なすぎず多すぎずにする

MVPでは、詳細な工程管理よりも **流れが止まっている場所が見えること** を重視する。

### 2.2 backlog と sprint 対象を分けて考える

GitHub Projects では同じボード上に置いてもよいが、

- 今回の sprint 対象
    
- まだ着手しない backlog
    

が見分けられるようにする。

### 2.3 実装・レビュー・検証を分ける

北極星は外部入力・AI出力・CSV取込など壊れやすい処理が多いため、  
**実装完了** と **確認完了** を分ける。

### 2.4 blocked を必ず可視化する

symbol 解決、AI schema、CSV parser のように依存が強いものは、  
詰まった瞬間に見える必要がある。

### 2.5 Done の基準を厳密にする

コードを書いただけではなく、最低限の完了条件を満たしてから Done に移す。

---

## 3. 推奨カラム構成（メイン案）

MVPでは、以下の 8 カラム構成を推奨する。

1. Inbox
    
2. Ready
    
3. In Progress
    
4. Blocked
    
5. In Review
    
6. QA / Verify
    
7. Done
    
8. Later
    

---

## 4. 各カラムの意味

## 4.1 Inbox

### 役割

- まだ整理しきれていないチケット
    
- 起票直後
    
- 粒度が大きすぎる、または不足しているもの
    

### 置くもの

- 新規 Issue
    
- 補助的アイデア
    
- 後で分解が必要な作業
    

### ここから移す条件

- タイトルが具体化されている
    
- 目的と完了条件が書かれている
    
- 区分と優先度が付いている
    

---

## 4.2 Ready

### 役割

- 着手可能なチケット
    
- 依存関係が満たされている
    
- スプリント対象として選ばれたもの
    

### 置くもの

- 今スプリントでやる P0 / P1 チケット
    
- 直近で取りにいける作業
    

### ここから移す条件

- 依存チケットが Done または不要
    
- 仕様が不足していない
    
- 担当が取れば実装開始できる
    

---

## 4.3 In Progress

### 役割

- 現在実装中のチケット
    

### 置くもの

- backend 実装中
    
- frontend 実装中
    
- migration 作成中
    
- parser 実装中
    

### ルール

- 1人が同時に持つカード数を絞る
    
- 長く留まるカードは細分化を検討する
    

---

## 4.4 Blocked

### 役割

- 着手はしたが依存や問題で止まっているチケット
    

### 典型例

- TradingView 実機確認待ち
    
- symbol seed 不足
    
- API schema 未確定
    
- AI 出力の shape が不安定
    
- CSV fixture 不足
    

### 運用ルール

- Blocked 理由をコメントで残す
    
- ブロッカーとなる親チケットを明記する
    
- 長く滞留する場合は別チケットを起票する
    

---

## 4.5 In Review

### 役割

- 実装は終わったがコードレビュー待ち
    

### 置くもの

- PR 作成済み
    
- reviewer 待ち
    

### ここへ移す条件

- コードが push 済み
    
- PR がある
    
- 自己確認が終わっている
    

---

## 4.6 QA / Verify

### 役割

- 実装レビュー後、動作確認や受け入れ確認を行う状態
    

### 北極星で特に重要なもの

- webhook 実疎通
    
- AI structured_json が schema 通りか
    
- CSV parser が fixture で通るか
    
- 画面で expected view model になっているか
    

### ここへ移す条件

- review を通過している
    
- 確認観点がコメントまたは checklist に書かれている
    

---

## 4.7 Done

### 役割

- 実装、レビュー、確認まで完了した状態
    

### Done 条件の基本

- 完了条件を満たしている
    
- 必要なテストが通っている
    
- 関連ドキュメント更新が必要なら反映済み
    

---

## 4.8 Later

### 役割

- 後回しチケット
    
- P2
    
- MVPではやらないが捨てないもの
    

### 置くもの

- 高度な比較総評
    
- ポートフォリオ機能
    
- 多時間足対応
    
- trailing stop 正式対応
    

---

## 5. カラム遷移の基本ルール

推奨遷移は以下。

```text
Inbox → Ready → In Progress → In Review → QA / Verify → Done
                  └────────────→ Blocked ───────────┘
Later → Ready
```

### 補足

- In Progress から直接 Done にしない
    
- Review を飛ばさない
    
- Verify を飛ばすのはごく小さい docs 修正だけにする

## 8. カードに最低限書くべき内容

各 Issue / カードには最低限以下を含める。

- 目的
    
- 作業内容
    
- 完了条件
    
- 依存チケット
    
- 備考（あれば）
    

### 良い例

- `T-026 symbol 解決ロジックを実装する`
    
    - tradingview_symbol 優先
        
    - exchange + symbol 補助
        
    - unresolved 時は processing_status を残す
        
    - 完了条件: representative payload 3種で解決可/不可が仕様通りになる
        

---

## 9. Done の定義

GitHub Projects 運用で最も重要なのは Done の定義。

### Done に入れる条件

- コードが main 相当に反映済み
    
- 完了条件を満たしている
    
- 必要なテストが通っている
    
- フロントから触る機能なら最低限の画面確認が終わっている
    
- ログ / エラーメッセージが必要なものは最低限確認されている
    

### Done に入れない例

- ローカルでしか動いていない
    
- PR だけ出して確認未了
    
- 仕様と違う簡易実装のまま
    
- fixture や schema が未整備

## 10. 北極星向けのビュー切り替え案

GitHub Projects では、以下のビューを分けると運用しやすい。

## 10.1 Main Board

- カラム: Inbox / Ready / In Progress / Blocked / In Review / QA / Done / Later
    
- 用途: 全体進行確認
    

## 10.2 Sprint View

- フィルタ: `Sprint = 現在スプリント`
    
- 用途: 今回やるものだけ見る
    

## 10.3 Backend View

- フィルタ: `Area = backend OR db OR ai OR integration`
    
- 用途: サーバ / DB 作業に集中する
    

## 10.4 Frontend View

- フィルタ: `Area = frontend`
    
- 用途: UI 実装確認
    

## 10.5 QA View

- フィルタ: `Area = qa OR column = QA / Verify`
    
- 用途: 確認待ち作業を集約する
    

## 10.6 Risk View

- フィルタ: `Risk = high`
    
- 用途: 高リスクタスクの先回り確認

## 11. MVP開始時の推奨カード投入順

最初にボードへ投入するなら、以下の順が扱いやすい。

### 最初に Ready へ入れる

- T-001 プロジェクト初期構成を作成する
    
- T-002 環境変数 / secrets 管理を整備する
    
- T-003 API サーバ雛形を作成する
    
- T-004 フロントエンド雛形を作成する
    
- T-010 DB 接続と migration 基盤を整備する
    
- T-012 symbols テーブルを追加する
    

### その次に Ready 候補

- T-024 TradingView webhook endpoint を実装する
    
- T-026 symbol 解決ロジックを実装する
    
- T-027 alert_event 正規化 / 保存処理を実装する
    

### Later に先に置いてよいもの

- タグ系
    
- 高度比較 UI
    
- P2 の運用強化タスク
    

---

## 12. Blocked の扱いルール

北極星では Blocked が重要なので、運用ルールを決める。

### Blocked に移す条件

- 自分だけでは解決できない依存がある
    
- fixture / seed / spec 不足で進められない
    
- TradingView 実機確認待ち
    
- AI 出力が schema 不一致で設計見直しが必要
    

### Blocked に置いたらやること

- なぜ止まっているかコメントを書く
    
- 何が解消条件か書く
    
- 必要ならブロッカー用チケットを別に立てる
    

### 悪い運用

- 単に後回しを Blocked に置く
    
- 理由なしで Blocked に移す

## 13. チケットサイズの運用ルール

## 13.1 XS / S

- 単一 migration
    
- 単一 API の軽微追加
    
- 単一 UI 部品

`L` や `XL` のチケットは Inbox か Ready に置く前に再分解を検討する。

## 15. 運用開始時のサンプルカラム配置

```text
[Inbox] [Ready] [In Progress] [Blocked] [In Review] [QA / Verify] [Done] [Later]
```

### 初期運用イメージ

- Inbox: 分解不足チケット
    
- Ready: Sprint 1 の確定チケット
    
- In Progress: いま着手中
    
- Blocked: webhook / seed / schema 待ち
    
- In Review: PR 待ち
    
- QA / Verify: 実機 / fixture 確認待ち
    
- Done: 完了
    
- Later: P2 / 後回し

## 18. 結論

北極星MVPの GitHub Projects 運用では、  
**Inbox / Ready / In Progress / Blocked / In Review / QA / Done / Later** の8カラム構成を基本とし、  
あわせて Priority / Area / Sprint / Stream / Estimate / Risk をカスタムフィールドとして持つのが扱いやすい。

そのうえで、MVPでは以下を重視する。

- backlog と sprint 対象を分ける
    
- blocked を明示する
    
- review と verify を分ける
    
- Done の基準を厳密にする
    
- 高リスクタスクと3本のコア導線を見失わない
    

この運用により、北極星のMVP実装を GitHub Projects 上で無理なく管理しやすい状態を作る。
