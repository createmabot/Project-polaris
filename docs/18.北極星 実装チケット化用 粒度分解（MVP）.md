# 北極星 実装チケット化用 粒度分解（MVP）

更新日: 2026-03-09

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVP開発に向けて、これまで整理してきた仕様・設計・画面・API・データモデルを、**実装チケットとして管理しやすい粒度** に分解することを目的とする。

単なる機能一覧ではなく、以下を含めて整理する。

- チケット単位の目的
    
- 作業内容
    
- 依存関係
    
- 完了条件
    
- 優先度
    
- 担当しやすい単位への分割
    

本資料は、スプリント計画、Issue 起票、GitHub Projects / Notion / Jira などへの転記をしやすくするための下敷きとする。

---

## 2. チケット設計の基本方針

### 2.1 1チケット = 1責務を原則とする

- DB スキーマ追加
    
- API 1本
    
- UI 1画面の1ブロック
    
- パーサ 1種類
    
- ジョブ 1種類  
    のように、責務をできるだけ分ける。
    

### 2.2 依存の強いものは分離して可視化する

特に北極星では以下の依存が強い。

- symbol マスタ → webhook 正規化
    
- alert_events → references 収集 → AI要約
    
- normalized_rule_json → Pine生成
    
- backtest_run → CSV取込 → 比較 / 総評
    

そのため、後続タスクの前提になるチケットは明示する。

### 2.3 完了条件を必ず書く

「実装した」で終わらず、最低限の受け入れ条件を明記する。

### 2.4 MVPで不要なものを混ぜない

後回し機能は別枠に分離し、実装チケットへ混入させない。

## 3. チケットの共通フォーマット

各チケットは以下の形式で記述する。

- チケットID
    
- タイトル
    
- 区分
    
- 目的
    
- 作業内容
    
- 依存チケット
    
- 完了条件
    
- 優先度
    

### 区分の例

- backend
    
- frontend
    
- db
    
- ai
    
- integration
    
- qa
    
- ops
    

### 優先度の例

- P0
    
- P1
    
- P2
    

---

## 4. チケット群の全体構造

MVPの実装チケットは以下の7群に分ける。

1. 基盤 / プロジェクトセットアップ
    
2. 共通データ / DB
    
3. 市場観測 / webhook / 要約
    
4. 銘柄研究 / ノート
    
5. ルール検証 / Pine
    
6. バックテスト取込 / 比較
    
7. テスト / 運用 / 品質

# 5. 基盤 / プロジェクトセットアップ

## T-001 プロジェクト初期構成を作成する

- 区分: backend / frontend / ops
    
- 目的: 開発を開始できるディレクトリ構成と共通ルールを整える
    
- 作業内容:
    
    - backend, frontend, docs, fixtures, scripts の作成
        
    - README 作成
        
    - 開発起動方法の明文化
        
- 依存チケット: なし
    
- 完了条件:
    
    - リポジトリを clone して起動手順が読める
        
    - backend / frontend の雛形が存在する
        
- 優先度: P0
    

## T-002 環境変数 / secrets 管理を整備する

- 区分: ops
    
- 目的: 環境依存設定を安全に扱えるようにする
    
- 作業内容:
    
    - `.env.example` 作成
        
    - DB URL、AI API key、webhook token 変数の定義
        
    - ローカル / 本番差分方針を整理
        
- 依存チケット: T-001
    
- 完了条件:
    
    - 必須環境変数が一覧化されている
        
    - `.env.example` が存在する
        
- 優先度: P0
    

## T-003 API サーバ雛形を作成する

- 区分: backend
    
- 目的: 以降の API 実装の土台を作る
    
- 作業内容:
    
    - health check endpoint
        
    - 共通レスポンス形式
        
    - error middleware
        
    - request_id 付与
        
- 依存チケット: T-001, T-002
    
- 完了条件:
    
    - `/health` が返る
        
    - 共通 error format が動作する
        
- 優先度: P0
    

## T-004 フロントエンド雛形を作成する

- 区分: frontend
    
- 目的: 画面実装を開始できる状態にする
    
- 作業内容:
    
    - routing
        
    - 共通 layout
        
    - API client 雛形
        
- 依存チケット: T-001
    
- 完了条件:
    
    - ホーム仮画面が表示される
        
    - API 呼び出し基盤がある
        
- 優先度: P0
    

## T-005 非同期ジョブ基盤を作成する

- 区分: backend
    
- 目的: AI 要約、references 収集、CSV パースの非同期処理基盤を用意する
    
- 作業内容:
    
    - queue 基盤
        
    - job enqueue / consume 雛形
        
    - job failure handling 雛形
        
- 依存チケット: T-003
    
- 完了条件:

- 優先度: P0

# 17. 結論

北極星MVPの実装チケット化では、仕様を大きな塊のまま扱わず、  
**1責務・依存関係明示・完了条件付き** の単位に分解することを基本方針とする。

そのうえで、MVPでは以下の3本の縦導線を成立させるためのチケット束を最優先にする。

- TradingView alert → 要約 → ホーム
    
- 銘柄詳細 → AI論点 → ノート
    
- 自然言語 → Pine → CSV取込 → 総評
    

この粒度まで落とすことで、実装・進行管理・受け入れ確認を進めやすい状態にする。
