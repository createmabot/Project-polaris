# 北極星 application-specific reports endpoint 設計（次フェーズ）

更新日: 2026-05-12

## 1. 目的

Symbol Strategy Application 配下の Backtest report 一覧を取得する endpoint 方針を整理する。

今回の目的は、既存 `GET /api/symbols/:symbolId/strategy-applications` の latest report summary、`BacktestDetail` の `related_reports` / metrics 横並び比較補助、`GET /api/symbol-strategy-applications/:applicationId/runs` 候補の run history と役割を分け、次の最小実装へ進む場合の候補を絞ることである。

この文書は docs-only の設計整理であり、今回 API / backend / frontend / DB / Prisma schema / test は変更しない。

## 2. 現在地

- P3 は完了扱いである。
- Symbol Strategy Application / Run / Backtest report の基本関係は docs/52 を正本とする。
- application-specific runs endpoint の設計は docs/54 を正本とする。
- 現行 DB には `SymbolStrategyApplication` と `SymbolStrategyApplicationRun` が存在する。
- 現行 `GET /api/symbols/:symbolId/strategy-applications` は application list 用 endpoint として、`latest_run`、`latest_backtest_report`、`latest_reports_by_source`、pagination meta を返す。
- 現行 list endpoint の `run_type` / `run_status` filter は latest run 基準であり、any run 履歴検索ではない。
- `BacktestDetail` は個別 report detail として、Symbol Strategy Application backlink、`current_report`、`related_reports`、metrics summary を read-only に返す。
- CSV import report は `Backtest` と `BacktestImport` を持つ。
- internal backtest report は `Backtest` と `InternalBacktestExecution` を run 経由で持ち、`BacktestImport` を作らない importless report として扱う。

## 3. endpoint 第一候補

第一候補は次とする。

```text
GET /api/symbol-strategy-applications/:applicationId/reports
```

目的:

- application 配下の Backtest report 一覧を取得する。
- CSV import report と internal backtest report を source 別に見比べる。
- report 一覧上で主要 metrics preview、linked run summary、BacktestDetail link summary を確認できるようにする。
- run history ではなく、検証 report 一覧 / 比較 UX を主語にする。
- `SymbolDetail` の application row を肥大化させず、必要になった時点で report list / comparison UI へ接続する。

## 4. runs endpoint との役割境界

### runs endpoint

```text
GET /api/symbol-strategy-applications/:applicationId/runs
```

役割:

- 実行履歴が主語。
- queued / running / succeeded / failed / canceled を含む run lifecycle を返す。
- CSV import run、internal backtest run、report conversion 前後の状態を時系列で確認する。
- report が存在しない failed / running / queued run も返す。
- `linked_backtest` は link summary に留め、metrics 横並び比較や AI summary 比較は背負わない。

### reports endpoint

```text
GET /api/symbol-strategy-applications/:applicationId/reports
```

役割:

- 検証 report 一覧 / 比較 UX が主語。
- Backtest report が存在する run を中心に返す。
- CSV import report と internal backtest report を source / metrics / linked run 観点で整理して返す。
- BacktestDetail へ遷移するための link summary と、一覧内 preview に必要な metrics summary を返す。
- queued / running / failed で report が存在しない run は通常返さない。必要な場合は `importless_report_marker` や linked run summary で「なぜ report として成立しているか」を示す。

判断:

- run の成否や過去実行履歴を調べる場合は runs endpoint。
- report を横断して見比べる場合は reports endpoint。
- existing list endpoint は symbol 配下 application list / latest summary 用として維持する。

## 5. query 候補

初回候補は以下に絞る。

| query | allowed value | 既定 | 意味 |
|---|---|---|---|
| `execution_source` | `tradingview` / `internal_backtest` | 未指定 | Backtest report の `execution_source` で絞る |
| `run_type` | `csv_import` / `internal_backtest` | 未指定 | report に linked した application run の type で絞る |
| `status` | string | 未指定 | Backtest report の status で絞る |
| `with_metrics` | `true` / `false` | `true` | metrics summary を response に含めるか |
| `page` | positive integer | `1` | pagination page |
| `limit` | positive integer | `20` | pagination limit。上限は既存 list endpoint と揃え、初回は `50` を候補にする |
| `sort` | `created_at` / `updated_at` | `created_at` | report 一覧の並び替え対象 |
| `order` | `asc` / `desc` | `desc` | report 一覧の並び順 |

判断:

- `execution_source` は Backtest report の由来を表す。`tradingview` は CSV import report、`internal_backtest` は internal backtest report を表す。
- `run_type` は application run の由来を表す。CSV import report は `csv_import`、internal backtest report は `internal_backtest` を想定する。
- `status` は application status ではなく Backtest report status として扱う。application parent の status filter は初回 query には入れない。
- `with_metrics=false` は一覧件数だけ必要な場合や軽量取得用の候補にする。
- `strategy_id` / `strategy_version_id` は application が固定しているため入れない。
- `run_status` は reports endpoint の初回 query には入れない。run lifecycle を調べる場合は runs endpoint を使う。

## 7. CSV import report の扱い

CSV import report は次の関係で返す。

- `execution_source = tradingview`
- `report_origin = csv_import`
- linked run の `run_type = csv_import`
- linked run の `backtest_id` はあり
- linked run の `backtest_import_id` はあり
- linked run の `internal_backtest_execution_id` は null
- `importless_report = false`
- metrics source は `BacktestImport.parsedSummaryJson`
- `BacktestDetail` では既存 import 履歴、parsed summary、inline comparison、AI summary 導線を維持する

parse failed report の扱い:

- `Backtest.status = import_failed` の report を返すかは `status` query と UX 方針で判断する。
- 返す場合、metrics は欠損し得るため `metrics` は null または空 summary とする。
- parse error 詳細は reports endpoint では背負いすぎず、必要なら `linked_import_summary` を後続候補にする。
- failed import の詳細確認は BacktestDetail に寄せる。

## 8. internal backtest report の扱い

internal backtest report は次の関係で返す。

- `execution_source = internal_backtest`
- `report_origin = internal_backtest`
- linked run の `run_type = internal_backtest`
- linked run の `backtest_id` はあり
- linked run の `backtest_import_id` は null
- linked run の `internal_backtest_execution_id` はあり
- `importless_report = true`
- metrics source は `Backtest.strategySnapshotJson.result_summary`
- `linked_internal_backtest_execution` は status / requested / started / finished / engine version 程度の summary に留める

report conversion 前:

- `Backtest` が存在しないため、reports endpoint には原則出さない。
- conversion 前の execution / run 状態は runs endpoint で確認する。

report conversion 後:

- `Backtest` が存在するため、reports endpoint に出す。
- `BacktestImport` がないことは欠損ではなく、`importless_report = true` として明示する。
- artifact pointer や result summary の詳細表示は BacktestDetail に寄せる。

## 9. metrics summary 方針

metrics summary は read-only preview として扱う。

共通 field 候補:

- `period_from`
- `period_to`
- `trade_count`
- `total_return_percent`
- `price_change_percent`
- `max_drawdown_percent`
- `profit_factor`
- `win_rate`
- `source`

CSV import report:

- `BacktestImport.parsedSummaryJson` から読める範囲に限定する。
- 現行 parsed summary の `totalTrades`、`winRate`、`profitFactor`、`maxDrawdown`、`netProfit`、`periodFrom`、`periodTo` を mapping する。
- `total_return_percent` / `price_change_percent` は現行 CSV summary からは直接取れない場合があるため null を許容する。

internal backtest report:

- `Backtest.strategySnapshotJson.result_summary.metrics` と `period` から読める範囲に限定する。
- `trade_count`、`total_return_percent`、`price_change_percent`、`max_drawdown_percent`、`profit_factor`、`win_rate` は存在する field だけ返す。
- artifact pointer、bar-level result、download は reports endpoint には含めない。

判断:

- metrics normalization table は初回不要。
- comparison entity / comparison result 保存は初回不要。
- 欠損値は null とし、UI 側で CSV / internal の source 差による欠損と説明する。

## 10. BacktestDetail の `related_reports` / metrics comparison との関係

現行 `BacktestDetail` は、表示中 report を主語にして次を返す。

- `symbol_strategy_application.current_report`
- `symbol_strategy_application.related_reports[]`
- current / related の metrics summary
- SymbolDetail / StrategyDetail / StrategyVersionDetail への backlink

reports endpoint は、application を主語にして同じ情報を一覧化する。

役割分担:

- `BacktestDetail` は個別 report detail と、その report から見た related reports を返す。
- reports endpoint は application 配下の report list を返し、比較 UX の入口にする。
- `BacktestDetail.related_reports` は現在表示中 report を除外する。
- reports endpoint の `reports[]` は application 配下の対象 report をすべて返す。
- metrics extraction logic は同じ考え方を使い、CSV は parsed summary、internal は result summary を source にする。
- 本格比較画面、AI summary 比較、artifact diff は reports endpoint の初回責務に含めない。

## 11. existing list endpoint との関係

既存 endpoint:

```text
GET /api/symbols/:symbolId/strategy-applications
```

維持する役割:

- SymbolDetail の saved application list 用。
- application summary、latest run、latest backtest report、`latest_reports_by_source` を返す。
- `report_presence` / `report_source` は application list の絞り込みとして維持する。
- `run_type` / `run_status` は latest run 基準として維持する。
- 同一 application の report 一覧を本格的に返す endpoint にはしない。

reports endpoint に寄せるもの:

- 同一 application 配下の report list。
- source 別 report pagination。
- metrics preview 付き report comparison entry。
- BacktestDetail link summary の一覧化。

## 12. DB / Prisma schema 判断

初回の `GET /api/symbol-strategy-applications/:applicationId/reports` は、既存 schema で実現可能と判断する。

利用する既存 relation:

- `SymbolStrategyApplication.runs`
- `SymbolStrategyApplicationRun.backtest`
- `SymbolStrategyApplicationRun.backtestImport`
- `SymbolStrategyApplicationRun.internalBacktestExecution`
- `Backtest.imports`
- `Backtest.strategySnapshotJson`
- `SymbolStrategyApplication.symbol`
- `SymbolStrategyApplication.strategyRule`
- `SymbolStrategyApplication.strategyRuleVersion`

今回の判断:

- DB migration は不要。
- Prisma schema change は不要。
- metrics normalization table は不要。
- comparison entity は不要。
- 初回検索は `applicationId + createdAt` / `applicationId + status` など既存 index の範囲で開始できる。
- 大量データ時のみ、`applicationId + backtestId + updatedAt` 相当の index、read model、metrics cache を後続判断にする。

## 13. error / validation 方針候補

- application が存在しない場合は `404 NOT_FOUND`。
- `execution_source` の invalid value は `400 VALIDATION_ERROR`。
- `run_type` の invalid value は `400 VALIDATION_ERROR`。
- `with_metrics` が `true` / `false` 以外の場合は `400 VALIDATION_ERROR`。
- `page` / `limit` が positive integer でない場合は `400 VALIDATION_ERROR`。
- `limit` は上限超過時に `400 VALIDATION_ERROR`。
- `sort` / `order` の invalid value は `400 VALIDATION_ERROR`。
- `status` は Backtest status として扱うため、allowed value を固定するか existing Backtest status string を許容するかは実装直前に決める。

## 14. 最小実装候補

次に実装へ進む場合は、候補を以下 4 つに絞る。

1. `GET /api/symbol-strategy-applications/:applicationId/reports` の read-only backend API を追加し、Backtest が linked された runs だけを pagination 付きで返す。
2. `execution_source` / `run_type` / `status` / `with_metrics` / `page` / `limit` / `sort` / `order` の validation と pagination meta を実装する。
3. response に application summary、reports[]、linked run summary、metrics summary、importless report marker、BacktestDetail link summary を返す。
4. 初回 frontend は新規画面を作らず、必要なら `SymbolDetail` または `BacktestDetail` から manual API 確認 / 小さな導線追加に留める。

やらないこと:

- 既存 `GET /api/symbols/:symbolId/strategy-applications` の response meaning 変更。
- latest run filter を any run filter に変えること。
- runs endpoint との同時実装。
- DB migration。
- Prisma schema change。
- metrics normalization table。
- comparison entity。
- new report comparison screen。
- AI summary 自動比較。
- artifact diff / download。
- Visual regression。

## 15. 確認観点

docs-only の今回確認:

- reports endpoint は `GET /api/symbol-strategy-applications/:applicationId/reports` を第一候補としている。
- runs endpoint は実行履歴が主語、reports endpoint は検証 report 一覧 / 比較 UX が主語として分離されている。
- CSV import report と internal backtest report を source / metrics / linked run 観点でどう返すかが分かる。
- internal backtest 由来 report が importless report であることが明示されている。
- `BacktestDetail.related_reports` / metrics comparison との関係が分かる。
- DB / Prisma schema 変更が初回不要である判断が明記されている。
- 次の最小実装候補が 2〜4 個に絞られている。

実装に進む場合の確認:

- `execution_source=tradingview` で CSV import report だけを返す。
- `execution_source=internal_backtest` で internal backtest report だけを返す。
- `run_type=csv_import` と `run_type=internal_backtest` が linked run 基準で効く。
- `with_metrics=false` で metrics summary を省略できる。
- CSV import report では metrics source が `BacktestImport.parsedSummaryJson` である。
- internal backtest report では metrics source が `Backtest.strategySnapshotJson.result_summary` である。
- internal backtest report では `importless_report = true` である。
- BacktestDetail link summary から個別 report detail へ遷移できる。
