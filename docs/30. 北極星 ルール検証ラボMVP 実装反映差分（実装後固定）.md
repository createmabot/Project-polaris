## 4. 現在のユーザーフロー

### 4-1. 新規作成フロー

1. `/strategy-lab` にて自然言語ルールを入力する。
    
2. strategy を作成する。
    
3. strategy version を作成する。
    
4. Pine を生成する。
    
5. TradingView 上で一次検証を行う。
    
6. backtest を作成する。
    
7. CSV を取り込む。
    
8. backtest 詳細画面で parse 結果および summary を確認する。
    

### 4-2. version 再利用フロー

1. `/strategies/:strategyId/versions` にて version 一覧を確認する。
    
2. 対象 version を `/strategy-versions/:versionId` で開く。
    
3. 既存 version を再生成する、または clone により新 version を作成する。
    
4. 派生 version を軽微編集する。
    
5. Pine を再生成する。
    
6. clone 元との差分を確認する。
    

### 4-3. 履歴振り返りフロー

1. `/backtests` で過去の検証履歴を探す。
    
2. `/backtests/:backtestId` を開く。
    
3. parse 状態、summary、import 履歴を確認する。
    
4. 「次アクション（Rule Lab）」から、当該検証に関連する Strategy Version の見直し導線へ戻る。
    

※ Backtest Detail には「使用した Strategy」セクションを実装済みとし、実行時 snapshot と current の導線を分離して参照できる。

- `GET /api/strategy-versions/:versionId`
    
- `PATCH /api/strategy-versions/:versionId`
    
- `POST /api/strategy-versions/:versionId/pine/generate`
    
- `POST /api/strategy-versions/:versionId/clone`
    

### 5-2. Backtest 系 API

- `POST /api/backtests`
    
- `POST /api/backtests/:backtestId/imports`
    
- `GET /api/backtests`
  - query: `q`, `page`, `limit`, `status`, `sort(created_at|updated_at)`, `order(asc|desc)`
    
- `GET /api/backtests/:backtestId`
    
  - `used_strategy` と実行時 `snapshot` を返却し、当時の strategy 情報と current 導線を分離して参照可能
    
- `GET /api/backtests/:backtestId/imports`
    
### 5-3. 内製バックテスト土台 API（最小）

- `POST /api/internal-backtests/executions`
  - execution job を `queued` で作成し、queue enqueue まで行う（最小土台）
  - execution input snapshot の最小検証（`strategy_rule_version_id` と `data_range` は必須、`market/timeframe` は optional）を実施
  - `engine_estimated` 利用時は `execution_target.symbol` を必須とし、実データ取得識別子として扱う
  - `market/timeframe` は request 値を優先し、未指定時は strategy version の値で補完する
- `GET /api/internal-backtests/executions/:executionId`
  - execution status の source of truth を返却
- `GET /api/internal-backtests/executions/:executionId/result`
  - `succeeded` 以外は `RESULT_NOT_READY` を返却（最小ガード）
- `GET /api/internal-backtests/executions/:executionId/artifacts/engine_actual/trades-and-equity`
  - `engine_actual` 実行結果の最小 artifact（`trades` / `equity_curve`）を返却
  - 実行状態が `succeeded` 以外は `RESULT_NOT_READY` を返却
- `GET /api/internal-backtests/observability/data-source-unavailable-summary?window=24h|7d`
  - 内部運用確認向けの最小集計（reason code 別件数・直近失敗 execution）を返却
  - 集計 source of truth は `internal_backtest_data_source_failure_events`（DB永続化イベント）
  - selective retry 効果観測の最小集計（`retry_effect`）を返却
    - `retry_targeted_count`
    - `retry_attempted_count`
    - `retried_and_succeeded_count`
    - `retried_and_failed_count`
    - `not_retried_failed_count`
    - `by_reason`（reason code 別の retry outcome 内訳）
  - outward 契約（consumer 向け `DATA_SOURCE_UNAVAILABLE` 単一コード）は不変

運用時の最小切り分け順（固定）:

1. summary API を `window=24h` で確認し、`total_failures` と `by_reason` の上位要因を把握する。
2. 必要に応じて `window=7d` で反復発生（傾向）を確認する。
3. `recent_failures` から `execution_id` を取得し、`GET /api/internal-backtests/executions/:executionId` で失敗ステータスと `error_code=DATA_SOURCE_UNAVAILABLE` を確認する。
4. 同一レコードの `provider_name / internal_reason_code / symbol / market / timeframe / from / to / http_status / endpoint_kind` で原因切り分けを行う。

reason code の最小確認観点（固定）:

- `provider_http_error`: `http_status` と `endpoint_kind` を優先確認
- `provider_timeout`: 時間帯集中と連続発生の有無を確認
- `provider_network_error`: 一時通信障害か継続障害かを確認
- `provider_invalid_response`: provider 応答 shape 崩れを確認
- `provider_parse_error`: parser 前提と応答フォーマット差分を確認
- `provider_not_configured`: 実行環境の provider 設定を確認
- `provider_unsupported_target`: `execution_target` と `market/timeframe` 組み合わせを確認
  
worker 骨組み（BullMQ）により、`queued -> running -> succeeded|failed` の最小状態遷移が成立している。
`result` は `result_summary.schema_version = "1.0"` の最小スキーマで保存・返却する。
`result_summary.summary_kind` は `scaffold_deterministic` / `engine_estimated` / `engine_actual` を併存させる（`engine_actual` は初回最小スライスのみ導入）。
`artifact_pointer` も `type` / `execution_id` / `path` の最小スキーマを runtime 検証して保存・返却する。
worker は status 遷移と永続化に責務を絞り、実行処理本体は service / adapter 境界へ分離する。

`engine_estimated` は最小実データ adapter（real provider 差し替え点）を経由し、再現性 metadata を `input_snapshot.data_source_snapshot` として保存・返却する。
この実装は `daily_ohlcv / JP_STOCK / D` のみを対象とし、それ以外は `DATA_SOURCE_UNAVAILABLE` で失敗する。
adapter の内部は provider 境界を持ち、provider 応答を normalize したうえで `data_source_snapshot` を組み立てる。

- 最小対象データ: 日足 OHLCV
- 最小対象 market/timeframe: `JP_STOCK` / `D`
- 実データ取得責務: data source adapter に限定（route/worker は取得責務を持たない）
- provider責務: 市場データ取得（最小 real provider 実装 + stub test double）
- provider選択: `INTERNAL_BACKTEST_MARKET_DATA_PROVIDER` で切替可能（未指定時は `test=stub`, `development/production=stooq`）
- adapter責務: provider 応答 normalize・snapshot 生成・provider error を `DATA_SOURCE_UNAVAILABLE` へ統一マッピング
- 取得失敗時: execution を `failed` にし、`errorCode = DATA_SOURCE_UNAVAILABLE` を保持
- outward 契約は維持し、consumer へは引き続き `DATA_SOURCE_UNAVAILABLE` のみを返す
- 内部観測性として、provider 失敗は `provider_http_error` / `provider_timeout` / `provider_network_error` / `provider_invalid_response` / `provider_parse_error` / `provider_not_configured` / `provider_unsupported_target` の reason code で分類し、worker 失敗ログに execution context（execution_id, symbol, market, timeframe, from, to, elapsed_ms など）を付与する
- selective retry（最小）を provider fetch 境界へ導入済み:
  - 対象: `provider_timeout` / `provider_network_error` / `provider_http_error` のうち `http_status` が 5xx の場合
  - 非対象: `provider_parse_error` / `provider_invalid_response` / `provider_not_configured` / `provider_unsupported_target` / `provider_http_error` の 4xx
  - `provider_http_error(429)` は現時点観測（24h/7d）で有意な改善根拠がないため、4xx 非対象のまま維持
  - 回数: 追加 1 回（最大 2 試行）
  - outward 契約は不変（最終失敗時は `DATA_SOURCE_UNAVAILABLE` へ収束）
- 再現性 metadata: `data_source_snapshot`（`source_kind`, `market`, `timeframe`, `from`, `to`, `fetched_at`, `data_revision`, `bar_count`）
- empty bars 契約: `data_source_snapshot.bar_count = 0` は `engine_estimated` の `failed` 条件ではなく、`succeeded` + zero metrics（価格要約0埋め）として扱う
- execution target: `input_snapshot.execution_target`（`symbol`, `source_kind`）
- execution target canonical (JP_STOCK): `symbol` は `7203` 形式を正とし、`TYO:7203` は受理後に `7203` へ正規化
- execution target validation (JP_STOCK): 前後空白は trim、4桁数字以外は `INVALID_EXECUTION_TARGET`
- consumer 判定規約（固定）:
  - 成否は `status`（status API）を正本として判定
  - 0件成功は `status = succeeded` かつ `summary_kind = engine_estimated` かつ `result_summary.metrics.bar_count = 0`
  - 取得失敗は `status = failed` かつ `errorCode = DATA_SOURCE_UNAVAILABLE`
  - `data_source_snapshot.bar_count` は再現性確認用途（summary 側判定の補助）
  - 判定カテゴリは `success_with_data` / `success_no_data` / `data_source_unavailable` / `internal_failure` / `not_ready` で共有化
  - `summary_kind = scaffold_deterministic` は現時点では `success_with_data` と同カテゴリ扱い
  - view-model helper で `state_label` / `is_error` / `is_empty` / `can_show_metrics` / `recommended_message_key` へ変換する最小層を追加済み
  - Rule Lab 側の最小接続として `StrategyVersionDetail` に internal-backtests 表示領域を追加済み（status/result 取得 + helper 経由の4パターン表示）
  - `engine_actual` では artifact read API（`/api/internal-backtests/executions/:executionId/artifacts/engine_actual/trades-and-equity`）を read-only 接続し、`trades / equity_curve` を最小表示
  - artifact 表示状態は `success(with trades)` / `success(no-trade)` / `RESULT_NOT_READY` / `NOT_FOUND` / `fetch error` を分離して扱う

`engine_estimated` の `result_summary.metrics` は、上記 adapter の normalized bars から算出した deterministic な価格要約（`bar_count`, `first_close`, `last_close`, `price_change`, `price_change_percent`, `period_high`, `period_low`, `range_percent`）として返却される（本格シミュレーションではない）。
実データ取得識別子は `strategy_rule_version_id` とは分離し、`execution_target.symbol` を正とする。

段階差（固定）:

- `scaffold_deterministic`: 入力条件ベースの決定論的足場値
- `engine_estimated`: 実データ接続を伴う推定段階（ただし本格シミュレーションではない）
- `engine_actual`: 売買シミュレーション / コスト反映 / 再現可能レポートまで含む本段階（初回最小スライス実装済み）

## 7. Strategy Version の現仕様

### 7-1. version 一覧

version 一覧では、少なくとも以下の情報を確認可能とする。

- 派生 version かどうか
    
- clone 元との差分があるかどうか
    
- 要確認差分バッジ（`派生` かつ `差分あり` の最小強調）
- フォワード検証ノート有無（`検証ノートあり` の最小表示）
- `要確認差分` かつ `検証ノートあり` の複合シグナルを `最優先確認` として最小強調表示
- `検証ノートあり` の行でノート更新目安を最小表示（`forwardValidationNoteUpdatedAt` 基準）
- `forwardValidationNoteUpdatedAt` がページ内で最も新しい行を `最新ノート` として最小表示し、先読み対象を判別しやすくする
- `最優先確認` かつ `最新ノート` の行を `今読む候補` として最小表示し、優先読了対象を即判別可能にする
    
- status
    
- market / timeframe
    
- warnings の有無
    
- 作成日時
    

### 7-2. version 詳細

version 詳細では、少なくとも以下の情報を確認可能とする。

- natural language rule
    
- generated Pine
    
- warnings
    
- assumptions
    
- status
    
- market / timeframe
    
- strategy id / version id
    
- createdAt / updatedAt
- フォワード検証ノートの更新目安（`forwardValidationNoteUpdatedAt`。ノート未設定時は `-`）
- 比較サマリの優先確認ポイント（ルール文 / Pine / status の確認順）
- 比較サマリ本体での判断順表示（naturalLanguageRule / Pine / status / updatedAt を変更有無つきで提示）
- Pine 差分抜粋の区分表示（変更 / 追加 / 削除）
- Pine 差分抜粋の確認順ガイド（変更 → 追加 → 削除）
    

### 7-3. clone の現仕様

- 元 version は更新しない。
    
- 同一 strategy 配下に新 version を作成する。
    
- clone 元参照を保持する。
    
- clone 後は新 version の detail 画面へ遷移する。
    

### 7-4. 軽微編集の現仕様

- 編集対象の主対象は `naturalLanguageRule` とする。
    
- 保存時は再生成前提の状態へ戻す。
    
- 具体的には以下の状態へ更新する。
    
    - `generatedPine = null`
        
    - `normalizedRuleJson = null`
        
    - `warnings / assumptions` はクリア
        
    - `status = draft`
        

### 7-5. 再生成の現仕様

- 同一 version を更新対象とする。
    
- `generatedPine`、`warnings`、`assumptions`、`status` を再計算する。

## 8. 差分表示の現仕様

### 8-1. 比較対象

- 比較対象は clone 元 version のみとする。
    
- `clonedFromVersionId` を正とする。
    

### 8-2. 表示項目

以下を差分表示対象とする。

- naturalLanguageRule 差分
    
- status 比較
    
- updatedAt 比較
    
- Pine の変更あり / なし
    
- Pine の行差分 / 文字差分
    
- Pine 差分抜粋（先頭 N 件）
    

### 8-3. 比較元が無い場合

- 「比較元の version はありません」と表示する。
    
- 画面は壊さず、通常操作を継続可能とする。

- 要確認差分の件数表示（ページ内）と最小強調表示
- フォワード検証ノートの有無を最小表示（比較後の次検証方針の有無を一覧段階で判別）
- `要確認差分` / `検証ノートあり` / `要確認差分かつ検証ノートあり` のページ内件数表示
- `最新ノート`（ページ内で最も新しいノート更新）の件数表示と行バッジ表示
- `最優先確認` かつ `最新ノート` の件数表示と行バッジ表示（`今読む候補`）
- `今読む候補` がある場合、件数サマリからページ内先頭の対象へジャンプ可能
- `今読む候補` が複数件ある場合、件数サマリから `次の今読む候補へ` で順送り確認が可能
- `要確認差分かつ検証ノートあり` がある場合、件数サマリからページ内先頭の `最優先確認` カードへジャンプ可能
- `最優先確認` が複数件ある場合、件数サマリから `次の最優先確認へ` で順送り確認が可能
- ジャンプ直後は対象カードを一時ハイライトし、見失いにくさを最小補助（自動解除）
- `StrategyVersionDetail` でも、同一一覧文脈に `最優先確認` が複数件ある場合は `次の最優先確認へ` で順送り確認が可能

### 11-5. internal-backtests observability の週次確認（固定）

- 週次確認の最小対象:
  - `total_failures`（`DATA_SOURCE_UNAVAILABLE` 総件数）
  - `by_reason`（reason code 別件数）
  - `retry_effect`
    - `retry_targeted_count`
    - `retry_attempted_count`
    - `retried_and_succeeded_count`
    - `retried_and_failed_count`
    - `not_retried_failed_count`
  - `recent_failures`（直近失敗 execution）
- 確認 window:
  - `24h`: 当週の直近状態（一次判断）
  - `7d`: 反復傾向（再判定判断）
- 429（`provider_http_error`）の扱い:
  - 現在は retry 非対象を維持する。
  - 次の再判定条件を全て満たした場合のみ、retry 対象化を検討する。
    1. `window=7d` で 429 件数が `>= 5`
    2. 429 件数が `provider_http_error` 全体の `>= 20%`
    3. 429 発生日が週内 `>= 3日`
  - 上記未達時は「非対象維持」を継続し、理由を週次記録へ残す。

### 11-6. ローカル Prisma migration drift 運用（固定）

- 正本は repo の `backend/prisma/schema.prisma` と `backend/prisma/migrations/*` とする。
- drift 切り分けは次の順で実施する。
  1. `npx prisma validate`
  2. `npx prisma migrate status`
  3. `npx prisma migrate dev`
  4. `npx prisma migrate deploy`
- `The migration \"...\" was modified after it was applied`（checksum 不一致）が出た場合は、`migrate resolve` を先に使わず、ローカル開発 DB を再生成して整合回復する。
  - `npx prisma migrate reset --force --skip-seed`
  - その後に `status -> dev -> deploy` を再実行して再現性を確認する。
- 本運用はローカル開発基盤の再現性回復を目的とし、public API / UI 契約には影響しない。
- Prisma 7 系移行として、設定正本は `backend/prisma.config.ts` に統一し、`package.json#prisma` は利用しない。
