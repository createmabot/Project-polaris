# 北極星 snapshot stale・market_status unknown整理（MVP後P2）

更新日: 2026-05-08

## 1. 目的

- snapshot stale / `market_status = unknown` の扱いを整理する
- reason code の意味と運用上の見方を固定する
- Home / SymbolDetail / Comparison の期待値を整理する

## 2. 前提

- snapshot stale / unknown は原則 failure ではない
- 価格情報が取得できる場合は部分成立として扱う
- snapshot が取得できない場合も、主要導線全体を落とさない
- API response shape は原則維持する
- DB 構造は変更しない

## 3. 現行実装の概要

現行実装の正本は `backend/src/market/snapshot.ts` である。

構成は大きく次のとおり。

1. primary source: `stooq_daily`
2. secondary source: `yahoo_chart`
3. freshness 評価
4. JP market session 判定
5. reason metric 記録
6. threshold warn

### 3.1 primary / secondary source

- まず `stooq_daily` を試行する
- 失敗時のみ `yahoo_chart` に failover する
- 両方失敗した場合は `current_snapshot = null` を返す

### 3.2 freshness 評価

`evaluateFreshness()` は snapshot の `as_of` を基準に、次を判定する。

- `fresh`
- `stale`
- `expired`
- `invalid`

現行しきい値:

- intraday stale: 30 分
- daily stale warning: 24 時間
- expired: 7 日

### 3.3 market status の折り畳み

`foldMarketStatus()` の現行ルール:

- candidate が `unknown` なら最終 `market_status` も `unknown`
- freshness が `invalid` / `expired` なら最終 `market_status` は `unknown`
- candidate が `open` かつ freshness が `stale` なら最終 `market_status` は `unknown`
- candidate が `closed` の場合は stale でも `closed` を維持する

### 3.4 JP market session 判定

Yahoo 側で `REGULAR` などが返っても、日本株で次の条件に当てはまる場合は `closed` 寄りに補正する。

- `jp_market_weekend`
- `jp_market_holiday`
- `outside_jp_session`

### 3.5 metric / warn

snapshot 評価では次の log / metric を使う。

- `current_snapshot_status_evaluated`
- `snapshotReasonDailyMetric`
- `snapshot_reason_threshold_exceeded`
- `current_snapshot_primary_failed`
- `current_snapshot_secondary_attempt`
- `current_snapshot_secondary_succeeded`
- `current_snapshot_secondary_failed`
- `current_snapshot_all_sources_failed`

## 4. reason code 一覧

| reason_code | 意味 | market_status | 運用上の見方 | 対応方針 |
|---|---|---|---|---|
| `daily_source_closed` | Stooq daily は日次終値ソースであり、市場 open 判定に使わず `closed` として扱う | `closed` | 異常ではない。日次系 source の通常挙動 | そのまま許容 |
| `candidate_unknown` | Yahoo `marketState` から open/closed を確定できず、同日・同セッション内でも判定不能 | `unknown` | provider 側の state 解釈確認対象 | log / metric を確認し、継続多発時に provider 値を再確認 |
| `freshness_invalid` | `as_of` が未来寄りなど不正で freshness 判定不能 | `unknown` | 異常寄り。timestamp 解釈または source 値確認対象 | warn 対象 |
| `freshness_expired` | `as_of` が古すぎて現況として使えない | `unknown` | stale より重い。取得停止・ provider 停滞の確認対象 | warn 対象 |
| `open_but_stale` | source 上は open 候補だが、更新時刻が stale | `unknown` | 市場 open 中の更新停滞。運用確認対象 | warn / threshold 監視対象 |
| `jp_market_holiday` | 日本市場の休場日により closed 扱いへ補正 | `closed` | 異常ではない。営業日カレンダー由来 | そのまま許容 |
| `jp_market_weekend` | 日本市場の週末により closed 扱いへ補正 | `closed` | 異常ではない。週末由来 | そのまま許容 |
| `outside_jp_session` | 日本市場の時間外であり closed 扱いへ補正 | `closed` | 異常ではない。時間外由来 | そのまま許容 |
| `closed_by_source` | source 側が closed を返したため closed として採用 | `closed` | 異常ではない。source 判定を尊重 | そのまま許容 |

## 5. Home / SymbolDetail / Comparison での扱い

### 5.1 Home `market_overview`

- `market_overview.indices / fx / sectors` は価格系 view model を返す
- `market_status` や `reason_code` は現行 API では返さない
- `sectors` は partial success を許容し、一部欠損でも返す
- 全欠損時は `sectors: []`

### 5.2 Home `watchlist_symbols`

- row は snapshot の有無に関係なく返す
- snapshot がある場合:
  - `latest_price`
  - `change_rate`
  が埋まる
- snapshot がない場合:
  - `latest_price = null`
  - `change_rate = null`
- frontend は `-` 表示にする
- stale / unknown を理由に row を落とさない

### 5.3 Home `positions`

- row は snapshot の有無に関係なく返す
- snapshot がある場合:
  - `latest_price`
  - `unrealized_pnl`
  が埋まる
- snapshot がない場合:
  - `latest_price = null`
  - `unrealized_pnl = null`
- frontend は `-` 表示にする

### 5.4 SymbolDetail `snapshot`

- `current_snapshot` が取得できれば `market_status` と `as_of` を表示する
- `current_snapshot = null` の場合は
  - `スナップショットを取得できませんでした。`
  を表示する
- stale / unknown 専用の追加文言は現行 UI では未実装

### 5.5 ComparisonDetail `snapshot`

- 各 symbol の `current_snapshot` があれば価格 / 変化率を使う
- snapshot がなければ `-` 表示
- 現行 UI では `market_status` 自体は出さない

## 6. failure / partial success の切り分け

運用上は次のように分ける。

### 6.1 failure

- primary / secondary の両方が失敗し `current_snapshot = null`
- 価格情報が取得できない

ただし API 全体を 500 にせず、row や画面自体は成立させる。

### 6.2 partial success

- snapshot は取得できるが `market_status = unknown`
- freshness が `stale`
- `reason_code = open_but_stale`
- `reason_code = candidate_unknown`
- `reason_code = freshness_expired`
- `reason_code = freshness_invalid`

この場合は「異常で即 failure」ではなく、価格は使いつつ運用確認対象とする。

### 6.3 normal closed state

次は異常ではなく、closed の説明理由として扱う。

- `daily_source_closed`
- `jp_market_holiday`
- `jp_market_weekend`
- `outside_jp_session`
- `closed_by_source`

## 7. metric / log の確認

確認先:

- `current_snapshot_status_evaluated`
- `snapshotReasonDailyMetric`
- `snapshot_reason_threshold_exceeded`
- `current_snapshot_primary_failed`
- `current_snapshot_secondary_succeeded`
- `current_snapshot_all_sources_failed`

見方:

- `open_but_stale` がしきい値超過:
  - intraday source の停滞確認対象
- `freshness_invalid` が増加:
  - timestamp 解釈不整合の確認対象
- `freshness_expired` が増加:
  - source 停止または取得失敗の確認対象
- `candidate_unknown` が増加:
  - provider 側 `marketState` の解釈確認対象

## 8. 今回の判定

### 8.1 現行実装で足りている部分

- stale / unknown を failure ではなく部分成立として扱う実装がある
- snapshot `null` でも Home row を落とさない
- SymbolDetail も 200 を維持し `current_snapshot = null` を返す
- log / metric / threshold warn の経路がある
- 既存 test で reason code と failover の主要挙動が固定されている

### 8.2 追加修正が必要な部分

- 今回は docs 整理のみ
- API shape / DB / snapshot provider 本体は変更しない

### 8.3 後続タスクに回す部分

- reason code の API 返却可否
- market calendar 更新運用
- 非日本銘柄の market status 本格対応
- snapshot provider 冗長化
- browser-based E2E
