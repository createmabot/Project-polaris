# 北極星 docs文字化け棚卸し範囲整理（MVP後P1）

更新日: 2026-05-08

## 1. 目的
- docs 全体の文字化け棚卸し対象を決める。
- 修復優先度を分類する。
- 旧 docs の扱いと、推測修復しない方針を明確にする。

## 2. 前提
- MVP は完了済みであり、現行運用の runbook を優先する。
- 全 docs を同一優先度で一括修復しない。
- 本文推測による復元は行わない。
- secret / token / shared_secret / API key の実値は扱わない。
- 文字化け棚卸しでは、本文が読めるか、現行運用に使うか、正本か、を分けて判断する。

## 3. 分類基準
### A. 現行runbook / 最優先修復対象
条件:
- 現在の開発・運用で頻繁に使う。
- MVP 後も継続参照する。
- 新しい作業者が最初に読む。

対応方針:
- 文字化けがあれば優先修復する。
- secret 非露出も同時確認する。
- P1 または P2 先頭の修復対象に置く。

### B. 現行仕様の正本 / 高優先修復対象
条件:
- 現行実装判断に使う。
- API / 画面 / データモデル / Rule Lab の正本として参照される。
- 文字化けしていると今後の実装判断に影響する。

対応方針:
- 現行実装に効く章から優先修復する。
- 新 docs に正本が移った箇所は、旧本文全面修復より参照先明記を優先してよい。

### C. 履歴資料 / 低優先または修復対象外候補
条件:
- 旧スプリント資料。
- すでに新しい docs に内容が吸収済み。
- 現行運用で直接参照しない。
- 文字化けしていても実務影響が小さい。

対応方針:
- 原則として一括修復しない。
- 必要なら「履歴資料」「参照注意」として扱う。
- 本文推測修復は避ける。

### D. 判断保留
条件:
- 文字化けがあるが現行影響が判断しにくい。
- 一部だけ現行仕様として使われている。
- 参照頻度が不明。

対応方針:
- 判断理由を記録する。
- 次タスクで個別確認する。

- `docs/34.北極星 TradingView CSV import 運用手順（MVP）.md`

- `docs/36.北極星 MVP受入確認チェックリスト（MVP）.md`

- `docs/41.北極星 Rule Lab日本語入力 UI経路再確認メモ（MVP後P1）.md`

確認結果:

- `docs/33` も runbook として重要だが、現ファイルは本文の大半が文字化けしている。

判断:
- `docs/32`, `docs/33` は A の中でも最優先修復対象。
- その他 A docs は現時点では可読であり、修復対象ではなく維持確認対象とする。

判断:
- `docs/29`, `docs/30`, `docs/25` を B の優先修復候補とする。
- `docs/16`, `docs/17`, `docs/3`, `docs/8` は現時点で利用可能なため、修復より維持と差分確認を優先する。

## 7. C. 履歴資料 / 低優先または修復対象外候補
対象候補:
- `docs/15`〜`docs/28` の旧計画・補助資料群
- `docs/webhook.md`
- `docs/webhook-refactor-report.md`
- `docs/webhook-final-compliance-report.md`
- `docs/snapshot-weekly-reviews/*`

確認結果:
- 現行運用の正本は `docs/32`〜`docs/41` に移っており、これらは履歴・補助資料の性格が強い。
- 本文を推測修復しても、現行仕様判断への寄与が薄い。

判断:
- 原則後回し。
- 必要になった docs だけ個別修復、または「履歴資料 / 参照注意」注記で十分。

## 8. D. 判断保留
対象候補:
- `docs/31.北極星 APIレスポンス差分棚卸しと次タスク候補.md`
- `docs/24.北極星 開発着手用 README セットアップ手順書（MVP）.md`
- `docs/14.北極星 TradingView 設定手順書（MVP）.md`
- `docs/7.北極星 TradingView alert message テンプレート設計（MVP）.md`
- `docs/9`〜`docs/13`

理由:
- 一部は今後も参照しうるが、実際の参照頻度が runbook 群ほど高くない。
- 文字化けの有無だけでなく、内容が新 docs に吸収済みかの判断が必要。

次の扱い:
- P2 で個別に開いて、A/B/C のどこへ寄せるかを判断する。

## 9. 修復方針
- 文字化けが軽微な場合は直接修正する。
- 本文推測が必要な場合は修復しない。
- 新しい docs が正本になっている場合は、旧 docs 修復より参照先明記を優先する。
- 主要 runbook の文字化けは優先修復する。
- 修復時も secret 非露出確認を必須にする。
- UTF-8 BOM の有無だけを問題化せず、実際に可読性と運用影響で判断する。

## 10. 次タスク候補
1. A 分類 docs のうち `docs/32`, `docs/33` の軽微修復または正常化。
2. B 分類 docs のうち `docs/25`, `docs/29`, `docs/30` で、現行実装判断に必要な章だけを修復。
3. C 分類 docs に `履歴資料 / 参照注意` 注記を付けるか検討する。
4. docs 全体の文字コード / BOM / 保存方針を整理する。

## 11. Codex修復依頼テンプレート
以下をベースに、個別 docs の修復を依頼する。

```text
北極星プロジェクトの docs 修復タスクとして、<対象docs> の文字化け修復可否を確認してください。

前提:
- docs 配下を正本として扱う
- 推測による本文復元はしない
- secret / token / shared_secret / API key の実値は残さない
- まず現行 runbook / 正本 / 履歴資料のどれかを判断する

やってほしいこと:
1. 対象 docs の文字化け範囲を確認する
2. 現行運用・正本性・参照頻度を確認する
3. 軽微修復で済むか、推測修復が必要かを判定する
4. 修復する場合は最小差分で行う
5. 修復しない場合は、参照先 docs への誘導や注記で代替できるかを提案する

最終報告に必ず含めること:
- 対象 docs
- 文字化け範囲
- 正本性の判定
- 修復可否
- 修復しない場合の理由
- secret 非露出確認
```
