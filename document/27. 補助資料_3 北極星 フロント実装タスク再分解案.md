# 北極星 フロント実装タスク再分解案

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、北極星MVPのフロントエンド実装を、**実際に着手・管理・レビューしやすいタスク単位** に再分解するための補助資料である。

既存資料では以下が整理済みである。

- 企画方針
    
- 画面別入出力サンプル
    
- ホームデータ供給仕様
    
- コンポーネント分割案のMVP範囲整理
    

本書ではそれらを前提に、フロント実装を以下の観点で再分解する。

- 依存関係を踏まえた実装順
    
- GitHub Issue / Project に落としやすい粒度
    
- Page / Section / Domain Component / Hook / ViewModel ごとの責務
    
- MVPで先に作るものと後回しにするもの
    
- 各タスクの完了条件
    

---

## 2. 再分解の基本方針

### 2.1 画面ではなく「成立導線」で切る

MVPでは以下の導線を優先する。

1. webhook → 要約 → ホーム
    
2. 銘柄詳細 → 論点 → ノート
    
3. 自然言語 → Pine → CSV → 総評
    

そのため、フロント実装も「ページ単位で一気に完成させる」のではなく、**導線を通すために必要な順** に切る。

### 2.2 UI基盤とデータ境界を先に固定する

後から全画面に波及しやすいものは最初に固める。

- app shell
    
- routing
    
- API client
    
- error/loading/empty handling
    
- view model mapper
    
- 共通 primitive UI
    

### 2.3 Section完結を基本単位にする

Issue粒度は、原則として **1 Section = 1実装単位** を基本とする。  
ただし以下は例外。

- 複数Sectionから参照される基盤部品
    
- Hook / ViewModel などのデータ境界実装
    
- 単独Sectionでは意味を持たない比較inlineブロック
    

### 2.4 完了条件をUI表示だけにしない

「見た目が出た」で完了にせず、以下を含めてDoneとする。

- API接続またはmock接続
    
- loading / empty / error の表示
    
- 型定義
    
- Storyまたは画面確認手順
    
- 最低限のテスト

## 8. Definition of Done

全タスク共通のDone条件を以下とする。

### 必須

- 型定義がある
    
- loading / empty / error のいずれかを考慮している
    
- 主要UIが確認できる
    
- mock か実APIで最低限動く
    
- 手動確認手順が残っている
    

### 望ましい

- Storybook または同等のUI確認導線がある
    
- 最低1件のテストがある
    
- 今後の拡張余地がコメントで残っている

## 11. 結論

北極星MVPのフロント実装は、  
**基盤 → Home → Symbol/Note → Strategy → Backtest → Comparison → Polish**  
の順で再分解するのが最も実装効率が高い。

また、タスク粒度は以下に揃える。

- 基盤は機能群単位
    
- 画面は Page shell + Section 単位
    
- 複数画面共通要素は Domain Component 単位
    
- data binding は Hook / ViewModel 単位
    

この分解により、企画意図・MVPスコープ・コンポーネント分割案を崩さず、  
実際のIssue管理とPRレビューに移しやすくする。
