# 北極星 AI出力 JSON schema 詳細設計（MVP）

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」において、AIが生成する各種出力の **JSON schema 詳細** を定義するものである。

北極星では、AI出力を単なる文章として扱わず、  
**UI表示・比較・再利用・検索・履歴管理** に使うプロダクトデータとして扱う。

そのため、MVP段階から AI出力について以下を明確にする必要がある。

- どの種類の AI出力が存在するか
    
- それぞれの構造化データにどのキーを持たせるか
    
- 必須 / 任意項目をどう分けるか
    
- 失敗時や情報不足時をどう表現するか
    
- バージョン管理をどう行うか
    

---

## 2. 対象範囲

MVPで対象とする AI出力は以下の 5 種類とする。

1. 日次サマリ
    
2. アラート起点サマリ
    
3. 銘柄論点サマリ
    
4. バックテスト総評
    
5. Pine 生成時の warnings / notes 補助出力
    

※ Pine Script 本体そのものは `pine_scripts.script_body` に保存するため、本ドキュメントでは主に `ai_summaries.structured_json` と、関連する補助 JSON 出力を対象にする。

---

## 3. 設計方針

### 3.1 文章と構造化データを分離する

AI出力は以下の二層で持つ。

- `body_markdown`: 人が読むための自然文
    
- `structured_json`: UIと内部処理のための構造化データ
    

### 3.2 スコープごとに schema を分ける

AI出力の用途が異なるため、単一の巨大 schema にはしない。

### 3.3 必須項目は少なく、意味のある最小単位に絞る

MVPでは壊れにくさを優先し、必須項目は最小限に抑える。

### 3.4 情報不足を表現可能にする

AIが十分な根拠を持たない場合でも、

- 空配列
    
- `confidence`
    
- `insufficient_context`  
    のような形で状態を表現できるようにする。
    

### 3.5 将来拡張のため schema version を持つ

各 JSON 出力には `schema_version` を持たせる。

---

## 4. 共通ラッパー設計

MVPでは、AI出力の `structured_json` は、種類ごとに中身が異なる一方で、最上位に共通メタ情報を持たせる。

## 4.1 共通ラッパー

```json
{
  "schema_name": "alert_reason_summary",
  "schema_version": "1.0",
  "confidence": "medium",
  "insufficient_context": false,
  "payload": {}
}
```

---

## 4.2 共通キー定義

### schema_name

出力種類を示す固定文字列。

候補:

- `daily_summary`
    
- `alert_reason_summary`
    
- `symbol_thesis_summary`
    
- `backtest_review_summary`
    
- `pine_generation_notes`
    

### schema_version

スキーマのバージョン。  
MVPではすべて `1.0` とする。

### confidence

AI出力の確からしさを 3 段階程度で保持する。

候補:

- `high`
    
- `medium`
    
- `low`
    

### insufficient_context

入力情報不足かどうか。

- `true`: 情報不足だが最低限の出力は行った
    
- `false`: 通常生成
    

### payload

種類ごとの本体。

---

## 5. 日次サマリ schema

## 5.1 用途

- ホーム画面の朝サマリ / 夜サマリ
    
- 市況や監視銘柄の全体把握
    

## 5.2 schema_name

`daily_summary`

## 5.3 payload 仕様

```json
{
  "highlights": [
    {
      "title": "半導体株が上昇",
      "summary": "関連セクター全体が強く、監視銘柄にも上昇が見られた。",
      "reason": "関連ニュースとセクター地合いの影響が考えられる。",
      "confidence": "medium",
      "reference_ids": ["ref_1", "ref_2"],
      "symbol_ids": ["sym_1"]
    }
  ],
  "watch_items": [
    "為替の継続性",
    "引け後決算"
  ],
  "focus_symbols": [
    {
      "symbol_id": "sym_1",
      "reason": "重要開示あり"
    }
  ],
  "market_context": {
    "tone": "risk_on",
    "summary": "指数と主要セクターは堅調。"
  }
}
```

---

## 5.4 必須項目

- `highlights`
    
- `watch_items`
    
- `focus_symbols`
    
- `market_context`
    

### 補足

空でもよいが、キー自体は存在させる。

---

## 5.5 highlights 要素仕様

### 必須

- `title`
    
- `summary`
    

### 任意

- `reason`
    
- `confidence`
    
- `reference_ids`
    
- `symbol_ids`
    

---

## 6. アラート起点サマリ schema

## 6.1 用途

- 「今日動いた理由」
    
- アラート詳細
    
- 銘柄詳細の最新イベント要約
    

## 6.2 schema_name

`alert_reason_summary`

## 6.3 payload 仕様

```json
{
  "what_happened": "25日移動平均線上抜けアラートが発火した。",
  "fact_points": [
    "当日出来高が増加している。",
    "同日付の適時開示がある。"
  ],
  "reason_hypotheses": [
    {
      "text": "開示内容が好感された可能性がある。",
      "confidence": "medium",
      "reference_ids": ["ref_10"]
    }
  ],
  "watch_points": [
    "翌営業日に上昇が継続するか",
    "開示内容の詳細確認"
  ],
  "next_actions": [
    "適時開示本文を確認する",
    "出来高の継続を確認する"
  ],
  "reference_ids": ["ref_10", "ref_11"]
}
```

---

## 6.4 必須項目

- `what_happened`
    
- `fact_points`
    
- `reason_hypotheses`
    
- `watch_points`
    
- `next_actions`
    
- `reference_ids`
    

---

## 6.5 設計上の重要点

### fact_points

事実ベースのみを書く。

### reason_hypotheses

推測や解釈を格納する。  
必ず fact と分離する。

### reference_ids

最上位にも持つ。  
個別仮説側にも持てる。

---

## 7.1 用途

- 銘柄詳細の AI論点カード
    
- 研究ノート補助
    

## 7.2 schema_name

## 7.3 payload 仕様

```json
    {
      "text": "業績見通しの改善が確認できる。",
      "reference_ids": ["ref_20"]
    }
  ],
  "bearish_points": [
    {
      "text": "バリュエーションはやや高め。",
      "reference_ids": []
    }
  ],
  "watch_kpis": [
    "営業利益率",
    "受注残高"
  ],
  "next_events": [

      "label": "次回決算",

  ],

    "主要KPIの減速"
  ],

}
```

---

## 7.4 必須項目

- `bullish_points`
    
- `bearish_points`
    
- `watch_kpis`
    
- `next_events`
    
- `invalidation_conditions`
    
- `overall_view`

## 7.5 点データの粒度

MVPでは、`bullish_points` と `bearish_points` は文字列配列でも動くが、将来の出典管理を考え、要素オブジェクト形式を標準とする。

## 9. Pine生成補助出力 schema

## 9.1 用途

- 自然言語→Pine変換時の補助情報
    
- warnings / assumptions / omitted 項目の構造化保存
    

## 9.2 保存先

これは `ai_summaries` ではなく、`pine_scripts.generation_note` または関連 JSON カラムで保持してもよい。  
ただし schema は統一しておく。

## 9.3 schema_name

`pine_generation_notes`

## 9.4 payload 仕様

```json
{
  "assumptions": [
    "RSI length は 14 を採用した。",
    "時間足はフォーム入力の D を採用した。"
  ],
  "warnings": [
    "『強い銘柄』という表現は未対応のため反映していない。"
  ],
  "omitted_expressions": [
    "強い銘柄だけ買う"
  ],
  "supported_conditions": [
    "close > sma(25)",
    "rsi(14) >= 50"
  ],
  "generation_summary": "主要条件は strategy として変換できたが、一部裁量表現は省略した。"
}
```

---

## 9.5 必須項目

- `assumptions`
    
- `warnings`
    
- `omitted_expressions`
    
- `supported_conditions`
    
- `generation_summary`
    

---

## 10. 列挙値設計

## 10.1 confidence

共通で以下を使う。

- `high`
    
- `medium`
    
- `low`
    

## 10.2 market_context.tone

日次サマリ用の簡易列挙値。

- `risk_on`
    
- `neutral`
    
- `risk_off`
    
- `mixed`
    

### 補足

MVPでは内部利用中心であり、厳密なマーケットレジーム定義までは不要。

---

## 11. null / 空配列 / 欠損の扱い

### 基本原則

- キー自体はできるだけ存在させる
    
- 情報が無い場合は `[]` か `null` を明示する
    
- キーごと欠落は避ける
    

### 推奨

- リスト項目: `[]`
    
- 単一説明文: 空文字ではなく `null` か短い説明文
    
- `overall_view` はなるべく空にしない
    

---

## 12. バリデーション方針

AI出力は保存前に schema バリデーションを行う。

## 12.1 共通バリデーション

- `schema_name` 必須
    
- `schema_version` 必須
    
- `confidence` は列挙値に一致
    
- `insufficient_context` は boolean
    
- `payload` は object
    

## 12.2 種類別バリデーション

- 必須キーが存在するか
    
- 配列項目が配列か
    
- object 項目が object か
    
- `reference_ids` が存在する場合は string 配列か
    

## 12.3 軽微な不整合

例:

- 空配列が多い
    
- optional 項目が欠ける
    

これは warning として保存し、完全失敗にはしない。

## 12.4 重大な不整合

例:

- payload が文字列
    
- 必須キーが大きく欠落
    
- schema_name と実際の payload 形が不一致
    

この場合は保存失敗または再生成対象とする。

---

## 13. 保存方針

## 13.1 ai_summaries

以下を保存する。

- summary_scope
- target_entity_type
- target_entity_id
- title
- body_markdown
- structured_json
- model_name
- prompt_version
- status
- generated_at

以下も合わせて保持する。
- input_snapshot_hash
- source_reference_ids_json
- generation_context_json

### 補足
- `input_snapshot_hash`: 生成入力の同一性確認用
- `source_reference_ids_json`: 根拠参照の再追跡用
- `generation_context_json`: target market / timeframe / language などの補助情報

## 13.2 structured_json の保存単位

`structured_json` 全体を JSON として保存する。  
トップレベルに共通ラッパーを含める。

## 15. バージョン管理方針

## 15.1 schema_version

MVPはすべて `1.0` とする。

## 15.2 将来拡張時

- キー追加だけなら `1.1`
    
- 互換性を壊す変更なら `2.0`
    

## 15.3 prompt_version との違い

- `schema_version`: JSON形のバージョン
    
- `prompt_version`: AI生成プロンプトのバージョン
    

両方を分けて管理する。

## 16. 失敗時 / 情報不足時の表現

## 16.1 情報不足だが出力可能

```json
{
  "schema_name": "alert_reason_summary",
  "schema_version": "1.0",
  "confidence": "low",
  "insufficient_context": true,
  "payload": {
    "what_happened": "価格条件のアラートが発火した。",
    "fact_points": ["関連する明確な外部材料は限定的。"],
    "reason_hypotheses": [],
    "watch_points": ["翌営業日の値動き確認"],
    "next_actions": ["適時開示の有無を再確認"],
    "reference_ids": []
  }
}
```

## 16.2 生成失敗

生成失敗は `ai_summaries` に不完全 JSON を保存するのではなく、`ai_jobs.failed` として扱う方針を優先する。

## 17. 設計上の重要判断

### 17.1 `structured_json` はトップレベル共通ラッパー付きにする

後で扱いやすいため。

### 17.2 事実と仮説を明確に分ける

特にアラート要約では重要。

### 17.3 点情報は文字列配列ではなくオブジェクト形式を基本とする

出典や将来拡張を見越すため。

### 17.4 空でもキーを維持する

UI実装とバリデーションが安定する。

### 17.5 schema_version と prompt_version を分離する

データ形と生成ロジックは別に変わるため。

## 20. 結論

北極星MVPの AI出力 JSON schema 設計では、AI生成物を単なる文章ではなく、  
**UI表示・比較・履歴管理・再利用に使える構造化データ** として扱う。

そのため、MVPでは以下を採用する。

- 共通ラッパー + 種類別 payload
    
- schema_name / schema_version / confidence / insufficient_context の共通保持
    
- 日次サマリ、アラート起点、銘柄論点、バックテスト総評、Pine生成補助の 5 種類を定義
    
- 必須キーを固定しつつ、空配列や低 confidence で情報不足を表現可能にする
    

この設計により、北極星の AIレイヤーをプロダクトの一部として安定運用できる状態を目指す。
