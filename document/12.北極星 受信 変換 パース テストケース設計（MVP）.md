# 北極星 受信 / 変換 / パース テストケース設計（MVP）

更新日: 2026-03-09

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVPにおいて、  
以下の中核処理に対する **テストケース設計** を定義するものである。

- webhook受信
    
- external references 収集連携
    
- AI要約生成前処理
    
- 自然言語→normalized_rule_json 変換
    
- normalized_rule_json→Pine 変換
    
- バックテスト CSV 取込 / パース
    

北極星のMVPでは、TradingView を外部基盤として活用するため、  
特に **外部入力を受ける処理** と **AI/変換系処理** が壊れやすい。  
そのため、単なる画面確認ではなく、  
**入力→正規化→保存→後続処理接続** の一連が壊れていないかを確認する必要がある。

---

## 2. テスト設計の基本方針

### 2.1 ユースケース単位ではなく処理境界単位で切る

今回のテスト設計では、以下の境界ごとにケースを整理する。

1. webhook受信境界
    
2. symbol解決境界
    
3. AI要約前処理境界
    
4. 自然言語正規化境界
    
5. Pine生成境界
    
6. CSV取込境界
    
7. CSVパース境界
    

### 2.2 正常系だけでなく「部分成功」を重視する

北極星MVPでは、完全成功 / 完全失敗だけでなく、  
以下のような状態が多い。

- symbol未解決だが payload 保存は成功
    
- CSVの一部列だけ読めた
    
- 自然言語の一部だけ変換できた
    
- external references が 0 件だが要約は生成可能
    

そのため、`partial` 系ケースを明示的に持つ。

### 2.3 永続化結果まで確認対象にする

レスポンスだけでなく、

- DB保存内容
    
- 状態遷移
    
- 後続ジョブ起動有無  
    も確認対象にする。
    

### 2.4 MVPの現実運用を優先する

理論上ありうるすべてではなく、MVPで現実に起きやすい問題を優先して設計する。

---

## 3. テスト対象の大分類

本ドキュメントでは、テスト対象を以下の6群に分ける。

1. webhook受信テスト
    
2. symbol解決・正規化テスト
    
3. AI要約前処理テスト
    
4. 自然言語→normalized_rule_json テスト
    
5. normalized_rule_json→Pine テスト
    
6. CSV取込 / パーステスト

## 5. webhook受信テスト

## 5.1 正常系

### TC-WH-001

**対象機能**: webhook受信

**目的**: 正常なJSON payloadを受信し、alert_event を作成できることを確認する。

**前提条件**:

- 有効な webhook token が発行済み
    
- symbols に `TSE:7203` が登録済み
    

**入力**:

- valid JSON payload
    
- `provider=tradingview`
    
- `tradingview_symbol=TSE:7203`
    
- `alert_kind=technical`
    
- `triggered_at` 有効
    

**手順**:

1. webhook endpoint に POST
    
2. レスポンス確認
    
3. alert_events を確認
    
4. ai_jobs を確認
    

**期待結果**:

- HTTP 200
    
- `accepted=true`
    
- alert_event が 1 件作成される
    
- `processing_status=received` または `normalized`
    
- 後続ジョブが queued される
    

---

### TC-WH-002

**対象機能**: webhook受信

**目的**: plain text payload を受信し、raw payload として保存できることを確認する。

**前提条件**:

- 有効な token
    

**入力**:

- text/plain body
    

**期待結果**:

- HTTP 200 もしくは受理扱い
    
- raw body が保存される
    
- `processing_status=needs_review` または `unresolved_symbol`
    
- symbol が明確なら alert_event 作成
    

---

## 5.2 異常系

### TC-WH-003

**対象機能**: webhook受信

**目的**: token 不正時に認証エラーとなることを確認する。

**入力**:

- 無効 token
    
- valid JSON body
    

**期待結果**:

- HTTP 401
    
- alert_event 未作成
    
- receipt log に auth failure が残る
    

---

### TC-WH-004

**対象機能**: webhook受信

**目的**: JSON 構文不正時に payload invalid として処理されることを確認する。

**入力**:

- 壊れた JSON
    

**期待結果**:

- HTTP 400 または 422
    
- `EXTERNAL_PAYLOAD_INVALID`
    
- alert_event 未作成
    
- raw body は保存可
    

---

### TC-WH-005

**対象機能**: webhook受信

**目的**: 必須キー不足時の挙動を確認する。

**入力**:

- `alert_name` 欠損
    
- `tradingview_symbol` 欠損
    
- `symbol` も欠損
    

**期待結果**:

- 受理不可または `failed`
    
- symbol 解決不能
    
- parse / validation error が残る
    

---

## 5.3 冪等性

### TC-WH-006

**対象機能**: 重複排除

**目的**: 同一 payload の再送時に重複 alert_event が作られないことを確認する。

**前提条件**:

- 1回目送信済み
    

**入力**:

- 同一 dedupe_key を持つ payload
    

**期待結果**:

- 新規 alert_event は作られない
    
- 既存 event_id を返す、または duplicate 扱いになる
    
- receipt log は追加されてもよい
    

---

### TC-WH-007

**対象機能**: 重複排除

**目的**: 近似 payload は別イベントとして扱われることを確認する。

**入力**:

- alert_name 同じ
    
- symbol 同じ
    
- triggered_at 同じ
    
- trigger_price が異なる
    

**期待結果**:

- 新規 alert_event が作成される、または仕様通りに区別される

**目的**: RSI + 出来高条件を and 条件で正規化できることを確認する。

**入力**:

**期待結果**:

- entry.logic = and
    
- indicator_threshold と volume_multiple が生成される
    

---

### TC-NRM-003

**対象機能**: 自然言語正規化

**目的**: 利確 / 損切り条件が exit に分離されることを確認する。

## 14. 設計上の重要判断

### 14.1 partial 成功をテスト対象に含める

北極星の現実運用に必要。

### 14.2 結合テストを早めに持つ

外部入力起点の機能が多いため、単体テストだけでは不十分。

### 14.3 AI文面の良し悪しより schema 安定性を優先する

MVPではここを最優先で守る。

### 14.4 取込系は fixture 管理を重視する

CSV と webhook payload の再現データが重要。

## 16. 結論

北極星MVPのテストケース設計では、  
**webhook受信・自然言語変換・Pine生成・CSVパース** の4系統を最重要として扱い、  
正常系だけでなく **partial 成功・曖昧入力・壊れた外部入力** を含めて設計する。

そのため、MVPでは以下を重視する。

- 外部入力境界のテスト
    
- 中間表現の妥当性テスト
    
- 正規化結果の保存確認
    
- 結合フローの疎通確認
    
- fixtures に基づく再現可能な検証
    

この設計により、北極星の中核機能を壊れにくくし、実装初期から安定性を確保することを目指す。
