# 北極星 自然言語→Pine変換フロー設計（MVP）

更新日: 2026-03-09

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」における **自然言語から Pine Script へ変換するフロー** を定義するものである。

北極星の重要コンセプトの一つに、  
**「ルール運用の検証：売買ルールを自然言語→バックテスト→レポート化（再現性重視）」**  
がある。

この設計では、ユーザーが自然文で入力した売買ルールを、北極星側で正規化し、Pine Script の strategy として出力するまでの処理を整理する。

---

## 2. この機能の役割

自然言語→Pine変換機能の主な役割は以下。

1. ユーザーが人間向けに書いた売買ルールを受け取る
    
2. 曖昧な記述を検出する
    
3. バックテスト可能な形式へ正規化する
    
4. Pine Script の strategy として出力する
    
5. 生成結果を保存し、再検証可能な状態にする
    
6. 変換できなかった点や注意点をユーザーへ明示する
    

---

## 3. MVPでの対象範囲

MVPでは、以下の種類のルールを対象とする。

### 3.1 テクニカル条件ベース

例:

- 25日移動平均線の上で買う
    
- RSIが50を超えたら買う
    
- MACDがゴールデンクロスしたら買う
    

### 3.2 価格条件ベース

例:

- 直近高値を上抜けたら買う
    
- 前日安値を割れたら売る
    

### 3.3 出来高条件ベース

例:

- 出来高が20日平均の1.5倍を超えたら買う
    

### 3.4 シンプルなイグジット条件

例:

- 終値が25日線を下回ったら手仕舞い
    
- 5%で利確、3%で損切り
    

### 3.5 ロング主体の単純戦略

MVPでは、最初はロング主体を標準とし、ショートは対応可能なら扱うが必須にしない。

---

## 4. MVPであえて扱わないもの

MVPでは以下は対象外または制限付きとする。

- 複雑なポジション分割
    
- 高度な資金管理
    
- 複数銘柄同時売買ロジック
    
- 複数時間足の混在戦略
    
- ファンダメンタル条件の自動参照
    
- 外部データ依存の複雑な条件
    
- 条件分岐が深い裁量ルール
    
- 自然言語だけで完全に曖昧さゼロの変換
    

これらは Phase 2 以降の対象とする。

---

## 5. 基本方針

### 5.1 自然言語をそのままコード化しない

自然言語を直接 Pine に変換するのではなく、  
必ず中間表現を挟む。

### 5.2 中間表現を正本にする

再現性の観点から、Pine 本文よりも **正規化されたルール定義** を正本とする。

### 5.3 曖昧さは消さずに明示する

変換時に不明点がある場合、勝手に過剰解釈せず、

- default を使った箇所
    
- 解釈した箇所
    
- 未対応箇所  
    を warnings として残す。
    

### 5.4 strategy として出力する

MVPでは検証用途を優先するため、indicator より strategy を標準出力にする。

### 5.5 アラート運用も見越す

生成する Pine は、将来的な alert / forward test にも接続しやすい形にする。

## 6. 全体フロー

自然言語→Pine変換の全体フローは以下。

1. 自然言語入力受付
    
2. ルール文の分解
    
3. 要素抽出
    
4. 中間表現への正規化
    
5. 妥当性チェック
    
6. Pine テンプレート選択
    
7. Pine コード生成
    
8. warnings / notes 生成
    
9. 保存
    
10. UI表示 / コピー / 再生成

## 7. 入力仕様

## 7.1 ユーザー入力の想定

### 自由記述例

- 25日移動平均線の上で、RSIが50以上、出来高が20日平均の1.5倍以上なら買い。終値が25日線を下回ったら手仕舞い。
    
- 直近高値を更新したら買い、5%上がったら利確、3%下がったら損切り。
    
- MACDがゴールデンクロスしたらエントリー。デッドクロスで手仕舞い。
    

### 補助入力項目

MVPでは自然文に加えて、以下をフォームで持たせる。

- 売買方向
    
- 対象市場
    
- 時間足
    
- バックテスト期間
    
- commission
    
- slippage
    

理由:  
自然文だけに寄せすぎると曖昧性が高すぎるため。

## 10. 曖昧さ処理方針

## 10.1 曖昧な自然文の例

- 「強いときに買う」
    
- 「トレンドが良ければ買う」
    
- 「出来高が多いとき」
    
- 「適度に利確する」
    

これらはそのまま Pine 化できない。

## 10.2 MVPの対応

以下の 3 パターンで扱う。

### パターンA: 自動解釈可能

例:

- 「25日線の上で買う」
    
- `close > sma(close, 25)` に変換
    

### パターンB: default 付きで解釈

例:

- 「RSIが高い」
    
- default で 70 と解釈するか、warning を出す
    

### パターンC: 自動変換不可

例:

- 「強い銘柄だけ買う」
    
- 未対応として明示し、Pine 生成を止めるか部分生成にする
    

## 10.3 warnings の設計

変換時に以下を保存する。

- default_value_applied
    
- unsupported_expression_detected
    
- partially_translated
    
- assumed_long_only
    
- timeframe_from_form_input

## 11. Pine出力方針

## 11.1 出力種別

MVPでは Pine の **strategy script** を出力する。

理由:

- Strategy Tester で検証しやすい
    
- order fill alert にも接続しやすい
    
- 検証レポートとの相性が良い
    

## 11.2 出力コードの基本構成

生成されるコードは以下の構造を持つ。

1. version 宣言
    
2. strategy() 宣言
    
3. input 定義（必要最小限）
    
4. インジケーター計算
    
5. entry 条件定義
    
6. exit 条件定義
    
7. strategy.entry() 呼び出し
    
8. strategy.exit() または strategy.close() 呼び出し
    
9. 必要に応じた plot / plotshape
    
10. 必要に応じた alert 用 alert_message
    

---

## 11.3 生成コードの設計原則

- 人が読んで修正しやすい
    
- 条件名をコメントで明示する
    
- 変数名は自明にする
    
- ロジックの順序を安定化する
    
- 不要に凝った短縮記法を使わない
    

---

## 12. 生成テンプレート設計

MVPでは、ルールカテゴリに応じて Pine テンプレートを選ぶ。

## 12.1 long_only_basic_strategy

対象:

- ロングのみ
    
- entry / exit が単純
    

## 12.2 long_only_with_tp_sl

対象:

- 利確 / 損切りを含む
    

## 12.3 long_short_basic_strategy

対象:

- ロング / ショート両対応（任意）
    

MVPの基本は `long_only_basic_strategy` と `long_only_with_tp_sl` の2系統で十分。

---

## 13. 生成コードの例

入力自然文:

- 25日移動平均線の上で、RSIが50以上、出来高が20日平均の1.5倍以上なら買い。終値が25日線を下回ったら手仕舞い。
    

生成イメージ:

```pine
//@version=6
strategy("Hokkyokusei Generated Strategy", overlay=true)

ma25 = ta.sma(close, 25)
rsi14 = ta.rsi(close, 14)
volMa20 = ta.sma(volume, 20)

entryCondition = close > ma25 and rsi14 >= 50 and volume >= volMa20 * 1.5
exitCondition = close < ma25

if entryCondition and strategy.position_size == 0
    strategy.entry("Long", strategy.long, alert_message = "entry_long")

if exitCondition and strategy.position_size > 0
    strategy.close("Long", alert_message = "exit_long")

plot(ma25)
```

## 14. alert連携方針

生成する Pine は、将来の webhook / forward test も見据えて alert 連携しやすくする。

## 14.1 strategy order fill ベース

MVPでは strategy の `alert_message` を基本的に埋める。

例:

- entry_long
    
- exit_long
    
- stop_loss
    
- take_profit
    

## 14.2 生成側の責務

北極星側では、Pine 生成時に `alert_message` を自動で入れておく。

理由:

- TradingView の order fill alert 連携がしやすい
    
- 後で webhook payload の意味づけがしやすい

## 18. 出力バリデーション設計

生成された Pine は、そのまま返すのではなく最低限の静的チェックを行う。

## 18.1 チェック対象

- version 宣言があるか
    
- strategy() があるか
    
- entry 条件があるか
    
- 変数名衝突がないか
    
- 未定義トークンが残っていないか

## 18.2 warnings 例

## 20. 失敗時の扱い

## 20.1 部分生成可能な場合

- 生成は行う
    
- omitted / assumed を warnings で返す
    

## 20.2 生成不可能な場合

- pine_scripts は保存しない
    
- failed として記録
    
- どの表現が未対応か返す
    

## 20.3 代表的な失敗例

- 条件が曖昧すぎる
    
- entry / exit が不完全
    
- 未対応インジケーター
    
- 複数時間足指定
    
- 裁量語が多すぎる
    

---

## 21. 品質管理方針

## 21.1 変換品質より再現性を優先

MVPでは「賢そうに見えるコード」より「何をどう解釈したかが追える」ことを優先する。

## 21.2 strategy の意味が一意になることを優先

ロジックの過度な最適化はしない。

## 21.3 テンプレート駆動を基本にする

完全自由生成にすると不安定になるため、可能な限りテンプレートで縛る。

---

## 22. 設計上の重要判断

### 22.1 中間表現を必須にする

自然言語→Pine の直接変換は採らない。

### 22.2 最初はロング主体に寄せる

複雑さを抑え、実用価値の高い範囲から始める。

### 22.3 未対応表現を無理に吸収しない

壊れた Pine を生成するより、未対応を明示する方がよい。

### 22.4 strategy alert 連携を見越して `alert_message` を埋める

後続の webhook 設計と整合する。

## 25. 結論

北極星MVPの自然言語→Pine変換フローでは、  
**自然言語をそのままコード化せず、中間表現へ正規化してから Pine strategy を生成する** ことを基本方針とする。

そのうえで、MVPでは以下を採用する。

- ロング主体のシンプル戦略を優先
    
- テクニカル / 価格 / 出来高 / 単純 exit を中心に対応
    
- normalized_rule_json を正本とする
    
- warnings で解釈や未対応を明示する
    
- strategy として出力し、alert_message も埋める
    

この設計により、北極星の「自然言語→バックテスト→レポート化」を支える安定した変換基盤を構築する。
