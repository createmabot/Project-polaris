# 北極星 AI要約フロー設計（MVP）

更新日: 2026-03-08

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」における **AI要約機能の処理フロー** を定義するものである。

北極星では、TradingView が担う「表示・監視・一次検証」に対して、北極星側が **意味づけ・整理・蓄積** を担う。その中心にあるのが AI要約機能である。

MVPにおいて AI要約は、以下の価値を実現するための中核機能と位置づける。

- 毎日5分で状況把握
    
- 銘柄研究の高速化
    
- ルール検証結果の解釈補助
    

---

## 2. AI要約機能の役割

北極星における AI要約は、単なる文章短縮ではない。以下の役割を担う。

1. **イベントの意味づけ**
    
    - 価格変動やアラートが「なぜ起きたか」を整理する
        
2. **論点の構造化**
    
    - 強気材料 / 弱気材料 / 注視KPI / 反証条件などに分解する
        
3. **比較の高速化**
    
    - 2銘柄以上の差分を比較可能な形に整える
        
4. **検証結果の解釈補助**
    
    - バックテスト結果がどういう傾向を持つかを自然言語で説明する
        
5. **知見の蓄積**
    
    - その時点の解釈を再利用できる資産として保存する
        

---

## 3. MVPで扱うAI要約の種類

MVPでは、AI要約を以下の4種類に限定する。

### 3.1 日次サマリ

用途:

- ホーム画面の朝サマリ / 夜サマリ
    
- 毎日5分の状況把握
    

### 3.2 アラート起点サマリ

用途:

- 「今日動いた理由」
    
- TradingViewアラートを意味づけする
    

### 3.3 銘柄論点サマリ

用途:

- 銘柄詳細画面の論点カード
    
- 銘柄研究の高速化
    

### 3.4 バックテスト総評

用途:

- ルール検証結果の要約
    
- 戦略の特徴と注意点の言語化
    

---

## 4. AI要約の基本方針

### 4.1 必ず入力根拠を持つ

AI要約は、必ず次のいずれかの入力根拠を持つ。

- alert_event
    
- external_references
    
- research_notes
    
- comparison_results
    
- backtest_metrics / backtest_trade_summaries
    

### 4.2 生成物は構造化データと文章の両方を持つ

表示用途だけでなく、再利用・比較・検索にも使うため、要約は以下の二層で保存する。

- `body_markdown`: 人が読むための本文
    
- `structured_json`: UIや比較処理に使うための構造化出力
    

### 4.3 断定しすぎない

特に値動き理由の要約では、事実と推測を分ける。

例:

- 事実: 価格上昇、出来高増加、開示発表
    
- 推測: 市場が好感した可能性がある
    

### 4.4 出典リンクを意識する

本文全文保存ではなくても、どのニュース・開示・決算情報に基づくかを追える状態にする。

### 4.5 まずは「信頼できる簡潔さ」を優先する

MVPでは高度な文章表現よりも、

- 読みやすい
    
- 根拠が追える
    
- UIで扱いやすい  
    ことを優先する。

## 6. AI要約の処理フロー全体

AI要約は以下の共通フローで処理する。

1. 対象ユースケースの確定
    
2. 入力データ収集
    
3. 入力データ正規化
    
4. 要約用コンテキスト構築
    
5. AI生成ジョブ実行
    
6. 出力バリデーション
    
7. ai_summaries 保存
    
8. 関連エンティティへの紐付け
    
9. UI反映

## 9. 出力フォーマット設計

MVPでは ai_summaries に保存する `structured_json` の標準形を定義する。

## 9.1 日次サマリ

```json
{
  "highlights": [
    {
      "title": "半導体株が上昇",
      "reason": "関連ニュースとセクター上昇",
      "confidence": "medium"
    }
  ],
  "watch_items": ["決算予定", "為替動向"],
  "focus_symbols": ["sym_1", "sym_2"]
}
```

## 9.2 アラート起点サマリ

```json
{
  "what_happened": "25日線上抜けアラートが発火",
  "fact_points": ["出来高増加", "関連開示あり"],
  "reason_hypotheses": ["開示内容を好感した可能性"],
  "watch_points": ["翌営業日の継続性"],
  "reference_ids": ["ref_1", "ref_2"]
}
```

## 9.3 銘柄論点サマリ

```json
{
  "bullish_points": ["業績見通し改善"],
  "bearish_points": ["バリュエーション高め"],
  "watch_kpis": ["営業利益率", "受注残"],
  "next_events": ["次回決算"],
  "invalidation_conditions": ["ガイダンス下方修正"]
}
```

## 9.4 バックテスト総評

```json
{
  "strategy_characteristics": ["順張り寄り", "勝率よりPF型"],
  "strengths": ["上昇局面に強い"],
  "risks": ["レンジ相場に弱い"],
  "next_improvements": ["出来高条件の見直し"]
}
```

## 10. 品質管理方針

## 10.1 事実と推測の分離

特にアラート起点サマリでは、出力時に以下を分ける。

- `fact_points`
    
- `reason_hypotheses`
    

これにより、AIの断定しすぎを抑える。

## 10.2 根拠不足時の挙動

十分な external_references が無い場合は、以下の方針とする。

- 要約生成は行う
    
- ただし「背景要因は限定的」と明示する
    
- `confidence = low` を付与する
    

## 10.3 出力長の制御

MVPでは読みやすさを優先し、要約本文は短めに制御する。

### 目安

- 日次サマリ: 250〜500字程度
    
- アラート起点: 150〜300字程度
    
- 銘柄論点: 箇条書き中心
    
- バックテスト総評: 250〜500字程度
    

## 10.4 再生成の扱い

要約は再生成可能とするが、過去版は消さずに残す設計が望ましい。  
MVPでは少なくとも `prompt_version` と `generated_at` を保持する。

## 11. 非同期ジョブ設計

AI要約は非同期ジョブとして扱う。

## 11.1 ai_jobs.job_type

MVPでは以下を定義する。

- generate_daily_summary
    
- generate_alert_summary
    
- generate_symbol_thesis
    
- generate_backtest_review
    

## 11.2 状態

- queued
    
- running
    
- succeeded
    
- failed
    

## 11.3 完了条件

- ai_summaries にレコード保存済み
    
- target_entity と紐付け済み
    

## 11.4 失敗時

- ai_jobs.status = failed
    
- error_message 保存
    
- UIに再生成導線を出す

## 12. リトライ設計

## 12.1 自動リトライ対象

- 一時的モデルエラー
    
- タイムアウト
    
- 外部参照取得遅延による一時不足
    

## 12.2 自動リトライしない対象

- 入力必須データ不足
    
- symbol未解決
    
- 参照情報ゼロかつ要約不能と判断された場合
    

## 12.3 手動再生成

以下では常に手動再生成可能とする。

- ホームの日次サマリ
    
- 銘柄詳細の論点カード
    
- アラート詳細の理由要約
    
- バックテスト総評

## 16. MVPであえてやらないこと

MVPでは以下は行わない。

- 長文レポート自動生成
    
- 自動売買判断の提示
    
- 価格予測の断定的出力
    
- ニュース本文の全文要約アーカイブ
    
- 多段エージェント構成
    
- 複数モデルの自動比較採択

## 17. 設計上の重要判断

### 17.1 要約は「文章」ではなく「プロダクト機能」として設計する

そのため、構造化出力を必須にする。

### 17.2 値動き理由の説明は、推測の透明性を優先する

市場要因の説明は確実性が低いため、confidence と仮説扱いを明示する。

### 17.3 ノートとの接続を持つ

銘柄論点サマリは、研究ノートと独立ではなく相互補完とする。

### 17.4 バックテスト総評は改善の方向づけに使う

単なる成績読み上げではなく、次に見直す点を返す。

## 20. 結論

北極星MVPのAI要約フローは、各ユースケースごとに必要な入力を集め、正規化し、  
**短く・根拠を持って・構造化して保存する** ことを基本方針とする。

MVPでは以下の4種類を中心に実装する。

- 日次サマリ
    
- アラート起点サマリ
    
- 銘柄論点サマリ
    
- バックテスト総評
    

この設計により、北極星独自の価値である **意味づけ・研究支援・検証結果の解釈補助** を安定して提供できる状態を目指す。
