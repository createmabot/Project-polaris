# 北極星 DBマイグレーション順序案（MVP）

更新日: 2026-03-15

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」のMVP開発に向けて、**DBマイグレーションを安全に進める順序** を定義するものである。

北極星では、以下のように依存関係の強いテーブル群が存在する。

- symbols を起点に webhook / notes / strategy / backtest がぶら下がる
    
- alert_events を起点に external_references と ai_summaries がぶら下がる
    
- strategy_rule_versions を起点に pine_scripts と backtest_runs がぶら下がる
    
- backtest_runs を起点に import / metrics / trade summaries がぶら下がる
    

そのため、DBテーブルを思いつき順に追加すると、

- foreign key が貼れない
    
- seed 順が壊れる
    
- API 実装を始めても参照先がまだ無い
    
- rollback が危険になる  
    といった問題が起きやすい。
    

本資料では、MVPで必要なテーブルを **依存関係順** に並べ、  
各段階で何を確認すべきかも含めて整理する。

---

## 2. 基本方針

### 2.1 まず親テーブル、次に従属テーブル

基本原則として、以下の順で進める。

1. owner / master
    
2. event / content
    
3. generated / derived
    
4. comparison / support
    

### 2.2 1 migration 1責務を原則にする

- users
    
- symbols
    
- alert_events  
    のように、原則 1テーブル 1migration に分ける。
    

ただし、密結合の join テーブルは同時追加でもよい。

### 2.3 初期 seed と migration の責務を分ける

- migration: schema を作る
    
- seed: 初期データを入れる
    

`symbols` のような初期データ必須テーブルでも、schema とデータ投入は分離する。

### 2.4 まず nullable 多めで入り、後で締める

MVP初期は、運用で必要なデータの形がまだ動くため、  
いきなり制約を厳しくしすぎない。

ただし、以下は初期から強く持つ。

- 主キー
    
- created_at
    
- 必須の owner 外部キー
    
- 明らかな一意制約
    

### 2.5 外部入力起点のテーブルを早めに作る

webhook と CSV 取込は外部入力起点で壊れやすいため、  
これらの保存先は早めに作る。

## 15. rollback 方針

### 基本方針

- 依存の深い後段テーブルから戻す
    
- seed rollback は必須化しない
    
- 本番相当データでは destructive rollback を安易に使わない
    

### rollback 順の考え方

追加順の逆順を基本とする。

特に注意:

- backtest 系
    
- pine_scripts
    
- note_revisions
    
- external_references
    
- alert_events
    

---

## 16. 実装時の注意点

### 16.1 ポリモーフィック参照は FK を貼りすぎない

- ai_jobs.target_entity_type / id
    
- ai_summaries.target_entity_type / id
    
- entity_tags.entity_type / id
    

これらは DB 上の厳格 FK より、アプリケーションレベル整合性を優先する。

### 16.2 unresolved を保存できるようにする

alert_events.symbol_id などは初期段階で nullable を許容する方が安全。

### 16.3 JSON カラムは早めに用意する

- trigger_payload_json
    
- metadata_json
    
- structured_json
    
- normalized_rule_json
    
- metrics_json
    
- trade_summary_json
    
- parse_log
    

北極星は schema 柔軟性が必要なため、MVPでは JSON を積極的に活用する。

### 16.4 created_at / updated_at 方針を統一する

更新履歴が重要なため、少なくとも主要テーブルは timestamps を揃える。

## 20. 結論

北極星MVPの DBマイグレーションでは、  
**users / symbols の親テーブルから始め、webhook・AI・研究・ルール検証・backtest の順に従属テーブルを積み上げる** のが安全である。

そのため、MVPでは以下を重視する。

- 依存順に migration を並べる
    
- seed は migration と分ける
    
- unresolved や partial を保存できる柔軟性を持つ
