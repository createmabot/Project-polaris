# 北極星 TradingView alert message テンプレート設計（MVP）

更新日: 2026-03-09

## 1. このドキュメントの目的

本ドキュメントは、株価評価AIツール「北極星」において、TradingView から webhook 送信する際の **alert message テンプレート** を標準化するための設計を定義するものである。

北極星では webhook 受信基盤をすでに設計しているが、実運用では TradingView 側の alert message 設定がばらつくと、  
以下の問題が発生しやすい。

- payload が JSON にならず、text/plain で送られる
    
- symbol 解決に必要な情報が欠ける
    
- alert 種別によって項目が不足する
    
- strategy alert と通常 alert で項目構造が揃わない
    
- 後続の AI 要約や履歴管理で扱いにくくなる
    

そのため、MVPでは **北極星用の推奨 alert message テンプレート** を定義し、運用を揃える。

---

## 2. この設計の対象範囲

MVPで対象とするのは以下の TradingView alert である。

### 2.1 価格アラート

例:

- 価格が指定値を上抜け
    
- 価格が指定値を下抜け
    

### 2.2 テクニカルアラート

例:

- 25日線上抜け
    
- RSI閾値到達
    
- 出来高急増
    

### 2.3 strategy / order fill alert

例:

- Pine strategy のエントリー
    
- Pine strategy のイグジット
    
- フォワード監視の補助通知
    

MVPでは watchlist alert も webhook 受信対象に含むが、message テンプレートの基本設計は上記 3 類型で統一できるようにする。

---

## 3. 設計方針

### 3.1 JSON固定を標準とする

北極星向け webhook では、alert message は **常に JSON 形式** を推奨標準とする。

理由:

- サーバ側パースが安定する
    
- 必須項目の不足を検出しやすい
    
- 将来の項目追加に強い
    
- plain text 運用より事故が少ない
    

### 3.2 キー名は北極星の内部標準に寄せる

TradingView 側の message から、北極星の internal event へマッピングしやすいよう、キー名は北極星側のドメインに合わせる。

### 3.3 alert 種別ごとにテンプレートを分ける

すべてを1つの巨大テンプレートにせず、用途別テンプレートを用意する。

### 3.4 必須キーを固定する

最低限、次のキーはどのテンプレートでも含める。

- provider
    
- alert_kind
    
- alert_name
    
- tradingview_symbol
    
- symbol
    
- exchange
    
- timeframe
    
- triggered_at
    
- message
    

### 3.5 将来拡張しやすい構造にする

MVPでは最小限に留めつつも、`context` や `metadata` を持たせる余地を残す。

---

## 4. TradingView の制約を踏まえた前提

MVPでは、TradingView の alert message は次の前提で扱う。

- valid JSON で送れば JSON として webhook に届く
    
- valid JSON でなければ plain text 扱いになる
    
- 一般 alert では placeholder を使って動的値を埋められる
    
- strategy alert では strategy 系 placeholder を使える
    
- alert() / alertcondition() / order fill alert では使いやすい値の出し方が異なる
    

そのため、北極星側では「何でも受ける」ではなく、**TradingView で設定すべきテンプレートを先に決める** 方針を採る。

---

## 5. 共通 payload 仕様

MVPでは、TradingView から北極星へ送る payload の共通形を以下とする。

```json
{
  "provider": "tradingview",
  "alert_kind": "price",
  "alert_name": "JP price breakout",
  "tradingview_symbol": "TSE:7203",
  "symbol": "7203",
  "exchange": "TSE",
  "timeframe": "D",
  "triggered_at": "2026-03-09T09:15:00Z",
  "price": 3021.5,
  "currency": "JPY",
  "message": "トヨタ自動車が価格条件を満たした",
  "context": {
    "condition_summary": "price crossed above threshold"
  }
}
```

---

## 6. 共通キー定義

## 6.1 必須キー

### provider

固定値 `tradingview`

### alert_kind

値の候補:

- price
    
- technical
    
- strategy
    
- watchlist
    

### alert_name

TradingView 上での alert 名称。  
北極星側でも主要識別に使う。

### tradingview_symbol

例:

- `TSE:7203`
    
- `NASDAQ:AAPL`
    
- `FX:USDJPY`
    

北極星側の symbol 解決で最優先に使う。

### symbol

銘柄コードやティッカー。  
補助的に使う。

### exchange

取引所コード。

### timeframe

例:

- `1`
    
- `5`
    
- `60`
    
- `D`
    
- `W`
    

### triggered_at

Alert 発火時刻。  
UTC の ISO 8601 文字列を想定。

### message

人が読むための短い説明文。

---

## 6.2 推奨キー

### price

トリガー時の価格。

### currency

通貨コード。

### volume

トリガー時の出来高。

### interval

TradingView の interval placeholder を使った補助値。  
`timeframe` と同じ意味で使わず、補助情報として保持してもよい。

### context

アラート種別ごとの補足情報。

### shared_secret

Payload 内で追加検証したい場合に利用。

---

## 6.3 将来拡張キー

### watchlist_name

watchlist alert 用。

### strategy_name

strategy alert 用。

### order_action

strategy order fill 用。

### order_id

strategy order fill 用。

### position_size

strategy alert 用。

### metadata

自由拡張用。

## 7. テンプレート種別

MVPでは以下の 3 テンプレートを標準採用する。

1. 価格 / テクニカルアラート用テンプレート
    
2. watchlist / 複数銘柄監視向けテンプレート
    
3. strategy / order fill 用テンプレート
    

---

## 8. テンプレートA: 価格 / テクニカルアラート用

## 8.1 用途

- 通常の価格アラート
    
- インジケーター起点アラート
    
- alertcondition() ベースのテクニカル通知
    

## 8.2 推奨テンプレート

```json
{
  "provider": "tradingview",
  "alert_kind": "technical",
  "alert_name": "MA25 breakout",
  "tradingview_symbol": "{{exchange}}:{{ticker}}",
  "symbol": "{{ticker}}",
  "exchange": "{{exchange}}",
  "timeframe": "{{interval}}",
  "triggered_at": "{{time}}",
  "price": {{close}},
  "open": {{open}},
  "high": {{high}},
  "low": {{low}},
  "volume": {{volume}},
  "currency": "{{syminfo.currency}}",
  "message": "{{ticker}} triggered technical alert",
  "context": {
    "condition_summary": "close crossed above MA25"
  }
}
```

---

## 8.3 運用上の注意

### alert_name は固定文字列で明示する

`alert_name` は抽象名ではなく、あとで一覧で見て意味が分かる名前にする。

良い例:

- `MA25 breakout`
    
- `RSI over 70`
    
- `Volume spike`
    

悪い例:

- `Alert1`
    
- `test`
    

### condition_summary は人手管理でもよい

TradingView だけで完全自動生成しなくてもよい。  
MVPでは alert ごとに意味の分かる説明文を入れておく。

### `{{time}}` は UTC を前提に扱う

北極星側で JST に変換して表示する。

---

## 9. テンプレートB: watchlist / 複数銘柄監視向け

## 9.1 用途

- watchlist alert
    
- 監視銘柄群への共通条件通知
    
- 量が多いイベントの最初の受け口
    

## 9.2 推奨テンプレート

```json
{
  "provider": "tradingview",
  "alert_kind": "watchlist",
  "alert_name": "Watchlist momentum signal",
  "watchlist_name": "JP Core Watchlist",
  "tradingview_symbol": "{{exchange}}:{{ticker}}",
  "symbol": "{{ticker}}",
  "exchange": "{{exchange}}",
  "timeframe": "{{interval}}",
  "triggered_at": "{{time}}",
  "price": {{close}},
  "volume": {{volume}},
  "currency": "{{syminfo.currency}}",
  "message": "{{ticker}} matched watchlist condition",
  "context": {
    "condition_summary": "watchlist momentum condition matched"
  }
}
```

## 9.3 運用上の意図

watchlist alert は北極星側で一括整理するため、個別条件の完全詳細よりも、

- どの watchlist で
    
- どの銘柄が
    
- いつ発火したか  
    が安定して取れることを優先する。
    

---

## 10. テンプレートC: strategy / order fill 用

## 10.1 用途

- Pine strategy の entry / exit
    
- フォワード監視
    
- ルール検証後の実運用補助
    

## 10.2 推奨テンプレート

```json
{
  "provider": "tradingview",
  "alert_kind": "strategy",
  "alert_name": "Strategy order fill",
  "strategy_name": "MA25 Swing Strategy",
  "tradingview_symbol": "{{ticker}}",
  "symbol": "{{ticker}}",
  "exchange": "{{exchange}}",
  "timeframe": "{{interval}}",
  "triggered_at": "{{timenow}}",
  "price": {{strategy.order.price}},
  "currency": "{{syminfo.currency}}",
  "message": "{{strategy.order.alert_message}}",
  "context": {
    "order_action": "{{strategy.order.action}}",
    "order_id": "{{strategy.order.id}}",
    "contracts": "{{strategy.order.contracts}}",
    "position_size": "{{strategy.position_size}}",
    "market_position": "{{strategy.market_position}}"
  }
}
```

---

## 10.3 strategy テンプレートの補足

### triggered_at は `{{timenow}}` を優先

order fill 時はバー時刻より発火時刻の方が運用上扱いやすい。

### message は `{{strategy.order.alert_message}}` を中心にする

個別 order の意味を Pine 側で作り込みやすい。

### `tradingview_symbol` は strategy 側で完全な形式が出しにくい場合がある

MVPでは `symbol` と `exchange` の組み合わせでも補えるようにする。

## 13. Pine Script と message の責務分担

## 13.1 alertcondition() ベース

用途:

- テクニカル条件通知
    

方針:

- message 側に JSON テンプレートを書く
    
- 動的値は placeholder を使う
    

## 13.2 alert() ベース

用途:

- script 内の柔軟な通知
    

方針:

- Pine 側で動的文字列を作る
    
- JSON を文字列として組み立てる
    
- 後で webhook へそのまま流せる形にする
    

## 13.3 strategy order fill

用途:

- 実際の order fill に応じた通知
    

方針:

- `alert_message` を Pine 側で組み立てる
    
- TradingView の message 側では `{{strategy.order.alert_message}}` を利用する

## 15. バリデーションルール

## 15.1 送信前ルール

- JSON として妥当であること
    
- 数値は数値として出すこと
    
- 文字列は必ずダブルクォートで囲むこと
    
- 必須キーを欠かさないこと
    

## 15.2 北極星側での受信ルール

- `provider != tradingview` なら warning 扱い
    
- `tradingview_symbol` 欠損時は `symbol + exchange` で補完を試みる
    
- `alert_kind` 未知値は受理するが `unknown` 扱いにする
    
- `message` は空でも受理するが warning を出す

## 17. テンプレート運用ルール

## 17.1 まずは 3 テンプレートに限定する

MVPでは alert ごとに payload を自由化しない。  
まずは標準テンプレートを使う。

## 17.2 alert_name は運用命名規則を持つ

例:

- `MA25 breakout`
    
- `RSI over 70`
    
- `Volume spike`
    
- `Watchlist momentum signal`
    
- `Strategy order fill`
    

## 17.3 同じ系統の alert は message 構造を変えない

構造がブレるとサーバ側処理が不安定になるため、条件文だけ自由にしすぎない。

## 17.4 将来的にバージョン管理を入れられる形にする

必要なら後で `template_version` をキー追加できるようにする。

---

## 18. 失敗しやすいポイント

MVPで特に注意すべき点は以下。

### 18.1 JSON のダブルクォート崩れ

これが起きると plain text 扱いになりやすい。

### 18.2 数値を文字列にしてしまう

北極星側で再変換はできるが、初期は型ブレの原因になる。

### 18.3 `{{interval}}` の期待値ズレ

価格ベース alert では実運用上 `1` になることがあるため、timeframe の唯一の根拠にしすぎない。

### 18.4 strategy で `{{strategy.order.alert_message}}` を使わない

その場合、個別 order の意味が message に乗りにくい。

### 18.5 `tradingview_symbol` の形式不統一

後段の symbol 解決精度に直接影響する。

---

## 19. 北極星MVPでの採用決定事項

MVPでは以下を採用する。

- alert message は JSON を標準とする
    
- テンプレートは 3 種類に限定する
    
- 必須キーは固定する
    
- strategy alert は専用テンプレートを使う
    
- watchlist alert も単一 webhook endpoint に送る
    
- plain text は互換目的でのみ許容する

## 22. 結論

北極星MVPにおける TradingView alert message 設計では、  
**「壊れにくい JSON」「用途別テンプレート」「必須キーの固定」** を優先する。

そのため、MVPでは以下を標準採用する。

- 価格 / テクニカル用テンプレート
    
- watchlist 用テンプレート
    
- strategy / order fill 用テンプレート
    

この設計により、TradingView から北極星への webhook 連携を安定化させ、後段の symbol 解決・要約生成・履歴管理を扱いやすくする。
